给 Node 补元数据（标题、关键词、它能回答的问题、摘要）能显著提升检索与过滤。Extractor 用 LLM 自动为每个 Node 生成这些元数据。Enriching Nodes with metadata (title, keywords, the questions it answers, a summary) boosts retrieval and filtering. Extractors use an LLM to generate that metadata per Node automatically.

元数据 = 检索的第二通道Metadata: retrieval's second channel

向量相似度是第一检索通道，元数据是第二条。一方面，标题、来源、日期这类字段可做精确过滤（“只在某产品手册里检索”）；另一方面，像 questions_this_excerpt_can_answer 这样的字段，把“用户会怎么问”预先写进块里，让“问句”和“片段”在语义空间里更容易对齐、召回更稳。代价是每个 extractor 都要调用一次 LLM，按需启用即可。
Vector similarity is the first retrieval channel; metadata is the second. On one hand, fields like title, source and date enable exact filtering (“search only this product manual”); on the other, fields like questions_this_excerpt_can_answer bake “how users would ask” into the chunk itself, aligning “question ↔ chunk” in semantic space for steadier recall. The cost: each extractor is one LLM call, so enable them selectively.

❓ 为什么这么设计❓ Why designed this way
纯向量检索只会“按相似度”召回，遇到同义改写或宽泛提问容易漏。补上元数据这条第二通道，既能按字段过滤缩小范围，又能用“该块能回答的问题”把用户问句和片段对齐，召回更稳、更可控。
Pure vector search only recalls “by similarity” and can miss paraphrases or broad questions. Adding metadata as a second channel lets you both filter by field to narrow scope and align user questions with chunks via “questions this chunk answers” — steadier, more controllable recall.
⚙️ 内部怎么跑⚙️ How it runs inside
每个 extractor 都是一个 transformation，按提示词生成内容并写进固定的 metadata 键（document_title / excerpt_keywords / questions_this_excerpt_can_answer / section_summary）。但调用次数要分两类看：逐块的抽取器（KeywordExtractor / QuestionsAnsweredExtractor / SummaryExtractor）对每个 Node 各调一次 LLM，开销随块数 N 线性增长；文档级的抽取器（如 TitleExtractor）只为每篇文档推断一次标题、再共享给该文档的所有 Node，大致每篇文档一次而非每块一次。所以总开销并不是简单的 N×M——只有逐块抽取器才真正乘以 N。
Each extractor is a transformation that generates content from a prompt and writes it to a fixed metadata key (document_title / excerpt_keywords / questions_this_excerpt_can_answer / section_summary). But the call count splits in two: per-node extractors (KeywordExtractor / QuestionsAnsweredExtractor / SummaryExtractor) make one LLM call per Node, so cost grows linearly with the chunk count N; document-level extractors (e.g. TitleExtractor) infer one title per document and share it across that document's Nodes — roughly once per document, not once per chunk. So the total is not simply N×M — only the per-node extractors truly multiply by N.
🔀 替代方案🔀 Alternatives
确定性的字段（来源、路径、日期、作者）直接在 Reader 阶段手工写入 metadata，零成本也零误差；只有需要“理解内容”才能得到的字段（标题、关键词、能回答的问题）才值得花 LLM 去抽取。两者常常混用。
Deterministic fields (source, path, date, author) are best written by hand at the Reader stage — free and exact; only fields that require “understanding the content” (title, keywords, answerable questions) are worth an LLM extraction. In practice you mix both.
抽取器即“管道里的一道工序”extractors are just steps in the pipeline
from llama_index.core.extractors import TitleExtractor, QuestionsAnsweredExtractor
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.ingestion import IngestionPipeline

pipeline = IngestionPipeline(transformations=[
    SentenceSplitter(chunk_size=512),
    TitleExtractor(nodes=5),               # 用 LLM 推断标题
    QuestionsAnsweredExtractor(questions=3) # 该块能回答的 3 个问题
])
nodes = pipeline.run(documents=docs)
print(nodes[0].metadata)   # 多了 document_title / questions_this_excerpt_can_answer
手工元数据 + excluded_embed_metadata_keys：让 ID 之类字段不污染向量Hand-written metadata + excluded_embed_metadata_keys: keep fields like IDs out of the vector
from llama_index.core import Document
from llama_index.core.schema import MetadataMode

# 手工写入业务元数据（零 LLM 成本）
doc = Document(
    text='退款将在 7 个工作日内原路退回。',
    metadata={'source': 'faq.md', 'product': 'wallet', 'internal_id': 'X-92'})

# 控制元数据可见性：哪些不进 embedding、哪些不进给 LLM 的 prompt
doc.excluded_embed_metadata_keys = ['internal_id']
doc.excluded_llm_metadata_keys = ['internal_id']
print(doc.get_content(metadata_mode=MetadataMode.EMBED))  # internal_id 不污染向量
✅ 本课要点✅ Key points
  • 元数据是检索的第二通道：不止向量相似，还能按标签/来源过滤。Metadata is retrieval's second channel: beyond vector similarity, filter by tag/source.
  • QuestionsAnsweredExtractor 让“问题↔片段”更易对齐。QuestionsAnsweredExtractor aligns “question ↔ chunk” better.
  • 抽取器是 LLM 调用，有成本——按需启用。Extractors call the LLM, so they cost tokens — enable selectively.
💡 设计亮点💡 Design insight
Extractor 与 Splitter 都是 transformation，可任意串联——元数据增强变成管道里可插拔的一环。
Extractors and splitters are both transformations you can chain freely — metadata enrichment becomes a pluggable link in the pipeline.
