到这里，RAG 的输出一直是“一段自然语言”——给人读很顺，可一旦要把它接进下游程序（写进数据库、触发工作流、做条件判断），麻烦就来了：你得用正则、字符串切割去“猜”LLM 这次把金额写在哪一行、日期是什么格式。措辞一变、模型一升级，解析脚本就碎。这一课换一种拿结果的方式：别让 LLM 写一段话你再去 parse，直接让它产出一个 Pydantic 对象——字段和类型都是你预先声明的，模型必须照着填，框架还会校验。一句话：类型即契约，LLM 直接交回一个校验过的结构，下游 invoice.total 拿来就用，再没有脆弱的解析这一环。Up to here, RAG's output has always been “a piece of natural language” — nice for a human to read, but the moment you need to feed it into downstream code (write to a database, trigger a workflow, branch on a condition), the trouble starts: you must use regex and string-splitting to “guess” which line the LLM put the amount on and what format the date took. Change the wording or upgrade the model and the parsing script breaks. This lesson swaps the way you collect the result: instead of having the LLM write prose you then parse, have it emit a Pydantic object directly — the fields and types are declared up front by you, the model must fill them in, and the framework validates them. In a line: the type is the contract — the LLM hands back a validated structure, and downstream invoice.total just works, with the fragile parsing step gone entirely.

痛点：用正则/手工 parse 自由文本，脆弱易碎The pain: parsing free text with regex/by hand is brittle

朴素做法是让 LLM“用文字描述”结果，再写代码把字段抠出来：正则匹配“总计：¥1,200”、按冒号切“Vendor: Acme”…… demo 里跑得通，但有三个结构性脆点：① 措辞漂移——同一个 prompt，模型今天写“总计”、明天写“合计金额”，正则就漏了；② 格式不稳——日期一会儿“2024-01-05”、一会儿“Jan 5, 2024”，类型转换处处是坑；③ 静默失败——parse 不到时往往返回空串或默认值，错误一路混进下游，等发现时已经写进库。本质问题是：自由文本没有契约，你在用字符串处理去逼近一个本该是结构化的东西——解析脚本永远在追着模型的措辞跑。
The naive approach has the LLM “describe the result in words” and then writes code to dig the fields out: regex-match “Total: ¥1,200”, colon-split “Vendor: Acme”… it runs in a demo, but has three structural weak points: (1) wording drift — with the same prompt the model writes “Total” today and “Amount due” tomorrow, and the regex misses it; (2) unstable formats — a date is “2024-01-05” one time and “Jan 5, 2024” the next, so type conversion is a minefield; (3) silent failure — when a parse misses it often returns an empty string or a default, and the error slips downstream until it has already been written to the store. The root issue: free text has no contract, and you're using string handling to approximate something that should have been structured — the parser is forever chasing the model's phrasing.

Pydantic program：直接让 LLM 产出对象，类型即契约Pydantic program: have the LLM emit an object directly — the type is the contract

结构化输出把顺序倒过来：先用 Pydantic 的 BaseModel 把你要的形状写成一个类——Invoice 有 vendor: str、total: float、due_date: str——这个类就是契约。把它交给一个 program，program 负责把 schema 塞进给 LLM 的指令、解析模型回的内容、再用 Pydantic 校验，最后交还一个真正的 Invoice 对象。于是下游不再 parse 字符串，而是直接 invoice.total + tax：字段一定存在、类型一定正确，是被校验保证过的。LLM 仍然在“生成”，但它生成的目标从“一段话”变成了“一个对象”——你拿到的不再是待解析的文本，而是即用的数据。
Structured output flips the order: first write the shape you want as a Pydantic BaseModel class — Invoice has vendor: str, total: float, due_date: str — and that class is the contract. Hand it to a program, which injects the schema into the LLM's instructions, parses what the model returns, validates it with Pydantic, and hands back an actual Invoice object. Downstream no longer parses strings but writes invoice.total + tax directly: the field is guaranteed to exist and the type to be right, because it was validation-guaranteed. The LLM still “generates”, but the target of generation shifts from “a paragraph” to “an object” — what you get back is no longer text to parse but data ready to use.
声明 Pydantic 模型 → 交给 program → 拿回校验过的对象：下游 invoice.total 直接用，没有正则Declare a Pydantic model → hand it to a program → get back a validated object: downstream invoice.total just works, no regex
from pydantic import BaseModel
from llama_index.core.program import LLMTextCompletionProgram

class Invoice(BaseModel):
    vendor: str
    total: float
    due_date: str

program = LLMTextCompletionProgram.from_defaults(
    output_cls=Invoice,
    prompt_template_str="Extract the invoice fields from:\n{doc}",
)
invoice = program(doc=text)     # -> Invoice(vendor=..., total=..., due_date=...)

两条路：prompt 模板塞 schema vs 函数调用更稳Two routes: stuff the schema into a prompt vs function calling for more stability

落地有两条路，区别在“怎么逼模型守约”。① LLMTextCompletionProgram（纯 prompt 模板路线）：把 schema 的描述写进提示词，请模型“按这个 JSON 形状回”，再解析它回的文本。胜在不挑模型（任何会生成文本的 LLM 都能用），但约束是“软”的——模型可能多写一句解释、少个引号，解析就得容错、重试。② FunctionCallingProgram（函数调用路线）：把 Pydantic 模型当成一个“函数签名”交给模型原生的 function/tool calling 能力，模型在 API 层就被约束着按字段产出，结构更稳、更少跑偏，代价是要模型支持函数调用。经验法则：模型支持就优先函数调用，更稳；要兼容老模型或本地小模型才退回纯 prompt 模板——而 llm.structured_predict 正是把这条法则做成默认：它自动按模型能力选路（支持函数调用就用，否则回退到 prompt 模板），不用你手动二选一。一个易错点：structured_predict 收的是一个 PromptTemplate，不是裸字符串。
There are two routes in practice, differing in “how you force the model to honor the contract”. (1) LLMTextCompletionProgram (pure prompt-template route): write the schema's description into the prompt, ask the model to “reply in this JSON shape”, then parse the text it returns. Its win is being model-agnostic (any text-generating LLM works), but the constraint is “soft” — the model might add a sentence of explanation or drop a quote, so parsing needs tolerance and retries. (2) FunctionCallingProgram (function-calling route): hand the Pydantic model to the model's native function/tool-calling ability as a “function signature”, so the model is constrained to emit by field at the API level — more stable, less drift — at the cost of needing a model that supports function calling. Rule of thumb: prefer function calling when the model supports it; fall back to the pure prompt template only to support older or small local models — and llm.structured_predict bakes that rule in as the default: it auto-picks the route by model capability (function calling if supported, else falls back to the prompt-template route), so you don't choose by hand. One easy slip: structured_predict takes a PromptTemplate, not a bare string.

用途：信息抽取、表单填充、把 RAG 答案结构化Uses: information extraction, form-filling, structuring the RAG answer

结构化输出最实在的三类用途：① 信息抽取——把发票、合同、简历、邮件里的关键字段一次性抽成对象，替代成片的正则；② 表单填充——让 LLM 读一段对话或文档，直接产出表单/API 需要的参数对象，省掉中间的人工誊写；③ 把 RAG 答案结构化——这是和前面 30 课最直接的衔接：与其让 RAG 回一段话，不如让它回 {answer, sources, confidence}——answer 给人看，sources 让下游能溯源/展示引用，confidence 让程序能按阈值兜底（低置信就转人工或追问）。一旦答案有了结构，RAG 就从“生成一段文字”变成“返回一个可被程序消费、可校验、可监控的数据”——这才是把 RAG 接进真实系统的关键一步。
Structured output's three most concrete uses: (1) information extraction — pull the key fields from invoices, contracts, résumés, and emails into an object in one shot, replacing sheets of regex; (2) form-filling — have the LLM read a conversation or document and directly emit the argument object a form/API needs, skipping the manual transcription in between; (3) structuring the RAG answer — the most direct tie-in to the previous 30 lessons: rather than letting RAG reply with a paragraph, have it return {answer, sources, confidence} — answer for the human, sources so downstream can trace/show citations, and confidence so code can fall back on a threshold (escalate to a human or ask again when confidence is low). Once the answer has structure, RAG shifts from “generating text” to “returning data that programs can consume, validate, and monitor” — the key step to wiring RAG into a real system.

🧩 生活类比🧩 Analogy
与其让对方写一段话、你再从字里行间猜重点，不如直接递给他一张“表格”让他逐格填——栏目（字段）和格式（类型）都是你定好的，他只能照着填。结构化输出就是给 LLM 递表格：你先把 Invoice 的“厂商 / 金额 / 到期日”这几栏画好，模型不再自由发挥写小作文，而是把答案填进你的格子里，你拿到就能用。
Rather than having someone write a paragraph and then guessing the key points between the lines, hand them a “form” to fill in box by box — the columns (fields) and formats (types) are all set by you, and they can only fill them in. Structured output is handing the LLM a form: you draw the “vendor / total / due-date” columns of Invoice first, and instead of free-styling an essay the model drops its answer into your boxes, ready to use.

💡 设计亮点💡 Design insight
结构化输出的精髓，是把“解析自由文本”这件脆活，换成“契约化的类型”这件稳活——类型即契约。朴素 RAG 让 LLM 写一段话、你在下游用正则去猜字段，措辞一变就碎；结构化输出反过来：你先用 Pydantic 声明结果的形状，LLM 直接产出被校验过的对象，下游 .total 拿来就用，解析这一环整个消失。要记住三件事：① 两条路按“约束有多硬”分——纯 prompt 模板（软约束、不挑模型）vs 函数调用（硬约束、更稳、要模型支持），能用函数调用就优先，而 structured_predict 会自动按模型能力选路、替你执行这条法则；② 它不是免费午餐——校验失败仍要重试/修复，schema 太复杂模型也会填错，从简单字段做起；③ 它的工程价值是“让 LLM 的输出可被程序消费”——把 RAG 答案变成 {answer, sources, confidence} 才能接监控、接兜底、接工作流。而这一步——“把 Pydantic 模型当函数签名交给模型”——正是下一部分 Agent 的地基：工具调用的参数本质就是一个 schema，“产出受约束的结构”和“调用一个工具”是同一种能力。L32 的多智能体，就是踩着结构化输出这块砖往上走的。
The essence of structured output is swapping the brittle job of “parsing free text” for the stable one of “a contracted type” — the type is the contract. Plain RAG has the LLM write a paragraph and you guess fields downstream with regex, breaking when wording shifts; structured output flips it: you declare the result's shape with Pydantic, the LLM emits a validated object directly, downstream .total just works, and the parsing step vanishes entirely. Remember three things: (1) the two routes split by “how hard the constraint is” — a pure prompt template (soft, model-agnostic) vs function calling (hard, steadier, needs model support), and prefer function calling when you can, while structured_predict auto-picks the route by model capability for you; (2) it's no free lunch — validation can still fail and need retries/repair, and an over-complex schema gets misfilled, so start from simple fields; (3) its engineering value is “making the LLM's output program-consumable” — turning the RAG answer into {answer, sources, confidence} is what lets you wire in monitoring, fallbacks, and workflows. And that very move — “handing a Pydantic model to the model as a function signature” — is the bedrock of the next part's Agents: a tool call's arguments are themselves a schema, and “emitting a constrained structure” and “calling a tool” are the same ability. L32's multi-agent systems are built standing on this structured-output brick.

🔑 重点：校验失败就重试/修复而不是放行；能用 function-calling 模式就优先——它的约束来自模型原生能力，比纯 prompt 模板把 schema 写进提示更稳。① 校验是闸门：program 用 Pydantic 校验模型回的内容，不合格不放行——这一步本身就是“静默失败”的解药（对比 L31 痛点里手工 parse 的悄悄返回默认值）；② 失败就重试/修复：把校验错误回喂给模型让它改（“你少了 due_date 字段，请补全”），或限定重试次数后降级；③ 换更硬的约束：纯 prompt 模板是“软约束”，模型容易多写少写——改用 FunctionCallingProgram / structured_predict，让模型在函数调用层按字段产出，跑偏概率大降；④ schema 从简：字段越少越浅、类型越基础，模型越不容易填错；复杂结构先拆小再组合。一句话：校验当闸门、失败就重试、约束尽量硬、schema 尽量简。🔑 Key: on a validation failure, retry/repair rather than let it through; prefer the function-calling mode when you can — its constraint comes from the model's native ability, steadier than writing the schema into a pure prompt template. (1) Validation is the gate: the program validates the model's output with Pydantic and won't pass what fails — that step is itself the antidote to “silent failure” (vs the manual parse in L31's pain point that quietly returns a default); (2) retry/repair on failure: feed the validation error back to the model to fix it (“you're missing due_date, please complete it”), or degrade after a capped number of retries; (3) switch to a harder constraint: a pure prompt template is a “soft” constraint and the model easily over/under-writes — use FunctionCallingProgram / structured_predict so the model emits by field at the function-calling layer, sharply cutting drift; (4) keep the schema simple: fewer, shallower fields and basic types are harder to misfill; split complex structures small first, then compose. In a line: gate on validation, retry on failure, make the constraint as hard as you can, and keep the schema as simple as possible.

🔑 重点：本质是同一种机制——都让模型产出“受约束的结构”；工具调用的参数本身就是一个 schema，模型“按 Pydantic 模型填字段”和“按函数签名填实参”是同一件事——这正是 L32 agent 工具调用的地基。① 同一能力，两种用法：function-calling 路线的结构化输出，就是把你的 Pydantic 模型当成一个“函数签名”交给模型，让它在 API 层产出合规的字段——这和 agent “调用一个工具”时模型要产出“该工具的入参对象”是同一套机制；② 差别只在用途：结构化输出把这个“受约束的结构”当最终结果返回给你；agent 则把它当一次工具调用的参数、拿去执行、再把结果喂回循环；③ 所以是铺垫：你在本课学会“让模型可靠地产出一个schema 对象”，到 L32 只是把这个对象接上一个可执行的工具，模型就从“填表”升级成“调工具”。一句话：结构化输出是“静态的工具调用”，工具调用是“会执行的结构化输出”。🔑 Key: they're the same mechanism — both make the model emit a “constrained structure”; a tool call's arguments are themselves a schema, and “filling a Pydantic model's fields” and “filling a function signature's arguments” are one and the same — exactly the bedrock of L32's agent tool calling. (1) One ability, two uses: the function-calling route of structured output hands your Pydantic model to the model as a “function signature” so it emits conformant fields at the API level — the same machinery an agent uses when, to “call a tool”, the model must produce “that tool's argument object”; (2) the only difference is purpose: structured output returns that “constrained structure” to you as the final result; an agent treats it as the arguments of a tool call, runs the tool, and feeds the result back into the loop; (3) hence the setup: here you learn to “make the model reliably emit a schema object”, and by L32 you simply wire that object to an executable tool, upgrading the model from “filling a form” to “calling a tool”. In a line: structured output is a “static tool call”, and a tool call is “structured output that executes”.

🎯 3. 什么时候不该上结构化输出？给我一个会让它“看着省事、实则添乱”的反例。When should you not reach for structured output? Give me a counter-example where it “looks convenient but actually adds trouble”.
看参考要点Show key points 点击展开expand
🔑 重点：当输出本就是给人读的自由表达（解释、写作、开放对话），或 schema 复杂多变到模型频繁填错时，硬套结构化输出只会增加重试与维护成本——结构化的收益是“下游要程序化消费”，没有这个需求就别套。① 自由表达类：让模型“解释这段代码”“写一段安抚用户的话”，答案的价值在语言本身，塞进 {summary: str} 这种壳子既没增益、还可能截断表达；② schema 过重：字段几十个、深层嵌套、还常改——模型填错率高、重试烧钱，维护 schema 比写正则还累，这时不如拆小或只structured 关键几个字段；③ 判断标准：问一句“下游是程序消费还是人消费”——程序消费（写库、触发流程、按字段分支）才值得上；人消费就让它好好说话。一句话：结构化是为“机器读”服务的，别拿它去框“人读”的内容。🔑 Key: when the output is inherently free expression for a human to read (explanations, writing, open conversation), or the schema is so complex and volatile that the model misfills it often, forcing structured output only adds retry and maintenance cost — its payoff is “downstream needs to consume it programmatically”, so without that need, don't force it. (1) Free-expression cases: “explain this code”, “write a calming note to the user” — the value is in the language itself, and cramming it into a {summary: str} shell adds nothing and may truncate the expression; (2) over-heavy schema: dozens of fields, deep nesting, frequently changing — high misfill rate, costly retries, and maintaining the schema is wearier than writing regex; better to break it down or structure only the few key fields; (3) the test: ask “is downstream a machine or a human” — machine consumption (writing to a store, triggering a flow, branching on fields) earns it; human consumption should let the model just talk. In a line: structuring serves “machine reading” — don't box in content meant for “human reading”.
