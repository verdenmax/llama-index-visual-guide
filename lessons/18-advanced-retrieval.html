基础 top-k 不够用时，进阶检索登场：融合多查询（QueryFusionRetriever）、自动合并相邻块（AutoMergingRetriever）、递归（RecursiveRetriever）、按问题路由到不同库（RouterRetriever）。它们都实现同一个 BaseRetriever 接口。When plain top-k isn't enough, advanced retrieval steps in: fuse multiple queries (QueryFusionRetriever), merge adjacent chunks (AutoMergingRetriever), recurse (RecursiveRetriever), and route by question (RouterRetriever). All implement the same BaseRetriever interface.

🧩 生活类比🧩 Analogy
不只问一次、不只取一处：多角度改写再合并、把碎片拼回完整段落、按问题选对资料库。
Don't ask once or look in one place: rephrase from several angles and merge, stitch fragments back into a passage, and pick the right corpus per question.

基础 top-k 的三类短板，进阶检索逐一补齐Top-k's three weak spots — advanced retrieval patches each

把这些进阶检索当成一堆零散“高级技巧”去记，很容易迷失。换个视角：它们都在补朴素 top-k 相似度检索的三类典型短板——召回不全（换个问法本能命中的内容被漏掉）、命中碎片（块切得太碎，单个 Node 只剩半句上下文）、多库分流（不同问题其实该查不同索引）。看清这三道缺口，下面四种检索器就各有归属——其中“命中碎片”有横向合并、纵向递归两种补法；而且它们都实现同一个 BaseRetriever，可即插即换、互相嵌套。
Memorizing advanced retrieval as a pile of scattered “tricks” gets you lost. Flip the view: they all patch plain top-k's three classic weak spots — missed recall (a different phrasing would have hit, but it's dropped), fragmented hits (chunks so small a single Node holds half a thought), and multiple corpora (different questions belong to different indexes). Name those three gaps and the four retrievers below each find their place — “fragmented hits” gets two cures, merging sideways and recursing down — and since all implement one BaseRetriever, you swap or nest them freely.

🧪 示例：一题改写成 4 版🧪 Example: rephrase one question into 4
QueryFusionRetriever(retrievers=[base], num_queries=4)：原问题加上 LLM 生成的 3 个改写各自检索，再用 reciprocal rerank 融合成一份去重结果，召回通常明显高于单版 top-k。
QueryFusionRetriever(retrievers=[base], num_queries=4): the original question plus 3 LLM-generated rewrites each retrieve, then reciprocal rerank fuses them into one deduped result set — recall usually beats single-query top-k by a clear margin.
❓ 为什么这么设计❓ Why designed this way
所有检索器都实现同一个 BaseRetriever，统一返回 list[NodeWithScore]，于是它们能互相嵌套、随意替换，并直接塞进任何 QueryEngine——上层完全无感。
Every retriever implements the same BaseRetriever and returns list[NodeWithScore], so they nest and swap freely and drop straight into any QueryEngine — the layer above never notices.
⚙️ 内部怎么跑⚙️ How it runs inside
num_queries=n 让 LLM 额外生成 n-1 个改写（原问题也算一版）；融合方式由 FUSION_MODES 决定，默认 simple，常显式设 reciprocal_rerank（RRF）——按每条结果在多个排名中的倒数位次累加打分，再统一排序。
num_queries=n makes the LLM generate n-1 extra rewrites (the original counts as one); fusion is governed by FUSION_MODES (default simple), and you commonly set reciprocal_rerank (RRF) — it sums each result's reciprocal rank across the several rankings, then sorts once.
🔀 替代方案🔀 Alternatives
不是每个问题都靠“多问几遍”：碎片化严重就用 AutoMergingRetriever 合并回父块；知识有层级或引用就用 RecursiveRetriever 跟随引用到原文；面对多个索引就用 RouterRetriever 让 LLM 选库。它们可单用，也能叠在 fusion 之上。
Not every problem yields to “asking more times”: for heavy fragmentation use AutoMergingRetriever to merge back into parents; for hierarchical or cited knowledge use RecursiveRetriever to follow references to the source; for multiple indexes use RouterRetriever to let an LLM pick the corpus. Each works alone or stacked atop fusion.
进阶检索器仍是 BaseRetriever，可直接塞进 QueryEnginestill a BaseRetriever — drops into any QueryEngine
from llama_index.core.retrievers import QueryFusionRetriever

# 把一个问题改写成多版、分别检索，再用 reciprocal rerank 融合
fusion = QueryFusionRetriever(
    retrievers=[index.as_retriever(similarity_top_k=5)],
    num_queries=4,
    mode='reciprocal_rerank',
)
nodes = fusion.retrieve('退款和换货政策有何不同？')
print(len(nodes))
另一种思路：AutoMerging 把命中的碎片拼回完整父块A different angle: AutoMerging stitches hit fragments back into a whole parent
from llama_index.core.retrievers import AutoMergingRetriever
from llama_index.core.node_parser import HierarchicalNodeParser

# 层级切块：同一段文本切成大/中/小三层父子块
parser = HierarchicalNodeParser.from_defaults()
nodes = parser.get_nodes_from_documents(docs)
# ...把叶子块建进 index，得到一个基础 base_retriever...
retriever = AutoMergingRetriever(
    base_retriever,                  # 任意 BaseRetriever（这里检索叶子块）
    storage_context=storage_context,
)
# 命中同一父块的多个叶子，会被自动合并成更完整的父块
merged = retriever.retrieve('保修条款有哪些例外情况？')
print(len(merged))
✅ 本课要点✅ Key points
  • 进阶检索针对“召回不全/碎片化/多库”等基础 top-k 的短板。Advanced retrieval targets top-k's weak spots: missed recall, fragmentation, multiple corpora.
  • 它们都实现 BaseRetriever，可无缝接入现有 QueryEngine。All implement BaseRetriever, so they drop into an existing QueryEngine.
  • Fusion 用改写+融合提升召回；AutoMerging 用关系合并提升上下文完整度。Fusion lifts recall via rephrase+merge; AutoMerging lifts context completeness via relationships.
💡 设计亮点💡 Design insight
进阶检索没有免费的午餐：fusion 多跑几轮检索来提召回，但更慢更贵；auto-merging 合并父块补上下文，却可能引入冗余。每一种都是在用更多计算换更好结果。
Advanced retrieval is no free lunch: fusion runs extra retrieval passes to lift recall but costs more and runs slower; auto-merging merges parent blocks to restore context but can add redundancy. Each trades more compute for better results.

🎯 2. AutoMerging 检索和“直接调大 chunk_size”有何不同？什么时候用 Router？How is AutoMerging retrieval different from 'just use a bigger chunk_size', and when do you use a Router?
看参考要点Show key points 点击展开expand
🔑 重点：AutoMerging 用小块检索(精准)，命中同一父块的多个小块时再合并成父块喂 LLM(补上下文)——兼得精准与上下文。直接调大 chunk_size 则全程用大块，检索更糊、噪声更多。Router 用于多个 Index/数据源分流：用 selector 把问题路由到最合适的那个。🔑 Key: AutoMerging retrieves with small chunks (precise), then merges small chunks under the same parent into the parent for the LLM (context) — precision and context together. A bigger chunk_size uses large chunks throughout, blurring retrieval and adding noise. A Router is for routing across multiple indices/sources: a selector sends each question to the best one.
