到了生产环境，朴素 top-k 的“一路向量、直接取前 k”常常不够：精确编号查不准、罕见词召不回、真正相关的排在后面。生产级检索的标准答案是混合检索（向量 + BM25 两路召回）+ Rerank 精排，必要时再加 HyDE 改写——核心是把召回和精度分两步拿。In production, naive top-k — one vector path, take the first k — often isn't enough: exact ids miss, rare words don't get recalled, and the truly relevant hits rank too low. The standard answer is hybrid retrieval (vector + BM25 dual recall) + reranking, plus HyDE rewriting when needed — the core idea is to get recall and precision in two separate steps.

为什么生产 RAG 几乎都要混合检索 + RerankWhy production RAG almost always needs hybrid retrieval + rerank

纯向量检索懂语义——近义、换种问法都能命中，但它把文本压成一个语义向量，对精确 token 和罕见词（产品编号、报错码、专有名词）不敏感，这类查询常常排不进 top-k。BM25 正好相反：按字面词频匹配，精确编号一击即中，却读不懂同义改写。于是生产里把两路并联召回、用 RRF 融合，语义与字面互补。但召回求“全”难免带进噪声，所以再加一层 rerank：用交叉编码器把 query 和每个候选放在一起重新打分，把真正相关的提到最前——召回与精度分两步拿，这就是几乎所有生产 RAG 的检索骨架。
Pure vector search understands meaning — synonyms and rephrasings still hit — but it squeezes text into one semantic vector and is weak on exact tokens and rare words (product ids, error codes, proper nouns), so such queries often fall out of top-k. BM25 is the mirror image: it matches literal term frequency, nailing an exact id but blind to synonymy. So production runs both paths in parallel and fuses them with RRF, letting meaning and letter complement each other. Recall chasing completeness inevitably drags in noise, so a rerank layer follows: a cross-encoder scores the query and each candidate together, lifting the truly relevant to the top — getting recall and precision in two separate steps, the retrieval backbone of nearly every production RAG.

再进一步：HyDE 用“假设答案”贴近文档Going further: HyDE uses a “hypothetical answer” to match the docs

混合检索和 rerank 解决“召回 + 精排”，HyDE（Hypothetical Document Embeddings）则换个角度提升召回：先让 LLM 根据问题写一段假设答案，再用这段更像文档的文本去做向量检索。短问句和文档措辞往往对不齐，而“假设答案”天然贴近文档语气，召回常常更准。
Hybrid retrieval and rerank handle recall-plus-rerank; HyDE (Hypothetical Document Embeddings) lifts recall from another angle: let the LLM first draft a hypothetical answer to the question, then run vector search with that doc-like text. Terse questions rarely match document wording, but a hypothetical answer naturally echoes the document's tone, so recall is often better.

❓ 为什么这么设计❓ Why designed this way
一步到位很难同时做到“全”和“准”。生产 RAG 索性解耦：召回阶段只求别漏（向量 + BM25 多路、each top-10~20，宁可多召），精排阶段只求排得准（rerank 用更贵的交叉编码器在小候选集上细判）。两步各自优化，比单一相似度分数既快又准。
One pass can't be both complete and precise, so production RAG decouples them: the recall stage only avoids misses (vector + BM25, top-10~20 each, over-fetch on purpose), and the rerank stage only gets the order right (a pricier cross-encoder judges the small candidate set). Optimizing each step separately beats a single similarity score on both speed and accuracy.
⚙️ 内部怎么跑⚙️ How it runs inside
QueryFusionRetriever 把每路结果按倒数排名（RRF）累加打分再合并去重，num_queries=1 表示不改写问句、只融合多个检索器；rerank 不在检索器里，而是作为 node_postprocessor 挂在 QueryEngine 上，对融合后的候选逐条用 query+doc 一起打分、取 top-n。
QueryFusionRetriever merges each path's results by reciprocal rank (RRF), deduping as it goes; num_queries=1 means don't rewrite the query, just fuse the retrievers. Rerank lives not in the retriever but as a node_postprocessor on the QueryEngine, scoring each fused candidate with query+doc together and keeping the top-n.
🔀 替代方案🔀 Alternatives
精排器可换：CohereRerank（托管 API，最省事最准，但按调用计费且数据出网）、本地 SentenceTransformer 交叉编码器（SentenceTransformerRerank，数据不出网、免调用费，但要本地 GPU/算力）、LLMRerank（让 LLM 直接判相关性，最灵活但最慢最贵）。按数据合规、延迟预算和成本三角去选。
The reranker is swappable: CohereRerank (hosted API — easiest and most accurate, but billed per call and data leaves your network), a local SentenceTransformer cross-encoder (SentenceTransformerRerank — data stays in, no per-call fee, but needs local GPU/compute), and LLMRerank (an LLM judges relevance directly — most flexible but slowest and priciest). Pick along the data-compliance / latency / cost triangle.
生产配方：向量 + BM25 两路召回 → RRF 融合 → Cohere 精排 top-3Production recipe: vector + BM25 recall → RRF fusion → Cohere rerank to top-3
from llama_index.core.retrievers import QueryFusionRetriever
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.retrievers.bm25 import BM25Retriever            # pip install llama-index-retrievers-bm25
from llama_index.postprocessor.cohere_rerank import CohereRerank  # pip install llama-index-postprocessor-cohere-rerank

# 两路召回：向量(语义) + BM25(精确 token)，先各取 10
vector = index.as_retriever(similarity_top_k=10)
bm25 = BM25Retriever.from_defaults(docstore=index.docstore, similarity_top_k=10)

# 用 reciprocal-rank fusion 融合两路（num_queries=1 表示不改写、只融合检索器）
hybrid = QueryFusionRetriever([vector, bm25], num_queries=1,
                              similarity_top_k=10, mode='reciprocal_rerank')

# 再用 rerank 模型把 10 条精排到最相关的 3 条（需 COHERE_API_KEY）
engine = RetrieverQueryEngine.from_args(
    retriever=hybrid, node_postprocessors=[CohereRerank(top_n=3)])
print(engine.query('X-2000 的保修期是多久？'))
HyDE：先写“假设答案”，再用它的向量检索——更贴文档措辞HyDE: draft a hypothetical answer, then retrieve with its vector — closer to the doc's wording
from llama_index.core.indices.query.query_transform import HyDEQueryTransform
from llama_index.core.query_engine import TransformQueryEngine

# HyDE：先让 LLM 写一个“假设答案”，用它的向量去检索——往往比用问句本身更贴近文档措辞
hyde = HyDEQueryTransform(include_original=True)
engine = TransformQueryEngine(index.as_query_engine(similarity_top_k=5), query_transform=hyde)
print(engine.query('远程办公需要审批吗？'))
✅ 本课要点✅ Key points
  • 生产检索 = 广召回（向量 + BM25 混合）+ 精排（rerank），召回与精度分两步拿。Production retrieval = broad recall (vector + BM25 hybrid) + precise rerank — recall and precision in two steps.
  • 纯向量对精确编号 / 罕见词不敏感；BM25 补字面匹配，二者用 RRF 融合。Vectors miss exact ids / rare tokens; BM25 adds literal matching, fused via RRF.
  • Rerank 是 node_postprocessor（不是检索器），用交叉编码器把真正相关的提前。Rerank is a node_postprocessor (not a retriever) — a cross-encoder that lifts the truly relevant ones.
  • HyDE 用“假设答案”的向量检索，往往比原问句更贴文档措辞。HyDE retrieves with a hypothetical answer's vector, often closer to the document's wording than the raw question.
💡 设计亮点💡 Design insight
生产检索的精髓是把召回和精度拆开：先用混合检索“宁可多召、别漏”，再用 rerank“在小候选集里排得准”——一步求全、一步求准，比单一相似度分数又快又稳。
The essence of production retrieval is splitting recall from precision: hybrid retrieval first (over-fetch, never miss), then rerank (order the small candidate set precisely) — one step for completeness, one for correctness, beating a single similarity score on both speed and reliability.

🎯 3. 有人提议给检索加 HyDE。它的原理是什么、什么场景最有效、又有什么风险？你怎么决定上不上？Someone proposes adding HyDE to retrieval. How does it work, when does it help most, what's the risk, and how do you decide whether to adopt it?

🔑 重点：HyDE 先让 LLM 写一个“假设答案”，用这段更像文档的文本的向量去检索——当问句很短、措辞和文档差距大时，召回往往明显变好。风险：多一次 LLM 调用（更慢更贵），且假设答案若跑偏会把检索带偏（垃圾进、垃圾出）；所以 include_original=True 保留原问句一起检索更稳。决定上不上：在 gold 集上对比 HyDE 前后的命中率与端到端质量，再权衡多出的延迟/成本是否值得。🔑 Key: HyDE first has the LLM draft a “hypothetical answer”, then retrieves with that doc-like text's vector — when questions are terse and worded unlike the documents, recall often improves clearly.
