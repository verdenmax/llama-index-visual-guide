L26 已经把 RAG 从“写死的链”升级成“会决策的循环”——单个 agent + 一条最小 workflow。但任务一旦要跨好几种能力：先查资料、再写答复、最后核事实，把工具和规则全塞进一个 agent就会乱：工具一多越容易选错，system_prompt 越写越长越打架，出错还分不清是哪一环。这一课就把“一个全能选手”升级成“一支有分工、会交接的小队”，并让它的行进路线变成可检查的代码。L26 already upgraded RAG from a “hard-wired chain” to a “deciding loop” — one agent plus a minimal workflow. But once the task spans several capabilities — first search for facts, then write the reply, finally check them — cramming all those tools and rules into one agent gets messy: with many tools it more easily picks wrong, its system prompt grows long and self-contradictory, and on a failure you can't tell which part broke. This lesson upgrades “one all-round player” into “a small team with clear roles that hands off”, and makes that team's route inspectable code.

① 单 agent 不够：拆成多 agent 分工 + 交接(handoff)① One agent isn't enough: split into multiple agents with division of labor + handoff

这一课往前走两步：① 把一个全能 agent 拆成几个各管一段的专精 agent，用 handoff（交接）把任务往下传，由 AgentWorkflow 编排；② 再往底层落到 workflow，用 @step + 事件把分支 / 循环写成显式、可单测的控制流。下面分节展开，先看为什么单 agent 不够。
This lesson moves two steps: (1) split one do-everything agent into several specialized agents, each owning a slice, and use handoff to pass the task on, orchestrated by AgentWorkflow; (2) drop to a workflow underneath, where @step + events make branches / loops into explicit, testable control flow. The sections unpack these — starting with why one agent isn't enough.
L26 的单 agent 已经会“看问题、选工具、循环到够答”。但当任务要跨好几种能力，把工具和规则全堆在一个 agent 上，麻烦就来了：工具一多，它得在十几个里挑，选错的概率上升；system_prompt 要同时交代“怎么查、怎么写、怎么核”，越写越长、彼此打架，模型顾此失彼；真出了错，你也分不清是“查错了”还是“写跑了”。
L26's single agent already “reads the question, picks a tool, and loops until it can answer”. But when the task spans several capabilities, piling all the tools and rules onto one agent invites trouble: with many tools it must choose among a dozen, so the odds of a wrong pick rise; the system prompt has to spell out “how to search, write, and check” at once, growing long and self-contradictory until the model is pulled every way; and when something breaks you can't tell whether it “searched wrong” or “wrote off-track”.
解法是分工：把一个全能 agent 拆成几个各管一段的专精 agent，每个只配它那段需要的工具、只写它那段的 system_prompt——窄而精，自然更准、更好维护。它们之间靠 handoff（交接）协作：一个 agent 干完自己那段，把控制权连同上下文交给下一个更合适的 agent，像接力赛传棒（查→写→核三个角色就能串成一条线）。
The fix is division of labor: split one do-everything agent into several specialized agents that each own one slice, each wired only with the tools its slice needs and only its slice's system prompt — narrow and sharp, naturally more accurate and easier to maintain. They cooperate via handoff: one agent finishes its slice and passes control along with the context to the next, more suitable agent, like a relay baton (search→write→check threads three roles into one line).
📌
要点：handoff（交接）不是“关掉某个工具”，也不是“换个更大的模型”，而是把任务移交给队友。
The point: a handoff is not “turning off a tool” and not “switching to a bigger model” — it is handing the task to a teammate.

② AgentWorkflow 编排多 agent：name / description / system_prompt / tools / can_handoff_to② AgentWorkflow orchestrates the agents: name / description / system_prompt / tools / can_handoff_to

把这支小队编排起来的，是 AgentWorkflow。每个成员是一个 FunctionAgent，五样东西定义它的角色：name（队友靠名字点名交接）、description（一句话说明它负责什么，handoff 时别的 agent 据此判断要不要把任务交给它）、system_prompt（它这段的职责与风格）、tools（它能用的工具，撰稿 agent 可以一个都不带）、can_handoff_to（它允许把任务交给哪些 agent——交接路线是你显式声明的，不是模型乱传）。构造时把所有 agent 放进 agents=[...]，再指定 root_agent 作为入口（第一个接手的人）。
What orchestrates this team is AgentWorkflow. Each member is a FunctionAgent defined by five things: name (teammates hand off by name), description (one line on what it does — other agents use it to decide whether to hand the task to it), system_prompt (its slice's duty and style), tools (the tools it may use — a writing agent can carry none), and can_handoff_to (which agents it is allowed to pass the task to — the handoff routes are declared by you, not improvised by the model). You put all agents into agents=[...] and name a root_agent as the entry (whoever takes the task first).
运行 await workflow.run(user_msg=...)，root agent 先处理，需要时调用一次 handoff 把任务交给下一位——其实 handoff 在底层就是框架自动注入的一个工具，所以“交接”和 L26 的“调用工具”本质同源，只不过这次被调用的“工具”是另一个 agent。（这也正好踩在 L31 的砖上：让模型可靠产出“受约束的结构”，就是工具调用、也是 handoff 的地基。）
Calling await workflow.run(user_msg=...), the root agent handles it first and, when needed, invokes one handoff to pass the task on — under the hood a handoff is just a tool the framework auto-injects, so “handing off” and L26's “calling a tool” share the same root; it's only that this time the “tool” being called is another agent. (This stands on L31's brick: making the model reliably emit a “constrained structure” is the bedrock of both tool calls and handoffs.)
三个专精 agent + handoff 链：research 查完交给 write 写草稿、write 再交给 review 复核定稿；AgentWorkflow 编排，root_agent 是入口；多 agent 下每个都要 description（handoff 据此择人）Three specialized agents + a handoff chain: research gathers and hands to write for a draft, write then hands to review to finalize; AgentWorkflow orchestrates, root_agent is the entry; in a multi-agent workflow each agent needs a description (handoff targets are chosen by it)
from llama_index.core.agent.workflow import AgentWorkflow, FunctionAgent

research = FunctionAgent(name="research", description="查资料的研究员", tools=[search_tool], llm=llm,
                         system_prompt="查资料，把要点交给 writer", can_handoff_to=["write"])
write = FunctionAgent(name="write", description="根据要点写草稿的写手", tools=[], llm=llm,
                      system_prompt="据要点写成草稿，交给 reviewer", can_handoff_to=["review"])
review = FunctionAgent(name="review", description="复核定稿的把关人", tools=[], llm=llm,
                       system_prompt="复核草稿，通过则定稿", can_handoff_to=[])
workflow = AgentWorkflow(agents=[research, write, review], root_agent="research")
resp = await workflow.run(user_msg="写一段关于 X-2000 的简报")

③ 底层是 workflow：@step + 事件，把分支 / 循环显式写出来③ Underneath is a workflow: @step + events make branches / loops explicit

再往下看一层：L26 说过，agent 其实是一个内置了“工具循环”的 workflow。AgentWorkflow 把这套循环和 handoff 都封好了，开箱即用。但当你要的控制流不只是“循环调工具”——比如“琐碎问题直接答、复杂问题才检索”这种分支，或“没检索够就回到检索再来一轮”这种循环——就可以落到裸 Workflow，亲手把流程画清楚。
Drop one more layer: as L26 noted, an agent is really a workflow with a built-in “tool loop”. AgentWorkflow packages that loop and the handoffs for you, ready out of the box. But when the control flow you want is more than “loop and call tools” — say a branch like “answer trivial questions directly, retrieve only for complex ones”, or a loop like “if retrieval wasn't enough, go back and retrieve again” — you can drop to a raw Workflow and draw the flow by hand.
写法承接 L26 的最小 workflow：每个 @step 方法吃一种事件、返回一种事件，StartEvent 进、StopEvent 出。关键升级在返回类型：当一个 step 的返回标成 Retrieved | StopEvent（联合类型），就等于声明“我这步会分叉——可能往下检索，也可能直接收尾”，框架据此把分支连进流程图。
The style continues L26's minimal workflow: each @step method consumes one event and returns one, StartEvent in, StopEvent out. The key upgrade is the return type: when a step is annotated to return Retrieved | StopEvent (a union), it declares “this step forks — it may retrieve or finish right here”, and the framework wires that branch into the graph.
🔑
于是控制流不再藏在一长串 if/else 里，而是变成显式的事件流转：每个 step 是个纯函数式的小盒子（收什么、发什么一目了然），可以单独测试，分支和循环都画得出、测得到——这正是比“写死一个大函数”更可控的地方。
So control flow no longer hides inside a long if/else; it becomes an explicit flow of events: each step is a small functional box (clear about what it takes and emits), testable in isolation, with branches and loops you can draw and test — exactly what makes it more controllable than “one big hard-wired function”.
裸 Workflow：route 这个 @step 按问题难易返回不同事件（Retrieved 或 StopEvent）——分支被写成了可见、可测的控制流Raw Workflow: the route @step returns different events by difficulty (Retrieved or StopEvent) — the branch becomes visible, testable control flow
from llama_index.core.workflow import Workflow, step, StartEvent, StopEvent, Event

class Retrieved(Event):
    nodes: list

class RAGFlow(Workflow):
    @step
    async def route(self, ev: StartEvent) -> Retrieved | StopEvent:
        if is_trivial(ev.query):
            return StopEvent(result="直接答，无需检索")   # 分支：跳过检索
        return Retrieved(nodes=retrieve(ev.query))

④ 取舍：越多 agent / 越自由越强，但越难调、越贵④ Trade-off: more agents / more freedom is stronger, but harder to debug and pricier

能力越强，代价越实。每多一个 agent、每一次 handoff，都是又一次 LLM 决策：更多 token、更高延迟，还多了“会不会交错人、会不会在两个 agent 间反复踢皮球”这种新失败点。多 agent 也更难调——出了错你得先弄清“是哪个 agent、在哪一步、为什么这么交接”，这让 L23 的 trace 从“锦上添花”变成“必需品”。
The more capable, the realer the cost. Every extra agent and every handoff is another LLM decision: more tokens, higher latency, plus new failure modes — “did it hand off to the wrong one” and “are two agents ping-ponging the task back and forth”. Multi-agent is also harder to debug — on a failure you must first work out “which agent, at which step, and why it handed off that way”, which turns L23's tracing from a nice-to-have into a must-have.
⚠️
越多 agent、越自由越强，但每多一次 handoff，就多一次 LLM 决策——更慢、更贵、更难调。
More agents and more freedom make the system more capable, but each extra handoff is one more LLM decision — slower, pricier, harder to debug.
所以延续 L26 的那把尺子，按需选最低自主度：固定管道 → 单 agent → 多 agent / 裸 workflow，一级不够再上一级。判断要不要拆多 agent，就看任务能不能按角色清晰切开：能（查 / 写 / 核各管一段、要交接），拆了更清晰更好测；不能（其实一个角色几个工具就够），拆了只是徒增 handoff 的成本与不确定。一句话：先用最简单能解的那个。
So keep L26's yardstick and pick the lowest autonomy that works: fixed pipeline → single agent → multi-agent / raw workflow, climbing only when a rung isn't enough. To decide whether to split into multiple agents, ask whether the task cuts cleanly along roles: if it does (search / write / check each owning a slice, with handoffs), splitting is clearer and more testable; if it doesn't (one role with a few tools really suffices), splitting just adds handoff cost and uncertainty. In a line: use the simplest thing that solves it.

🧩 生活类比🧩 Analogy
单 agent 像一个样样都会的全能选手：一个人要查资料、写稿、还要自己校对，事情一多就顾此失彼。多 agent 像一支有分工的小队：调研员只管查、撰稿员只管写、审校员只管挑错，每人手里的活窄而精，干完自己那段就把接力棒（handoff）连同笔记交给下一位。而 workflow 就像这支小队写在白板上的流程图：先做什么、什么情况下跳过哪步、什么情况下返工重来——路线画明白了，哪一步出问题一眼就能定位，也能单独演练。
A single agent is like one all-round player who must research, draft, and proofread alone — overwhelmed once there's a lot to do. Multi-agent is like a team with clear roles: the researcher only gathers, the writer only writes, the reviewer only catches mistakes; each person's slice is narrow and sharp, and when done they pass the baton (handoff) — along with their notes — to the next. A workflow is that team's flowchart on a whiteboard: what to do first, when to skip a step, when to loop back and redo — with the route drawn out, you can spot exactly where a step breaks and even rehearse it in isolation.
✅ 本课要点✅ Key points

  • AgentWorkflow 编排多 agent：每个 FunctionAgent 有 name / description / system_prompt / tools / can_handoff_to，root_agent 是入口。AgentWorkflow orchestrates the agents: each FunctionAgent has name / description / system_prompt / tools / can_handoff_to, and root_agent is the entry.
  • agent 底层跑在 workflow 上；用 @step + 事件把分支 / 循环这类控制流显式写出来——看得见、每步可单测（联合返回类型即分支）。An agent runs on a workflow; use @step + events to make control flow like branches / loops explicit — visible and unit-testable per step (a union return type is a branch).
  • 取舍：agent 越多、越自由越强，但每次 handoff 多一次 LLM 决策——更慢、更贵、更难调；先用最简能解的（固定管道 → 单 agent → 多 agent）。Trade-off: more agents and more freedom are stronger, but each handoff is another LLM decision — slower, pricier, harder to debug; use the simplest thing that works (fixed pipeline → single agent → multi-agent).
💡 设计亮点💡 Design insight
多智能体与控制流的精髓，是把 L26 的“一个会决策的 agent”再拆开、再摊平：拆开成一支有分工、会 handoff 的小队（AgentWorkflow 编排，每个 agent 只精一段、按 can_handoff_to 把任务交给下一位），摊平成一张显式的事件流程图（裸 Workflow 用 @step + 事件把分支 / 循环写成看得见、可单测的控制流）。两步都在解同一个问题：当任务复杂到一个 agent 的提示词和工具都装不下时，与其把复杂性塞进一个黑箱，不如把它拆成可命名、可观测、可测试的小块。
The essence of multi-agent and control flow is to take L26's “one deciding agent” and both split it apart and flatten it out: split into a team with roles that hands off (AgentWorkflow orchestrates; each agent masters one slice and passes the task on per can_handoff_to), and flatten into an explicit event flowchart (a raw Workflow uses @step + events to write branches / loops as visible, unit-testable control flow). Both moves solve the same problem: when a task grows too complex for one agent's prompt and tools to hold, rather than stuffing the complexity into one black box, break it into named, observable, testable pieces.
但更强从来不是免费的——多一个 agent 就多一次 LLM 决策、多一处可能跑偏，调试也从“看一个 agent”变成“看一支队伍怎么交接”，因此越 agentic 越离不开 L23 的 trace。真正的判断不是“能不能上多 agent”，而是“这个任务配不配得上多 agent 的代价”：能按角色干净拆开、且确实需要交接，才拆；否则一个 agent、甚至一条固定管道，往往又快又稳。
But more power is never free — each extra agent is another LLM decision and another place to drift, and debugging shifts from “watching one agent” to “watching a team hand off”, so the more agentic you go, the more you lean on L23's tracing. The real call isn't “can we use multi-agent” but “is this task worth multi-agent's cost”: split only when the work cuts cleanly along roles and genuinely needs handoffs; otherwise a single agent — or even a fixed pipeline — is often faster and steadier.
还有一道绕不开的关：当这支自由的小队要去删数据、下订单、发邮件这类高风险动作时，光靠它们自己拍板就太冒险了——得有人能在回路里喊停、点头确认。那正是下一课 L33 人在回路（HITL）要补上的“刹车”。
And one gate is unavoidable: when this free-roaming team goes to delete data, place orders, or send email — high-risk actions — letting them decide alone is too risky; someone must be able to step into the loop, halt, and confirm. That's the “brake” the next lesson, L33 Human-in-the-Loop (HITL), adds.

答案：BAnswer: B handoff 是多 agent 协作的核心——一个 agent 干完自己那段，把控制权和上下文交给下一个专精 agent；它与关工具、压历史、换模型都无关。把任务“移交给更合适的 agent”才是它的本义：底层框架其实把 handoff 实现成一个自动注入的工具，被调用的“工具”是另一个 agent。“关闭某个工具的调用”是改工具集、“压缩对话历史省 token”是省上下文成本、“切换更大的模型”是换 LLM——三者都不是把任务交接给队友。A handoff is the core of multi-agent cooperation — one agent finishes its slice and passes control and context to the next, specialized agent; it has nothing to do with disabling a tool, compressing history, or switching models. Its real meaning is “handing the task to a more suitable agent”: under the hood the framework implements a handoff as an auto-injected tool whose “tool” is another agent. “Disable calling a tool” edits the toolset, “compress history to save tokens” trims context cost, and “switch to a bigger model” swaps the LLM — none of them hand the task off to a teammate.

🎯 2. 为什么 LlamaIndex 的 Workflow 要用“事件驱动的 @step”来表达控制流，而不是直接写一长串 if-else？这对调试和测试意味着什么？Why does LlamaIndex's Workflow express control flow with “event-driven @step” instead of a long chain of if-else? What does that mean for debugging and testing?
看参考要点Show key points 点击展开expand
🔑 重点：事件驱动把分支 / 循环从“藏在一个大函数里的 if-else”显式化成事件——每个 @step 收一种事件、发一种事件，于是每步都能单独单测、中间事件能流式观测，整条控制流画得出、测得到。① 控制流变成数据：一个 step 把返回类型标成 Retrieved | StopEvent，就等于把“这里会分叉”写进了类型；框架据此连图，分支不再埋在缩进里。② 每步可单测：每个 step 是“输入事件 → 输出事件”的纯函数式小盒子，可以单独喂一个事件、断言它发出的事件，不必把整条链跑起来——这是一长串 if-else 很难做到的。③ 可观测 / 可干预：事件在 step 间流转，框架能流式吐出中间事件，你能实时看到“走到哪一步、产出了什么”，也方便插入 / 重排 / 并行新步骤（如加一步校验或重排）。④ 一句话：if-else 把流程焊死在控制流里，事件把流程摊开成可观察的数据流——后者天然更好测、更好维护。🔑 Key: event-driven turns branches / loops from “if-else buried in one big function” into explicit events — each @step consumes one event and emits one, so every step is unit-testable in isolation, intermediate events are streamable / observable, and the whole control flow is drawable and testable. (1) Control flow becomes data: a step annotated to return Retrieved | StopEvent writes “this forks” into the type; the framework wires the graph from that, so the branch no longer hides in indentation. (2) Each step is unit-testable: every step is a pure-functional box of “input event → output event”, so you can feed it one event and assert the event it emits without running the whole chain — hard to do with a long if-else. (3) Observable / interceptable: events flow between steps, the framework can stream intermediate events, you see in real time “which step we're on and what it produced”, and it's easy to insert / reorder / parallelize steps (e.g. add a validation or re-rank step). (4) In a line: if-else welds the process into control flow, while events spread it into an observable data flow — the latter is naturally more testable and maintainable.

🎯 3. 多 agent 自由地互相 handoff，可能交接给错的 agent、来回踢皮球、甚至绕圈不收敛，把延迟和成本打爆。设计和上线时你怎么防，又怎么验证它不会失控？Agents handing off freely may hand off to the wrong agent, ping-pong back and forth, or even loop without converging, blowing up latency and cost. How do you prevent this in design and at launch, and how do you verify it won't run away?
看参考要点Show key points 点击展开expand
🔑 重点：给这支小队设“护栏”——用 can_handoff_to 收窄交接路线、设最大步数 / 超时 / 成本预算、做循环检测，并把高风险动作隔离（呼应 L25、铺垫 L33）；上线靠 trace + 一组“刁钻用例”回归来验证不失控。① 收窄路线：can_handoff_to 只声明确有必要的交接，别让每个 agent 都能交给所有人——路线越少越不容易乱。② 硬性上限：最大交接 / 步数、超时、token 预算，越界就停下并降级（转固定管道或礼貌兜底）。③ 循环检测：同样的 handoff 在两个 agent 间反复出现就熔断。④ 高风险动作隔离：删除、下单、发信这类绝不让 agent 自己拍板（这正是 L33 要补的人在回路）。⑤ 怎么验证：维护一组容易诱发多次交接的刁钻用例纳入回归，断言交接次数、成本、是否答对都在阈值内；线上监控每问步数与成本的 p95，越界即告警。🔑 Key: give the team “guardrails” — narrow handoff routes with can_handoff_to, set a max-step / timeout / cost budget, add loop detection, and isolate high-risk actions (echoing L25, setting up L33); verify with traces plus a regression suite of “tricky” cases. (1) Narrow the routes: have can_handoff_to declare only the handoffs that are genuinely needed; don't let every agent hand off to everyone — fewer routes, less chaos. (2) Hard caps: max handoffs / steps, timeout, token budget; on breach, stop and degrade (fall back to a fixed pipeline or a polite default). (3) Loop detection: trip a breaker when the same handoff recurs between two agents. (4) Isolate high-risk actions: deletion, ordering, sending mail must never be the agent's own call (exactly the human-in-the-loop L33 adds). (5) How to verify: keep a regression suite of cases that easily trigger many handoffs, asserting handoff count, cost, and correctness stay within thresholds; in production, monitor the p95 of steps and cost per question and alert on breach.
