L19 学会了给单个答案打分；到了生产和团队协作，评估必须规模化、还要自动守门。三件事：① 用 DatasetGenerator 从文档自动造一批金标问题（再人工挑高价值子集标注）；② 用 BatchEvalRunner 并发批量跑几十上百题、聚合成一个通过率；③ 把通过率接进 CI 做回归闸——低于阈值就 fail，坏改动合不进主干。一句话：把“凭感觉调”升级成“分数门槛守门”。L19 taught you to score a single answer; in production and on a team, evaluation must scale and gate automatically. Three moves: (1) use DatasetGenerator to auto-build a batch of gold questions from your docs (then hand-label a high-value subset); (2) use BatchEvalRunner to run dozens-to-hundreds of them concurrently and aggregate into one pass-rate; (3) wire that pass-rate into CI as a regression gate — below threshold the build fails and the bad change can't reach main. In a line: upgrade “tune by feel” into “a score threshold guards the door”.

🧩 生活类比🧩 Analogy
像工厂出货前的自动质检关：每批货（每次改动）都要过同一组固定抽检（回归金标集），合格才放行上线（合并主干），不合格当场拦下（回退）。检验标准事先定死、机器自动判，没人能靠“感觉还行”把次品偷偷放出去。
Like an automated QC gate before shipping: every batch (each change) must pass the same fixed sample inspection (the regression gold set); only a pass ships (merges to main), a fail is stopped on the spot (revert). The spec is fixed in advance and a machine rules on it — nobody can sneak a defect through on “feels fine”.

把“凭感觉调”变成“分数门槛守门”Turning “tune by feel” into “a score threshold guards the door”

评估要当“闸”用，前提是有一套稳定、可复跑的金标集。从零造太慢，所以先自动起步：DatasetGenerator 按文档块批量生成问题，几分钟就能铺出几十上百题的基线。但自动题良莠不齐，于是再做一层人工策展：挑出高价值、易回归的场景，补上人工核对的参考答案，沉淀成一个越用越准的回归子集。有了固定金标集，每次改动都对同一批题重跑、比趋势——这正是挡住“修一个坏一批”的关键：你盯着的那道题修好了，分数却告诉你另外十道悄悄退化了。最后把“低于阈值就 fail”写进 CI，守门这件事就不再依赖人的自觉。
Using evaluation as a gate requires a stable, re-runnable gold set. Building one from scratch is slow, so start automatically: DatasetGenerator mass-produces questions per document chunk, laying down a dozens-to-hundreds baseline in minutes. But auto-generated questions are uneven, so add a layer of human curation: pick the high-value, regression-prone scenarios, attach human-checked reference answers, and let them harden into a regression subset that grows more trustworthy with use. With a fixed gold set, every change re-runs against the same questions and you watch the trend — exactly what blocks “fix one, break many”: the case you were staring at got fixed, but the score tells you ten others quietly regressed. Finally write “below threshold → fail” into CI, and gatekeeping no longer depends on anyone's discipline.

一道闸怎么挡住“修一个坏一批”How one gate blocks “fix one, break many”

回归闸的价值不在“这次改得好不好”，而在“别处有没有被悄悄带坏”。把同一套金标接进 CI，单题的局部改进再也盖不住整体的退化——分数跌破阈值，改动就进不了主干。
A regression gate's value isn't “is this change good?” but “did it quietly break something elsewhere?”. Wire the same gold set into CI and a local single-question win can no longer mask an overall regression — break the threshold and the change can't reach main.

❓ 为什么这么设计（可度量闭环）❓ Why designed this way (a measurable loop)
L19 把单次改动变得“可度量”，生产要的是让度量自动守门。把通过率接进 CI，坏改动在合并前就被分数拦下，质量不再依赖谁记得手动跑评估——“修一个坏一批”的回归被一道常态化的闸挡住。免参考的忠实度/相关性最适合当这道闸（不需要参考答案、能完全自动化）。
L19 made one change “measurable”; production wants the measurement to gate automatically. Wire pass-rate into CI and a bad change is blocked by the score before merge — quality no longer depends on who remembers to run evals by hand, and the “fix one, break many” regression is held back by a standing gate. Reference-free faithfulness/relevancy fit this gate best (no gold answers needed, fully automatable).
⚙️ 内部怎么跑（并发 · EvaluationResult.passing）⚙️ How it runs inside (concurrency · EvaluationResult.passing)
BatchEvalRunner 用 workers 控制并发度，底层走 async：先用 QueryEngine 批量产生答案，再把每个 (query, response, source_nodes) 喂给各评估器（多为 LLM-as-judge）。返回是 dict[str, list[EvaluationResult]]，每个 EvaluationResult 带 passing（是否达标）与 score；把一列 passing 求平均就是通过率，CI 闸比的就是它。
BatchEvalRunner uses workers to set concurrency, async under the hood: it first batch-produces answers via the QueryEngine, then feeds each (query, response, source_nodes) to the evaluators (mostly LLM-as-judge). It returns a dict[str, list[EvaluationResult]]; each EvaluationResult carries passing (met the bar) and score — averaging a column of passing gives the pass-rate the CI gate compares against.
🔀 替代方案（自动生成 vs 人工标注 vs 线上反馈）🔀 Alternatives (auto-gen vs human labeling vs prod feedback)
金标从哪来，有三条路：自动生成（DatasetGenerator，最快铺量，但题目参差、可能问得很浅）；人工标注（最准、能给参考答案支撑 Correctness，但慢且贵，只配给高价值子集）；线上真实反馈（用户点踩、人工申诉、客服转人工的会话——最贴近真实分布，但要脱敏与清洗）。生产里三者叠用：自动起量、人工提纯、线上补真实长尾。
Gold data comes from three sources: auto-generation (DatasetGenerator — fastest to scale, but uneven and often shallow questions); human labeling (most accurate, supplies the reference answers Correctness needs, but slow and costly — reserve it for a high-value subset); and real production feedback (thumbs-down, escalations, handed-off chats — closest to the true distribution, but needs redaction and cleanup). Production stacks all three: auto for volume, humans for purity, production for the real long tail.
造集 + 并发批量评估：DatasetGenerator 自动出题 → BatchEvalRunner（workers=8）并发跑 50 题Build the set + batch-evaluate concurrently: DatasetGenerator drafts questions → BatchEvalRunner (workers=8) runs 50 in parallel
from llama_index.core.evaluation import (DatasetGenerator, BatchEvalRunner,
                                         FaithfulnessEvaluator, RelevancyEvaluator)

# 1) 自动造金标问题（生产里再人工挑选/标注更可靠的子集）
questions = DatasetGenerator.from_documents(
    docs, num_questions_per_chunk=2).generate_questions_from_nodes()

# 2) 并发批量评估：忠实度 + 相关性
runner = BatchEvalRunner(
    {'faithfulness': FaithfulnessEvaluator(), 'relevancy': RelevancyEvaluator()}, workers=8)
results = await runner.aevaluate_queries(index.as_query_engine(), queries=questions[:50])
把通过率变成 CI 回归闸：低于阈值就 assert 失败、挡住合并Turn pass-rate into a CI regression gate: below threshold, assert fails and the merge is blocked
# 把评估变成 CI 回归闸：均通过率低于阈值就 fail（放进 pytest / CI 脚本）
def pass_rate(rs):
    return sum(r.passing for r in rs) / len(rs)

faith = pass_rate(results['faithfulness'])
print(f'faithfulness pass-rate: {faith:.0%}')
assert faith >= 0.9, '忠实度回退，拦截本次变更'   # 守门：不达标就别合并
✅ 本课要点✅ Key points
  • 规模化评估 = 自动造金标（DatasetGenerator）+ 并发批量评（BatchEvalRunner）+ CI 回归闸（通过率阈值），把评估从手动升级成自动守门。Evaluation at scale = auto-built gold (DatasetGenerator) + concurrent batch eval (BatchEvalRunner) + a CI regression gate (pass-rate threshold) — manual evaluation upgraded to automatic gatekeeping.
  • 固定回归金标集是闸的前提：自动生成起量、人工标注高价值子集，挡住“修一个坏一批”。A fixed regression gold set is the gate's prerequisite: auto-generate for volume, hand-label a high-value subset — blocking “fix one, break many”.
  • BatchEvalRunner 用 workers 并发跑、按指标聚合 EvaluationResult.passing；通过率就是闸比对的数。BatchEvalRunner runs concurrently via workers and aggregates EvaluationResult.passing per metric; the pass-rate is what the gate compares.
  • 免参考的忠实/相关最适合当 CI 闸；正确性需参考答案，留给人工金标子集。Reference-free faithfulness/relevancy fit the CI gate best; correctness needs references — reserve it for the human gold subset.
💡 设计亮点💡 Design insight
规模化评估的精髓是把“评估”变成“闸”：一套固定可复跑的金标集 + 一个通过率阈值，让每次改动都必须用同一把尺子自证没有变差——质量从“谁记得测”升级成“不达标就合不进来”。
The essence of evaluation at scale is turning “evaluation” into a gate: a fixed, re-runnable gold set plus a pass-rate threshold forces every change to prove — against the same ruler — that it didn't regress; quality goes from “whoever remembers to test” to “below the bar, it can't get in”.

🔑 重点：金标集先自动生成起量、再人工策展高价值子集；阈值用基线分布定、且只卡“相对回退”；judge 误判靠固定 judge+rubric、看趋势、留容忍带；闸的有效性要拿“已知好/坏改动”去回归测试它。① 造集：DatasetGenerator 自动铺几十上百题，人工挑高价值、易回归的场景补参考答案，沉淀成固定回归集；② 定阈值：先跑基线得到通过率分布，阈值卡在“比基线明显下降”（如低于基线 5 个点就 fail），而非拍一个绝对数；③ 误判：固定 judge 模型与 prompt、用二值 rubric、对边界样本人评校准，关注整体通过率趋势而非单题抖动；④ 验证闸本身：故意提交一个已知会变差的改动（把 chunk 改坏 / 砍掉 rerank），闸必须变红；再提交一个已知无害的改动，闸必须放行——闸也要回归测试，否则它只是个绿灯摆设。🔑 Key: auto-generate the gold set for volume then human-curate a high-value subset; set the threshold from the baseline distribution and gate on relative regression; tame judge misfires with a pinned judge+rubric, trends and a tolerance band; and prove the gate works by regression-testing it with known-good/known-bad changes. (1) Build: DatasetGenerator lays down dozens-to-hundreds, then hand-pick high-value, regression-prone cases and add reference answers, hardening a fixed regression set; (2) threshold: run a baseline for the pass-rate distribution and gate on “clearly below baseline” (e.g. fail if it drops 5 points), not an arbitrary absolute number; (3) misfires: pin the judge model and prompt, use a binary rubric, calibrate edge cases with human review, and watch the overall pass-rate trend rather than single-question jitter; (4) verify the gate: deliberately submit a known-bad change (break chunking / drop the reranker) and the gate must go red, then a known-harmless one and it must pass — the gate itself needs regression testing, or it's just a green rubber stamp.

🎯 2. 有同事说“金标集直接用 DatasetGenerator 自动生成就行，省得人工标注”。你同意吗？自动题有什么坑，你会怎么把这套金标做得可信、可长期当闸用？A teammate says “just auto-generate the gold set with DatasetGenerator — skip human labeling”. Do you agree? What are the pitfalls of auto-generated gold, and how would you make the set trustworthy enough to gate on long-term?
看参考要点Show key points 点击展开expand
🔑 重点：自动生成适合“快速起量”，但不能直接当唯一金标——它问得浅、可能脱离真实分布，还无法支撑需要参考答案的 Correctness。坑：① 题目多是“文档里直接抄得到”的浅问，测不出真实难例；② 分布偏向文档密集处，真实高频问法反而漏；③ 没有人工核对的参考答案，Correctness 用不了，且自动题里混着噪声会让闸“误判”。做可信：自动生成只当草稿池，人工挑选 + 改写 + 标注出一个高价值子集，再掺入线上真实问题（点踩/转人工的会话）补真实长尾；CI 闸主要用免参考的忠实/相关跑全量草稿、用人工金标子集跑 Correctness。怎么验证金标本身够好：让它能区分已知的好坏配置（好配置分明显更高），区分不出就说明题目没区分度，得继续提纯。🔑 Key: auto-generation is great for fast volume but can't be the sole gold — it asks shallow questions, can drift from the real distribution, and can't support reference-needing Correctness. Pitfalls: (1) questions are mostly “copy-able straight from the doc”, missing real hard cases; (2) the distribution skews to doc-dense areas while real high-frequency phrasings are missed; (3) without human-checked references Correctness is unusable, and noise in auto questions makes the gate “misfire”. To make it trustworthy: treat auto-generation as a draft pool, have humans select + rewrite + label a high-value subset, and mix in real production questions (thumbs-down / handed-off chats) for the true long tail; the CI gate runs reference-free faithfulness/relevancy over the full draft and Correctness over the human subset. How to check the gold itself is good: it should discriminate known good vs bad configs (good scores clearly higher); if it can't, the questions lack discriminative power and need more curation.
🎯 3. 评估是 LLM-as-judge，搬进 CI 后同一个 PR 重跑两次分数会抖，构建一会儿红一会儿绿（flaky）。这种“闸不稳定”你怎么定位和治理？还要不要把它当硬性合并条件？Evaluation is LLM-as-judge; in CI the same PR re-run twice gives jittery scores and a flaky red/green build. How do you diagnose and tame this instability — and should it still be a hard merge requirement?

🔑 重点：抖动来自 judge 的随机性和小样本噪声；靠“降随机 + 增样本 + 留容忍带 + 看趋势”治理，硬闸只卡明显回退、细微抖动转为告警。① 降随机：judge 用 temperature=0、固定模型版本与评判 prompt、二值 rubric，减少不确定性；② 增样本：题太少则单题翻转就让通过率大跳，扩大回归集让比例稳定；③ 容忍带：阈值不卡“等于基线”，而是“低于基线一个明显幅度”才 fail，给噪声留缓冲；④ 分层：把昂贵/不稳的 Correctness 放夜间离线跑，CI 硬闸只用稳定、免参考的忠实/相关；⑤ 看趋势：连续多次低于阈值才阻断，单点抖动只告警。验证治理是否见效：固定一个不动的 PR 重跑 N 次，统计通过率方差，方差落进容忍带以内才算稳。🔑 Key: jitter comes from judge randomness and small-sample noise; tame it with “less randomness + more samples + a tolerance band + trend-watching”, and let the hard gate fire only on clear regressions while small jitter becomes a warning. (1) Less randomness: run the judge at temperature=0, pin the model version and judging prompt, use a binary rubric; (2) more samples: with too few questions one flip swings the pass-rate, so enlarge the regression set to stabilize the ratio; (3) tolerance band: gate on “clearly below baseline”, not “equal to baseline”, leaving room for noise; (4) tiering: run the costly/unstable Correctness offline nightly and keep only stable, reference-free faithfulness/relevancy as the hard CI gate; (5) trend: block only after several consecutive sub-threshold runs, warn on a single dip. Verify the fix by re-running one unchanged PR N times and measuring pass-rate variance — stable once it falls within the tolerance band.
