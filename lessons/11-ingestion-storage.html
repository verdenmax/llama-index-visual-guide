IngestionPipeline 把切块、抽取、向量化串成一条可缓存、可去重的管道，重复运行只处理变化的部分。索引建好后，再用 StorageContext 的 persist / load_index_from_storage 把它整体落盘、下次秒开，免去每次重建。An IngestionPipeline chains splitting, extraction and embedding into a cacheable, dedup-aware pipeline, so re-runs only touch what changed. Once the index is built, StorageContext's persist / load_index_from_storage write it to disk as a whole and reload it instantly — no rebuild next time.

🧩 生活类比🧩 Analogy
Ingestion 像自动化装配线（切块→抽取→向量化），缓存让“已加工的零件”不重做；persist 像把建好的图书馆整体存档，下次直接开门营业。
Ingestion is an assembly line (split→extract→embed); the cache skips “parts already made”; persist archives the finished library so next time you just open the doors.

把建索引做成幂等、可缓存的管道Make indexing an idempotent, cacheable pipeline

demo 里“每次重跑都全量重建”尚可忍受，生产里却致命：数据天天变，全量重算既慢又贵。IngestionPipeline 把摄取做成幂等管道——run 时对每个转换的输入算内容哈希，命中缓存就跳过；配上 docstore，相同文档不重复处理，只有变化的文档才走完整流程。再用 persist / load 把成品索引落盘，重启免重建。这三件事合起来，才让 RAG 从“能跑通”走到“可维护”。
“Rebuild everything on every run” is tolerable in a demo but fatal in production: data changes daily and full recomputation is slow and costly. IngestionPipeline makes ingestion idempotent — on run it hashes each transformation's input and skips on a cache hit; paired with a docstore, identical documents aren't reprocessed and only changed docs run the full path. Then persist / load put the finished index on disk so restarts skip the rebuild. Together these three take RAG from “it runs” to “it's maintainable”.

🧪 示例：带 docstore 的 IngestionPipeline🧪 Example: an IngestionPipeline with a docstore
IngestionPipeline(transformations=[...], docstore=SimpleDocumentStore()) 第一次 run 会完整处理所有文档；之后再 run，未变化的文档直接命中缓存/去重，只有新增或改动的文档才重新加工。
IngestionPipeline(transformations=[...], docstore=SimpleDocumentStore()) processes every document on the first run; on later runs unchanged documents hit the cache/dedup, and only new or modified documents are reprocessed.
❓ 为什么这么设计❓ Why designed this way
生产数据是活的：每天只有一小部分变化。幂等 + 缓存 + 去重让“重建索引”的代价正比于变化量而非总量，既省钱又省时，还保证重复运行结果一致。
Production data is alive: only a small slice changes each day. Idempotency + cache + dedup make the cost of “rebuilding the index” proportional to the delta, not the total — cheaper, faster, and consistent across repeated runs.
⚙️ 内部怎么跑⚙️ How it runs inside
run 时，管道对每个转换的（输入 Node + 转换配置）算一个哈希作为缓存键，命中就复用上次结果；docstore 则按 doc_id 记录每个文档的内容哈希，配合 DocstoreStrategy 决定是 UPSERT 还是跳过。
On run, the pipeline hashes each transformation's (input Nodes + config) into a cache key and reuses last time's output on a hit; the docstore records each document's content hash by doc_id and, via a DocstoreStrategy, decides whether to UPSERT or skip.
🔀 替代方案🔀 Alternatives
不用管道也能跑：每次都 from_documents 全量重建。小数据时无所谓，但随着库变大，全量重算的时间和 embedding 费用会线性膨胀——这正是增量摄取存在的理由。
You can skip the pipeline and just from_documents from scratch every time. Fine for small data, but as the corpus grows the recompute time and embedding bill scale linearly — exactly why incremental ingestion exists.
建一次、存起来、反复用build once, persist, reuse
from llama_index.core import VectorStoreIndex, StorageContext, load_index_from_storage
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.storage.docstore import SimpleDocumentStore

# 可缓存 + 去重的摄取管道：相同输入不重算，相同文档不重复处理
pipeline = IngestionPipeline(
    transformations=[SentenceSplitter(chunk_size=512)],
    docstore=SimpleDocumentStore(),   # 配 docstore 才能按文档去重做增量
)
nodes = pipeline.run(documents=docs)
index = VectorStoreIndex(nodes)

# 落盘，下次秒开
index.storage_context.persist(persist_dir='./storage')
index2 = load_index_from_storage(StorageContext.from_defaults(persist_dir='./storage'))
第二次 run 只处理变化的文档，其余命中缓存/去重——这就是增量摄取The second run touches only changed docs; the rest hit cache/dedup — that's incremental ingestion
from llama_index.core import Document
from llama_index.core.ingestion import IngestionPipeline, IngestionCache
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.storage.docstore import SimpleDocumentStore

pipeline = IngestionPipeline(
    transformations=[SentenceSplitter(chunk_size=512)],
    docstore=SimpleDocumentStore(),   # 去重：按 doc_id + 内容哈希判断是否变化
    cache=IngestionCache(),           # 缓存：相同输入直接复用上次结果
)

docs = [Document(text='退款政策……', doc_id='faq-1')]
print(len(pipeline.run(documents=docs)))   # 首次：完整处理

docs[0].text = '退款政策（已更新）……'        # 只改这一篇
print(len(pipeline.run(documents=docs)))   # 再跑：仅这篇重算，其余命中缓存
✅ 本课要点✅ Key points
  • IngestionPipeline 让“建索引”变成幂等、可缓存的管道。IngestionPipeline makes “building the index” an idempotent, cacheable process.
  • docstore 去重支持增量更新：只处理变化的文档。docstore dedup enables incremental updates: only changed docs are processed.
  • persist / load_index_from_storage 避免每次启动都重建。persist / load_index_from_storage avoid rebuilding on every startup.
💡 设计亮点💡 Design insight
把摄取做成幂等可缓存的管道，是 RAG 从“demo 跑通”走向“生产可维护”的关键一步——数据天天变，但只重算变化的部分。
Making ingestion an idempotent, cacheable pipeline is the step that takes RAG from “demo works” to “maintainable in production” — data changes daily, but only the deltas are recomputed.

🔑 重点：默认三件套——docstore(Node 内容)、index store(索引结构)、vector store(向量)。换生产库后，向量由外部数据库托管，本地不再是 vector_store.json；加载时 StorageContext 必须连回同一个库，且查询用与建索引相同的 embedding 模型。🔑 Key: by default three pieces — docstore (Node content), index store (index structure), vector store (vectors). With a production store the vectors live in an external DB, not a local vector_store.json; on load the StorageContext must reconnect to the same store, and queries must use the same embedding model the index was built with.
