全书最后一课，RAG 的零件你都见过了——只有最底层那个零件，我们一直原样拿来用：embedding 模型。它把文字变成向量、决定“谁和谁相近”，是检索的地基。通用 embedding（如 bge）在大众语料上训练，泛泛够用；可一进专业领域——医疗、法律、公司黑话——就常语义错位：本该相近的概念离得老远，召回不到位，后面 rerank、合成再好也救不回。这一课讲最后一招：用自家文档造 (问题, 相关块) 对、微调 embedding，把领域语义校准进向量空间。At the book's final lesson, you've met the parts of RAG — yet one foundational part we kept using as-is throughout: the embedding model. It turns text into vectors and decides “who is near whom” — the bedrock of retrieval. Generic embeddings (e.g. bge) train on broad corpora and serve general cases well; but step into a specialized domain — medicine, law, in-house slang — and they often misalign: concepts that should be near sit far apart, retrieval under-recalls, and no downstream rerank or synthesis can recover it. This lesson covers the last resort: use your own documents to build (question, relevant-chunk) pairs and fine-tune the embedding, calibrating domain semantics into the vector space.

① 痛点：通用 embedding 在专业领域“语义错位”，召回从地基就漏① The pain: generic embeddings “misalign” in a specialized domain — retrieval leaks from the foundation

动手前先记两条前提：① 微调成本高、易过拟合，该是穷尽更便宜手段之后的最后一步、不是第一反应（详见 ④）；② llama-index-finetuning 在 core 之外（pip install llama-index-finetuning），本课代码是示例，重点在思路与判断、不在 API 细节。
Two caveats before diving in: (1) fine-tuning is costly and overfit-prone, a last step after exhausting cheaper means rather than a first reflex (see ④); (2) llama-index-finetuning lives outside core (pip install llama-index-finetuning); the code here is illustrative — the point is the reasoning and judgment, not API details.
embedding 是检索的地基：它决定每段文字落在向量空间的哪个位置，而 top-k 召回只看谁离查询近。通用 embedding 在大众语料上训练，专业领域的词在它眼里要么稀薄、要么歧义。举两个例子：缩写 “MI” 在通用语料里更像 Michigan 或 Mutual Information，可在心内科它是“心肌梗死”；你公司内部的项目代号、产品简称、黑话，通用模型压根没见过。结果是：用户拿领域术语去问，真正相关的块因为向量不相近而排不进 top-k——召回阶段就把答案漏在了门外。
The embedding is retrieval's foundation: it fixes where each piece of text lands in vector space, and top-k recall only looks at what's nearest the query. Generic embeddings train on broad corpora, where domain words are either thin or ambiguous. Two examples: the acronym “MI” in general text reads as Michigan or Mutual Information, but in cardiology it means “myocardial infarction”; your company's project codenames, product nicknames, and slang the generic model has simply never seen. The result: a user asks with domain terms, and the truly relevant chunk — because its vector isn't near — never enters the top-k, so retrieval leaves the answer outside the door.
📌
embedding 是检索的地基：召回阶段漏掉的块，再聪明的 rerank / 合成也救不回——要修就得修地基。
The embedding is retrieval's foundation: a chunk missed at recall can't be recovered by any clever rerank / synthesis — to fix it you must fix the foundation.
L21 的 rerank 只能在已召回的候选里重排，召回时就没捞上来的块，rerank 给不出来；合成器再聪明，也只能拿手里这几个错块去拼。所以当你用 L23 的 trace + L12 的检索指标（hit-rate / MRR）确认错就错在“检索召回”、且根因是领域语义错位时，才轮到微调。
L21's rerank can only reorder already-retrieved candidates, and a chunk never recalled can't be re-ranked into existence; however clever the synthesizer, it can only stitch from the wrong chunks it holds. So only when L23's traces + L12's retrieval metrics (hit-rate / MRR) confirm the fault is in “retrieval recall” and the root cause is domain misalignment does fine-tuning's turn arrive.

② 思路：用自家 QA 对当“信号”，把正样本拉近、负样本推远② The idea: use your own QA pairs as the “signal” — pull positives closer, push negatives apart

微调 embedding 的本质是对比学习：喂给模型一批“这条问题和这个块是相关的（正样本）”的例子，让它微调权重，把相关的问题与块在向量空间里拉近，同时把不相关的推远。难点只有一个：哪来这么多标注好的 (问题, 相关块) 对？答案很巧——用你自家的文档自动造。
Fine-tuning an embedding is at heart contrastive learning: feed the model a batch of “this question and this chunk are relevant (a positive)” examples and let it nudge its weights to pull related questions and chunks closer in vector space while pushing unrelated ones apart. There's only one hard part: where do all those labeled (question, relevant-chunk) pairs come from? The neat answer — auto-build them from your own documents.
把文档切成块（你早在 L06 就会），对每个块让一个 LLM 反向生成“这个块能回答的问题”——于是 (生成的问题, 这个块) 天然是一对正样本，同一批里的其它块自然充当负样本。这正是 generate_qa_embedding_pairs(nodes, llm=...) 做的事：喂进 nodes，吐出一个可直接用于微调的 embedding QA 数据集。
Chunk the docs (you've done this since L06), and for each chunk have an LLM reverse-generate “a question this chunk can answer” — so (generated question, this chunk) is naturally a positive, and the other chunks in the batch serve as negatives. That is exactly what generate_qa_embedding_pairs(nodes, llm=...) does: feed in nodes, get back an embedding QA dataset ready for fine-tuning.
要清醒的是：这些问题是机器造的，质量参差，应当自动生成打底 + 人工抽检过滤 + 补难负样本（详见本课面试题）——数据质量直接决定微调的上限，“垃圾进”只会“垃圾出”。
Stay clear-eyed: these questions are machine-made and uneven, so you should auto-generate as a base + human spot-check to filter + add hard negatives (see this lesson's interview drills) — data quality sets the ceiling of fine-tuning, and “garbage in” only yields “garbage out”.

③ 流程：造对 → 微调 → 换模型重建索引 → 用 L19/L22 量化提升③ The flow: build pairs → fine-tune → rebuild the index with the new model → quantify the gain via L19/L22

整条链是一个四步闭环，每一步都有 llama-index-finetuning（core 之外，示例）的对应符号。① 造对：generate_qa_embedding_pairs(nodes, llm=...) 自动产出 QA 数据集（见 ②）。
The whole chain is a four-step loop, each step with a symbol from llama-index-finetuning (outside core, illustrative). (1) Build pairs: generate_qa_embedding_pairs(nodes, llm=...) auto-produces the QA dataset (see (2)).
② 微调：SentenceTransformersFinetuneEngine(dataset, model_id=基座, model_output_path=...) 在你的 QA 对上接着训练一个开源 sentence-transformer（如 bge-small），.finetune() 跑完、get_finetuned_model() 取回一个领域版 embedding。
(2) Fine-tune: SentenceTransformersFinetuneEngine(dataset, model_id=base, model_output_path=...) continues training an open-source sentence-transformer (e.g. bge-small) on your QA pairs; .finetune() runs and get_finetuned_model() returns a domain version of the embedding.
🔑
换 embedding 必须重建索引：旧索引里的向量是旧模型算的、不能和新模型混用——换模型就要把整库重新 embedding 一遍。
Swap the embedding, rebuild the index: the old index's vectors were computed by the old model and can't be mixed with the new one — a new model means re-embedding the whole corpus.
③ 换模型重建索引——最容易被忽略、却必须做的一步：必须把微调模型设成 Settings.embed_model、重新 embedding 整库、重建索引（这也提醒你微调有多贵——要重跑一遍建库）。
(3) Rebuild the index with the new model — the easily-missed but mandatory step: set the fine-tuned model as Settings.embed_model and re-embed the whole corpus and rebuild the index (a reminder of how costly fine-tuning is — you re-run the build).
④ 量化提升——最不能省的一步：微调到底有没有用、有没有过拟合，必须用数字证明。用 L12 的检索指标（hit-rate / MRR）——以 RetrieverEvaluator 在一组留出的测试问题上算——按 L19/L22 的“用数字证明、并入回归”纪律对比微调前后；再承 L22 把它固化成回归集，确保这次微调没在别的查询类型上悄悄带崩。只看训练 loss 下降是自欺——loss 降只说明模型记住了这批 QA 对，既不等于检索更准，更可能是过拟合的信号。
(4) Quantify the gain — the step you must not skip: whether fine-tuning actually helped or overfit must be proven with numbers. Compute the L12 retrieval metrics (hit-rate / MRR) with RetrieverEvaluator on a held-out set, following L19/L22's prove-with-numbers + regression discipline, and compare before vs after; per L22, freeze it into a regression set so this fine-tune didn't quietly tank other query types. Watching only the training loss drop is self-deception — a falling loss just means the model memorized these QA pairs, which is not sharper retrieval, and more likely a sign of overfitting.
示例：用自家 QA 对微调一个开源 embedding；llama-index-finetuning 在 core 之外，代码仅作演示Illustrative: fine-tune an open-source embedding on your own QA pairs; llama-index-finetuning is outside core, code is for demonstration only
# pip install llama-index-finetuning  —— 示例，微调在 core 之外
from llama_index.finetuning import generate_qa_embedding_pairs, SentenceTransformersFinetuneEngine

train = generate_qa_embedding_pairs(nodes, llm=llm)            # 用文档自动造 (问题, 相关块) 对
engine = SentenceTransformersFinetuneEngine(train, model_id="BAAI/bge-small-en",
                                            model_output_path="ft_model")
engine.finetune()
ft_model = engine.get_finetuned_model()                        # 用它重建索引，再按 L19/L22 评估

④ 性价比：微调贵又易过拟合——先穷尽更便宜的手段④ Cost-effectiveness: fine-tuning is pricey and overfit-prone — exhaust the cheaper levers first

微调不该是第一反应，而是最后一招。它贵：要造数据、要 GPU 训练、要重建整个索引、还要长期维护一个自有的模型版本；它容易过拟合：QA 对是机器造的、数量又有限，模型可能学到的是“这批问题的腔调”而非“领域语义”，在真实问题上反而变差。
Fine-tuning shouldn't be your first reflex but your last resort. It is expensive: you build data, train on a GPU, rebuild the whole index, and maintain a private model version long-term; and it overfits easily: the QA pairs are machine-made and limited, so the model may learn “the phrasing of this batch” rather than “domain semantics” and get worse on real questions.
⚠️
微调是最贵的最后一招：要造数据、训 GPU、重建整库、还易过拟合——先穷尽更便宜、更稳的手段。
Fine-tuning is the most expensive last resort: building data, GPU training, rebuilding the whole index, and overfit-prone — exhaust the cheaper, steadier levers first.
所以在动微调之前，先把这串更便宜、风险更低的手段穷尽：① 调 chunking（L06）——块太大太小都稀释语义，先把切块切对；② 加 metadata（L07）——把标题 / 章节 / 关键词喂给检索，常常立竿见影；
So before touching fine-tuning, exhaust this chain of cheaper, lower-risk levers: (1) tune chunking (L06) — chunks too big or small dilute meaning, so cut them right first; (2) add metadata (L07) — feed titles / sections / keywords to retrieval, often an instant win;
③ 上 hybrid + rerank（L21）——向量召回配上关键词（BM25）兜住术语的精确匹配，再用 reranker 精排，多数“召回不准”到这一步就解决了；④ 换更强的基座 embedding（L08）——直接换一个更大 / 更贴领域的开源或商用 embedding，常比自己微调一个小模型更省事也更稳。把这四样都试过、并用 L19/L22 确认“检索仍是瓶颈、根因确实是领域语义错位”，再考虑微调——这时它才真正划算。
(3) bring in hybrid + rerank (L21) — pair vector recall with keywords (BM25) to nail exact term matches, then a reranker to refine — most “imprecise recall” is solved by here; (4) switch to a stronger base embedding (L08) — swap in a bigger / more domain-fit open-source or commercial embedding, often easier and steadier than fine-tuning a small one yourself. Only after trying all four and using L19/L22 to confirm “retrieval is still the bottleneck and the root cause really is domain misalignment” should you consider fine-tuning — that's when it's genuinely worth it.
🧩 生活类比🧩 Analogy
通用 embedding 像一本通用词典：覆盖面广，但对你行业内部的术语、缩写、黑话常常词不达意——把意思相近的两个行话词条放到八竿子打不着的两页。微调就像给你的行业单独编一本“术语对照表”：把“同一个意思的不同叫法”并到一处，让相近的领域概念在向量空间里真正靠在一起。但编这本对照表费时费钱、还容易编偏，所以只有当通用词典实在不够用、且你确认问题就出在词义对不齐时，才值得动手编。
A generic embedding is like a general dictionary: broad coverage, but for your industry's own terms, acronyms, and slang it often misses the meaning — placing two in-context synonyms pages apart. Fine-tuning is like compiling a “glossary of terms” just for your industry: merging the different names for the same thing so related domain concepts actually sit together in vector space. But compiling that glossary is slow, costly, and easy to get wrong, so it's worth doing only when the general dictionary truly falls short and you've confirmed the problem is misaligned word meanings.
✅ 本课要点✅ Key points
  • 痛点在地基：通用 embedding 在专业领域语义错位（术语 / 缩写 / 黑话映射不准），相关块进不了 top-k；这是召回阶段的漏，rerank / 合成都救不回。The pain is in the foundation: generic embeddings misalign in a domain (jargon / acronyms / slang mapped imprecisely), so relevant chunks never enter the top-k — a recall-stage leak that rerank / synthesis can't recover.
  • 思路是对比学习：用 generate_qa_embedding_pairs 让 LLM 从自家文档反向造 (问题, 相关块) 对当信号，正样本拉近、负样本推远，校准领域语义。The idea is contrastive learning: use generate_qa_embedding_pairs to have an LLM reverse-build (question, chunk) pairs from your docs as signal, pulling positives closer and pushing negatives apart to calibrate domain semantics.
  • 流程是四步闭环：造对 → SentenceTransformersFinetuneEngine.finetune() → 换模型重建索引（向量不可混用）→ 按 L19/L22 的纪律、用 L12 的检索指标（RetrieverEvaluator 算 hit-rate / MRR）并入 L22 回归集量化前后，别只看训练 loss。The flow is a four-step loop: build pairs → SentenceTransformersFinetuneEngine.finetune() → rebuild the index with the new model (vectors can't be mixed) → following L19/L22's discipline, quantify before/after with L12's retrieval metrics (RetrieverEvaluator for hit-rate / MRR) plus an L22 regression set — not just the training loss.
  • 性价比要算清：微调贵又易过拟合，先穷尽更便宜的手段——chunking（L06）、metadata（L07）、hybrid + rerank（L21）、换更强基座（L08）——确认检索仍是瓶颈、根因是领域错位再微调。Weigh the cost: fine-tuning is pricey and overfit-prone, so exhaust the cheaper levers first — chunking (L06), metadata (L07), hybrid + rerank (L21), a stronger base (L08) — and fine-tune only after confirming retrieval is still the bottleneck and the cause is domain misalignment.

💡 设计亮点💡 Design insight
这是全书最后一课，落点却和第一课遥相呼应：L01 我们就掂量过“RAG 还是微调”，结论是知识该外置成可检索的索引、而非焊进权重。今天微调的不是 LLM、而是 embedding——校准的是“领域语义 → 向量”这层检索地基，恰恰是为了让检索更准，和 L01 那条主线始终一致。
This is the book's final lesson, yet it lands in quiet echo of the first: back in L01 we weighed “RAG vs fine-tuning” and concluded that knowledge belongs in a searchable, external index rather than welded into weights. What we fine-tune today is not the LLM but the embedding — calibrating the “domain semantics → vector” layer that is retrieval's bedrock, precisely to make retrieval sharper, fully consistent with L01's through-line.
整本书其实在反复讲同一个判断：先量化、找瓶颈，再对症下药，永远用数字说话——切块（L06）、元数据（L07）、混合检索 + 重排（L21）、更强基座（L08）都是更便宜的药，微调是最贵的那一味，只在前面都不够、且 L19/L22 证明“检索仍是瓶颈、根因是领域错位”时才开；用完还得再用 L19/L22 证明它真有效、没过拟合——开头量化、结尾量化，正是这套手艺的纪律。
The whole book has, in truth, repeated one judgment: quantify first, find the bottleneck, then treat the cause — always speak with numbers. Chunking (L06), metadata (L07), hybrid + rerank (L21), a stronger base model (L08) are the cheaper medicines; fine-tuning is the priciest dose, prescribed only when those fall short and L19/L22 prove “retrieval is still the bottleneck and the cause is domain misalignment”; and after using it you must again use L19/L22 to prove it truly helped and didn't overfit — quantify at the start, quantify at the end is the discipline of this craft.
三十五课走下来，你已经把 RAG 从“能跑一次”搭成了一条可检索、可评估、可观测、可上线、还能持续校准的工程管线。最后这块拼图——微调 embedding——留下的是一件朴素的提醒：最强的优化，往往不是再加一层花哨的新组件，而是把最底层的地基，对着你的领域，校准准。
Thirty-five lessons on, you've grown RAG from “runs once” into an engineering pipeline that is searchable, evaluable, observable, shippable, and continuously recalibrated. This last piece — fine-tuning embeddings — leaves one plain reminder: the strongest optimization is often not bolting on another shiny component, but calibrating the deepest foundation to fit your domain.

答案：AAnswer: A 微调 embedding 调整的是把文字映射成向量那一层，校准的是“领域语义 → 向量”的映射——所以它直接抬升的是领域内的检索召回质量：让本该相近的领域概念在向量空间里真正靠近，top-k 召回更准。其余三项都不归 embedding 管：“LLM 的生成速度”由模型大小 / 解码 / 硬件决定，换 embedding 不影响；“prompt 的长度上限”是 LLM 的上下文窗口属性，与 embedding 无关；“多模态能力”取决于你用的是不是多模态模型（L29），微调一个文本 embedding 也不会凭空长出图像能力。Fine-tuning an embedding adjusts the layer that maps text to vectors, calibrating the “domain semantics → vector” mapping — so what it directly lifts is retrieval recall quality within the domain: related domain concepts truly sit closer in vector space and top-k recall gets sharper. The other three are not the embedding's job: the “LLM's generation speed” is set by model size / decoding / hardware and is untouched by swapping embeddings; the “prompt's maximum length” is the LLM's context-window property, unrelated to embeddings; and “multimodal capability” depends on whether you use a multimodal model (L29) — fine-tuning a text embedding grows no image ability from nothing.

🔑 重点：只有当①领域术语密集、②通用模型语义错位明显、③检索已被证明是瓶颈、④且更便宜的手段都试过仍不够时，微调才划算——否则别先动微调。 ① 先定位瓶颈：用 L23 trace + L12 检索指标（hit-rate / MRR）确认错在检索召回（top-k 里根本没有相关块），而不是 rerank / 生成；若召回没问题，微调是白搭。② 看根因是不是领域错位：抽几条失败 query，看是不是专业术语 / 缩写 / 内部黑话导致向量不相近——是，才对症。③ 先穷尽更便宜的药：chunking（L06）、metadata（L07）、hybrid + rerank（L21）、换更强基座 embedding（L08）——这些不用训练、风险低、见效快，多数“召回不准”在这里就解决了。④ 再算账：微调要造数据、要 GPU、要重建整个索引、还要长期维护一个自有模型版本，而且容易过拟合；只有当收益（召回提升）用 L19/L22 数字证明能覆盖这些成本，才值得做。一句话：微调是最贵的一味药，留到最后、对着确诊的病灶才开。🔑 Key: fine-tuning is only worth it when (1) the domain is jargon-dense, (2) the generic model clearly misaligns, (3) retrieval is proven to be the bottleneck, and (4) the cheaper levers were all tried and still fall short — otherwise don't reach for it first. (1) Locate the bottleneck first: use L23 traces + L12 retrieval metrics (hit-rate / MRR) to confirm the fault is in retrieval recall (no relevant chunk in the top-k at all), not rerank / generation; if recall is fine, fine-tuning is wasted. (2) Check the root cause is domain misalignment: pull a few failing queries and see whether jargon / acronyms / internal slang made the vectors non-adjacent — only then is it the right cure. (3) Exhaust the cheaper medicines: chunking (L06), metadata (L07), hybrid + rerank (L21), a stronger base embedding (L08) — no training, low risk, fast payoff — most “imprecise recall” is solved here. (4) Then do the math: fine-tuning means building data, a GPU, rebuilding the whole index, and maintaining a private model version long-term, and it overfits easily; do it only when the gain (recall lift) proven with L19/L22 numbers covers those costs. In a line: fine-tuning is the priciest dose — saved for last, prescribed only against a diagnosed cause.
🎯 2. 你微调完 embedding、重建了索引，训练 loss 也降了。怎么向团队证明它真的更好、而且没有过拟合？You've fine-tuned the embedding, rebuilt the index, and training loss went down. How do you prove to the team it's genuinely better and not overfit?
看参考要点Show key points 点击展开expand
🔑 重点：别信训练 loss——留出一个独立测试集，用 L12 的检索指标（hit-rate / MRR）、按 L19/L22 的纪律在测试集上对比微调前后，数字升了才算数。 ① 训练 loss 不作数：loss 是在训练对上算的，降了只说明模型记住了这批 QA 对，既不等于检索更准，更可能是过拟合的信号。② 切一份独立测试集：造 QA 对时就留出一部分绝不参与训练（或另起一批人工问题），它代表“没见过的真实问题”。③ 用检索指标对比：承 L12，用 RetrieverEvaluator 在测试集上算微调前 vs 微调后的 hit-rate / MRR，升了才有用；若训练集涨、测试集跌，就是典型过拟合。④ 纳入回归（L22）：把这套评估固化成回归集，确保微调没在别的查询类型上悄悄变差（局部变好、整体退化也不行）。⑤ 端到端复核：最后用 Faithfulness / 答案质量看看下游答案是否真的变好——检索指标升 + 端到端不退，才算微调真有用。🔑 Key: don't trust the training loss — hold out an independent test set and use L12's retrieval metrics (hit-rate / MRR), following L19/L22's discipline, to compare before vs after on it; it counts only if the numbers rise. (1) Training loss doesn't count: loss is computed on the training pairs, so a drop just means the model memorized these QA pairs — not sharper retrieval, and more likely a sign of overfitting. (2) Carve out an independent test set: when building QA pairs, hold some back from training entirely (or write a separate human set) to represent “unseen real questions”. (3) Compare with retrieval metrics: per L12, use RetrieverEvaluator on the test set to compute hit-rate / MRR before vs after — useful only if it rises; if train climbs while test drops, that's textbook overfitting. (4) Fold into regression (L22): freeze this evaluation into a regression set so the fine-tune didn't quietly worsen other query types (locally better, globally worse won't do). (5) End-to-end re-check: finally use Faithfulness / answer quality to see if downstream answers truly improved — retrieval metrics up + end-to-end not regressing is what makes the fine-tune genuinely useful.

🎯 3. generate_qa_embedding_pairs 用 LLM 自动造训练对，又快又省人力。但这些问题是机器生成的——你怎么保证训练数据的质量，不让“垃圾进、垃圾出”？generate_qa_embedding_pairs uses an LLM to auto-build training pairs — fast and cheap. But these questions are machine-generated; how do you ensure training-data quality and avoid “garbage in, garbage out”?
看参考要点Show key points 点击展开expand
🔑 重点：自动生成打底 + 人工抽检过滤 + 补难负样本——把“机器造的对”当初稿而非终稿。 ① 自动生成打底：用 LLM 给每个块反向造问题，快速铺量，但默认它有噪声（泛泛而问、答非所块、甚至幻觉出文档里没有的问题）。② 人工抽检：随机抽一批人工过目，估算噪声率；太高就改生成 prompt（要求问题具体、能被该块直接回答）或换更强的 LLM 生成。③ 过滤明显坏对：用规则 / 模型筛掉过短、过泛、与块无关的问题。④ 补难负样本（hard negatives）：光有正样本不够——刻意挑看起来相似但其实不相关的块当负样本，模型才学得会细粒度区分，这是微调效果的关键。⑤ 贴近真实分布：尽量让训练问题像真实用户会问的（可掺入线上真实 query / 日志），否则训练集与测试集分布不一致，指标会骗人。一句话：数据质量决定微调上限，自动生成只是起点，人工与难负样本才是关键。🔑 Key: auto-generate as a base + human spot-check to filter + add hard negatives — treat machine-built pairs as a first draft, not the final one. (1) Auto-generate as a base: have an LLM reverse-build a question per chunk to get volume fast, but assume it's noisy by default (vague questions, answers not in the chunk, even hallucinated questions the doc can't answer). (2) Human spot-check: eyeball a random sample to estimate the noise rate; if too high, fix the generation prompt (demand specific questions the chunk can directly answer) or use a stronger LLM. (3) Filter obvious bad pairs: use rules / a model to drop questions that are too short, too generic, or unrelated to the chunk. (4) Add hard negatives: positives alone aren't enough — deliberately pick chunks that look similar but aren't relevant as negatives, so the model learns fine-grained distinctions — the crux of fine-tuning's payoff. (5) Match the real distribution: make training questions resemble what real users ask (mix in real production queries / logs), or a train/test distribution mismatch will make the metrics lie. In a line: data quality sets the ceiling; auto-generation is only the start, while human curation and hard negatives are the crux.
