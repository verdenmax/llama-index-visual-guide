Index 不只是“向量库”——它是为某种检索方式组织 Node 的数据结构。不同 Index 对应不同问题：相似问答用 VectorStoreIndex，整库总结用 SummaryIndex，实体关系用 PropertyGraphIndex。An Index isn't just a vector store — it's a data structure that organizes Nodes for a particular way of retrieving. Different Indexes fit different questions: VectorStoreIndex for similarity Q&A, SummaryIndex for whole-corpus summaries, PropertyGraphIndex for entity relations.

选 Index = 选检索范式Choosing an Index = choosing a retrieval paradigm

Index 决定的不是“存哪儿”，而是“怎么找”。VectorStoreIndex 把 Node 摆进向量空间、按相似度近邻召回；SummaryIndex 把 Node 排成一列、逐条遍历做全局总结；PropertyGraphIndex 把 Node 连成实体关系图、按边跳转做多跳推理。同一堆资料，选不同 Index 就得到不同的检索范式——而它们都用 from_documents / as_query_engine 的同款入口，让你能用同一套代码切换甚至组合。
An Index decides not “where things live” but “how you find them”. VectorStoreIndex places Nodes in vector space and recalls by similarity; SummaryIndex lines Nodes up and walks them all for a global summary; PropertyGraphIndex links Nodes into an entity-relation graph and hops along edges for multi-hop reasoning. Same pile of data, different Index, different retrieval paradigm — yet all share the from_documents / as_query_engine entry points, so you switch or even combine them with the same code.

🧪 示例：VectorStoreIndex vs SummaryIndex🧪 Example: VectorStoreIndex vs SummaryIndex
同一批 docs，VectorStoreIndex 适合“退款政策是什么”这类定点问答（按相似度取 top-k）；SummaryIndex 适合“全文讲了什么”这类整库总结（遍历所有 Node 再归纳）。
On the same docs, VectorStoreIndex suits pinpoint questions like “what's the refund policy” (top-k by similarity), while SummaryIndex suits whole-corpus asks like “what does the text cover” (walk every Node, then summarize).
❓ 为什么这么设计❓ Why designed this way
不同问题需要不同的“组织 + 检索”策略：定点问答要近邻、全局总结要遍历、关系推理要图。把它们做成可互换的 Index，就能用同一套上层代码服务截然不同的问法。
Different questions need different “organize + retrieve” strategies: pinpoint Q&A wants nearest-neighbor, global summary wants iteration, relational reasoning wants a graph. Making them interchangeable Indexes lets one upper layer serve very different question shapes.
⚙️ 内部怎么跑⚙️ How it runs inside
所有 Index 都继承 BaseIndex，共享 from_documents（建索引）与 as_query_engine / as_retriever（装配查询）。差异只在于内部如何组织 Node、以及默认配哪种 retriever 与 response 合成策略。
Every Index inherits BaseIndex, sharing from_documents (build) and as_query_engine / as_retriever (assemble queries). They differ only in how they organize Nodes internally and which default retriever and response strategy they wire up.
🔀 替代方案🔀 Alternatives
当一个库里既有定点问答又有整库总结时，可建多个 Index，再用 RouterQueryEngine 按问题自动路由到合适的那个；也可以组合检索器、做多 Index 融合。单一 Index 简单，多 Index + Router 更全能。
When a corpus needs both pinpoint Q&A and global summaries, build several Indexes and let a RouterQueryEngine auto-route each question to the right one; you can also compose retrievers and fuse multiple Indexes. One Index is simpler; many Indexes + a Router is more capable.
选 Index 就是选检索范式choosing an Index = choosing a retrieval style
from llama_index.core import VectorStoreIndex, SummaryIndex

vindex = VectorStoreIndex.from_documents(docs)   # 相似问答
print(vindex.as_query_engine().query('退款政策？'))

sindex = SummaryIndex.from_documents(docs)        # 整库总结
print(sindex.as_query_engine(response_mode='tree_summarize').query('全文讲了什么？'))
多 Index + Router：让框架按问题自动选检索范式Many Indexes + a Router: let the framework auto-pick the retrieval paradigm per question
from llama_index.core import VectorStoreIndex, SummaryIndex
from llama_index.core.tools import QueryEngineTool
from llama_index.core.query_engine import RouterQueryEngine

vector_tool = QueryEngineTool.from_defaults(
    VectorStoreIndex.from_documents(docs).as_query_engine(),
    description='按相似度回答具体细节问题')
summary_tool = QueryEngineTool.from_defaults(
    SummaryIndex.from_documents(docs).as_query_engine(response_mode='tree_summarize'),
    description='对全文做整体总结')

# Router 读 description，自动把问题分给合适的 Index
engine = RouterQueryEngine.from_defaults([vector_tool, summary_tool])
print(engine.query('全文主要讲了什么？'))
✅ 本课要点✅ Key points
  • Index = 组织方式 + 检索范式，不等于向量库本身。An Index = organization + retrieval style, not the vector store itself.
  • 相似问答选 VectorStoreIndex；整库总结选 SummaryIndex。VectorStoreIndex for similarity Q&A; SummaryIndex for whole-corpus summaries.
  • 所有 Index 都用 from_documents / as_query_engine 同款入口。Every Index shares the from_documents / as_query_engine entry points.
💡 设计亮点💡 Design insight
“选 Index”本质是“选检索策略”。统一的 Index→QueryEngine 入口让你用同一套代码切换范式，甚至用 Router 在多个 Index 间自动路由。
“Choosing an Index” is really “choosing a retrieval strategy.” The uniform Index→QueryEngine entry lets you switch paradigms with the same code — even auto-route across indexes with a Router.
