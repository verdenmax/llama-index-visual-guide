把整本书拼成一个应用Assembling the whole guide into one app

前 19 课每一课都拆开了管道上的一站；这一课把它们按同一套接口拼回去。整条链路只有两个阶段：写入路径把数据变成可复用、可落盘的索引，查询路径在索引之上反复检索、合成、作答。因为每一站都遵守统一抽象，你可以只替换其中一件（换向量库、换检索器、加一道评估），而不动其余结构。举例：想要更强的召回，把 as_retriever 换成第 18 课的 QueryFusionRetriever 即可，主流程一行不改；而全局 Settings（第 17 课）从一开始就在背后统一着 LLM 与 embedding。
Each of the first 19 lessons opened up one stop on the pipeline; this one snaps them back together behind one shared interface. The whole chain has just two phases: the write path turns data into a reusable, persistable index, and the query path retrieves, synthesizes and answers on top of it — over and over. Because every stop obeys the same abstraction, you can swap a single part (a different vector store, a different retriever, an added evaluator) without disturbing the rest. For example, to boost recall just swap as_retriever for Lesson 18's QueryFusionRetriever — the main flow doesn't change a line; meanwhile the global Settings (Lesson 17) has been unifying the LLM and embedding behind the scenes all along.

🧪 示例🧪 Example
capstone 的核心是一个 if os.path.exists(PERSIST) 分支：存在就 load_index_from_storage 秒级复活，否则跑一遍 IngestionPipeline → VectorStoreIndex 再 persist。其余几行只是把检索器、后处理、合成器、评估器装到同一个 RetrieverQueryEngine 上。
The heart of the capstone is one if os.path.exists(PERSIST) branch: if it exists, load_index_from_storage revives it in seconds; otherwise run IngestionPipeline → VectorStoreIndex once, then persist. The remaining lines just bolt a retriever, postprocessor, synthesizer and evaluator onto one RetrieverQueryEngine.
❓ 为什么这么设计❓ Why designed this way
因为每一站都是实现同一接口的标准件，整台应用才能像搭积木一样拼装。换一个向量库或检索策略时，你不必重写主流程——只把对应那块换掉，RetrieverQueryEngine 的骨架原封不动。
Because every stop is a standard part implementing the same interface, the whole app assembles like building blocks. To change a vector store or retrieval strategy you never rewrite the main flow — just swap that one block while the RetrieverQueryEngine skeleton stays intact.
⚙️ 内部怎么跑⚙️ How it runs inside
运行分两支：首次走写入路径（摄取 → 建索引 → persist），之后走复用路径（load_index_from_storage）。关键时序：Settings.embed_model 必须在建索引之前设好——落盘的向量由它决定，加载后必须用同一个 embedding 才能对齐检索。
It runs in two branches: the first time takes the write path (ingest → index → persist); later runs take the reuse path (load_index_from_storage). Critical ordering: Settings.embed_model must be set before building the index — it determines the vectors written to disk, and after loading you must use the same embedding for retrieval to line up.
🔀 替代方案🔀 Alternatives
想把这台问答机改造成“带脚注的客服机器人”？只需替换三件：把 query engine 换成 chat_mode='condense_plus_context' 的 chat engine（多轮记忆）、加一道引用后处理把 source_nodes 渲染成脚注、再挂上评估当上线前的质量闸。骨架不变，只换这几块。
Want to turn this answering machine into a “support bot with footnotes”? Swap just three parts: replace the query engine with a chat engine (chat_mode='condense_plus_context') for multi-turn memory, add a citation postprocessor that renders source_nodes as footnotes, and attach evaluation as a pre-launch quality gate. The skeleton stays; only these blocks change.
一个文件跑通完整 RAG（含持久化、后处理、评估、多轮）one file, full RAG: persistence, post-processing, eval, chat
from llama_index.core import (
    SimpleDirectoryReader, VectorStoreIndex, StorageContext,
    load_index_from_storage, Settings, get_response_synthesizer,
)
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.postprocessor import SimilarityPostprocessor
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.evaluation import FaithfulnessEvaluator
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
import os

Settings.llm = OpenAI(model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model='text-embedding-3-small')

PERSIST = './storage'
if os.path.exists(PERSIST):                      # 已建过就秒加载
    index = load_index_from_storage(StorageContext.from_defaults(persist_dir=PERSIST))
else:                                            # 首次：摄取 -> 建索引 -> 落盘
    docs = SimpleDirectoryReader('./data').load_data()
    nodes = IngestionPipeline(transformations=[SentenceSplitter(chunk_size=512, chunk_overlap=50)]).run(documents=docs)
    index = VectorStoreIndex(nodes)
    index.storage_context.persist(persist_dir=PERSIST)

engine = RetrieverQueryEngine.from_args(
    retriever=index.as_retriever(similarity_top_k=4),
    node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=0.4)],  # 阈值随数据而定
    response_synthesizer=get_response_synthesizer(response_mode='compact'),
)
resp = engine.query('退款政策是什么？')
print('答案:', resp)
print('忠实度:', FaithfulnessEvaluator().evaluate_response(response=resp).passing)

chat = index.as_chat_engine(chat_mode='condense_plus_context')
print(chat.chat('那国际订单也一样吗？'))
批量评估当回归测试：改完管道就重跑这套用例Batch eval as a regression test: re-run this set after every pipeline change
from llama_index.core.evaluation import FaithfulnessEvaluator, RelevancyEvaluator

# 在 capstone 的 engine 之上：把单次评估扩成可回归的质量闸
faith = FaithfulnessEvaluator()        # 答案是否忠于检索到的上下文
relev = RelevancyEvaluator()           # 检索到的上下文是否切题

eval_set = ['退款要多久到账？', '国际订单能退吗？', '怎么申请换货？']
passed = 0
for q in eval_set:
    resp = engine.query(q)
    ok = (faith.evaluate_response(response=resp).passing
          and relev.evaluate_response(query=q, response=resp).passing)
    passed += int(ok)
    print(q, 'PASS' if ok else 'FAIL')
print(f'通过率 pass-rate: {passed}/{len(eval_set)}')
✅ 本课要点✅ Key points
  • 完整 RAG = 写入路径（建一次、落盘）+ 查询路径（反复问）。A full RAG = write path (build once, persist) + query path (ask repeatedly).
  • 每一站都是可替换组件：换库、换检索、换合成都不动主结构。Every stop is a swappable component: change store/retrieval/synthesis without touching the skeleton.
  • 持久化 + 评估让它从 demo 走向可维护、可回归。Persistence + evaluation take it from demo to maintainable and regression-safe.
💡 设计亮点💡 Design insight
整条管道每一站都能独立替换，却共享统一接口——这正是 LlamaIndex 的核心思想：用可组合的标准件搭出你自己的 RAG。
Every stop swaps independently yet shares one interface — that's LlamaIndex's core idea: compose your own RAG from standard, interchangeable parts.
