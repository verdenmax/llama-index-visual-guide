RAG 的最小检索单位是 Node，不是整篇 Document。Document 是一整份原始资料；切块后得到带元数据与关系的 Node——检索、返回、合成都围绕 Node 进行。The unit of retrieval in RAG is the Node, not the whole Document. A Document is one raw source; chunking yields Nodes carrying metadata and relationships — retrieval, results and synthesis all revolve around Nodes.

🧩 生活类比🧩 Analogy
Document 是整本书；Node 是书里的一页便利贴，背面记着“来自哪本书、上一页/下一页是谁”（relationships）和“章节/页码”（metadata）。
A Document is the whole book; a Node is one sticky note from it, whose back records “which book, previous/next note” (relationships) and “chapter/page” (metadata).

为什么以 Node 为检索单位Why the Node is the unit of retrieval

检索、排序、返回、合成全部以 Node 为单位，是为了让下游组件“说同一种语言”：向量库存的是 Node、检索器返回的是 Node、合成器读的也是 Node，无论原始来源是 PDF、网页还是数据库。Document 太粗——一篇上万字里真正相关的常常只有一两段，整篇压成一个向量时，相关内容会被大量无关文字稀释，命中既不准也不稳；Node 把粒度收敛到“一次讲清一件事”的片段，既适配 embedding 的长度上限，又让命中更准、引用更细。
Retrieval, ranking, results and synthesis all operate on Nodes so that every downstream component “speaks one language”: the vector store holds Nodes, the retriever returns Nodes, the synthesizer reads Nodes — whatever the original source. A Document is too coarse — in a 10k-word file usually only a paragraph or two is relevant, and squeezing the whole thing into one vector dilutes that signal with unrelated text, so hits are neither precise nor stable; a Node narrows granularity to “one idea at a time”, fitting the embedding's length limit while making hits sharper and citations finer.

🧪 示例：构造 Node 并连上关系🧪 Example: build a Node and wire its relationships
手动建两个 TextNode，把第二个的 SOURCE 指回 Document、PREVIOUS 指向第一个，就得到一条可前后扩展、可溯源的链：检索命中某个 Node 后，能顺着 NEXT 取下一段补全上下文。
Create two TextNodes by hand, point the second's SOURCE back to the Document and its PREVIOUS to the first, and you get a chain you can expand and trace: after a Node is hit you can follow NEXT to pull the next chunk for fuller context.
❓ 为什么这么设计❓ Why designed this way
把“检索/溯源的单位”固定为 Node，整条管道就只需理解一种对象。粒度统一后，过滤、排序、引用、前后文扩展都有了共同的承载体，组件之间也能自由替换而不必翻译彼此的数据结构。
Fixing the Node as the unit of retrieval and provenance means the whole pipeline only ever reasons about one kind of object. With uniform granularity, filtering, ranking, citation and context expansion share one carrier, and components stay swappable without translating each other's data.
⚙️ 内部怎么跑⚙️ How it runs inside
关系存在 node.relationships 字典里，键是 NodeRelationship 枚举（SOURCE / PREVIOUS / NEXT / PARENT / CHILD），值是 RelatedNodeInfo；node.ref_doc_id 就是顺着 SOURCE 找到的来源文档 id。
Relationships live in the node.relationships dict, keyed by the NodeRelationship enum (SOURCE / PREVIOUS / NEXT / PARENT / CHILD) with RelatedNodeInfo values; node.ref_doc_id is simply the source document id reached via SOURCE.
🔀 替代方案🔀 Alternatives
直接拿整篇 Document 去检索看似省事，却粒度太粗：长文里相关的只有一两段，整篇的向量被无关内容稀释，命中不准、还浪费上下文。以 Node 为单位，才能既命中精准、又能按需向前后扩展。
Retrieving whole Documents looks simpler but is too coarse: only a paragraph or two is relevant, the whole-doc vector gets diluted by unrelated text, hits are imprecise and context is wasted. Working per Node is what makes hits precise while still allowing on-demand prev/next expansion.
元数据与关系随 Node 一起流动metadata and relationships travel with the Node
from llama_index.core import Document
from llama_index.core.schema import TextNode, NodeRelationship, RelatedNodeInfo

doc = Document(text='LlamaIndex 让 RAG 更简单。', metadata={'source': 'faq.md'})

n1 = TextNode(text='第一段', metadata={'page': 1})
n2 = TextNode(text='第二段', metadata={'page': 2})
# 关系：n2 来自 doc，且紧跟在 n1 之后
n2.relationships[NodeRelationship.SOURCE] = RelatedNodeInfo(node_id=doc.doc_id)
n2.relationships[NodeRelationship.PREVIOUS] = RelatedNodeInfo(node_id=n1.node_id)
print(n2.metadata, n2.relationships.keys())
用 metadata 过滤检索，用 relationships 溯源filter retrieval by metadata, trace provenance via relationships
from llama_index.core import VectorStoreIndex
from llama_index.core.schema import NodeRelationship
from llama_index.core.vector_stores import MetadataFilters, MetadataFilter

nodes = [n1, n2]                       # 上一段创建的两个 Node
index = VectorStoreIndex(nodes)

# 只在 metadata['page'] == 2 的 Node 里检索（元数据过滤）
flt = MetadataFilters(filters=[MetadataFilter(key='page', value=2)])
engine = index.as_query_engine(filters=flt)
print(engine.query('第二页讲了什么？'))

# 顺着 SOURCE 关系溯源：命中的 Node 来自哪个 Document
src = nodes[1].relationships.get(NodeRelationship.SOURCE)
print(src.node_id if src else None)
✅ 本课要点✅ Key points
  • Node 是检索单位；Document 只是它的来源。The Node is the unit of retrieval; the Document is just its source.
  • metadata 随 Node 流动，可用于过滤与溯源。metadata flows with the Node, enabling filtering and provenance.
  • relationships（SOURCE/PREVIOUS/NEXT）是进阶检索（如前后文扩展、溯源）的基础。relationships (SOURCE/PREVIOUS/NEXT) underpin advanced retrieval (e.g. prev/next context expansion, provenance).
💡 设计亮点💡 Design insight
统一的 Node 抽象让整条管道“说同一种语言”：无论来源是 PDF 还是数据库，下游都只面对带元数据与关系的 Node。
A single Node abstraction makes the whole pipeline speak one language: whether the source is a PDF or a database, everything downstream only deals with metadata-and-relationship-bearing Nodes.

🔑 重点：把年份/版本写进 metadata，用 MetadataFilters 先过滤、再语义排序（检索第二通道）。纯向量只懂语义、不懂“限定 2024”这种结构化约束。前提是建索引时就把这些字段抽进每个 Node 的 metadata；否则过滤无从谈起。🔑 Key: put year/version into metadata and use MetadataFilters to filter first, then rank semantically (retrieval's second channel).
