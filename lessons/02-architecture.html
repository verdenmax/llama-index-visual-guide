LlamaIndex 分两层：core（稳定的抽象与编排）+ 300+ 集成包（具体的 LLM / Embedding / 向量库实现）。命名约定一眼区分：导入路径带 core 的是核心抽象，不带的是某个第三方集成。LlamaIndex has two layers: core (stable abstractions + orchestration) and 300+ integration packages (concrete LLM / embedding / vector-store implementations). The import path tells them apart: paths with core are core abstractions; those without are a specific third-party integration.

为什么把接口和实现拆开Why split interfaces from implementations

core 只定义“能做什么”的抽象接口，具体“怎么做”交给独立集成包。于是新增一个 LLM 或向量库厂商，只需发布一个集成包，core 不必改动、也不必跟着发版；你的应用只依赖 core 的稳定接口，换厂商几乎零成本。
core defines only the abstract “what”; each concrete “how” lives in a separate integration. So adding a new LLM or vector-store vendor is just publishing one integration package — core never changes or re-releases, and because your app depends only on core's stable interface, swapping vendors is near-free.

一个典型项目会装 llama-index-core + llama-index-llms-openai + llama-index-embeddings-openai：核心一份，其余按需各取所需。
A typical project installs llama-index-core + llama-index-llms-openai + llama-index-embeddings-openai: one core, the rest à la carte.
❓ 为什么这么设计❓ Why designed this way
让生态可独立演进：300+ 厂商各自迭代发版，不被 core 的节奏绑死，也不会把彼此的依赖互相拖进来。
It lets the ecosystem evolve independently: 300+ vendors iterate and release on their own cadence, unshackled from core's schedule and from each other's dependencies.
⚙️ 内部怎么跑⚙️ How it runs inside
命名空间约定是关键：llama_index.core.* 是抽象，llama_index.llms.x / llama_index.vector_stores.y 是实现，导入路径一眼可辨。
The namespace convention is the key: llama_index.core.* is abstractions, while llama_index.llms.x / llama_index.vector_stores.y are implementations — the import path tells you which at a glance.
🔀 替代方案🔀 Alternatives
单体框架（所有实现塞进一个包）安装简单，但依赖臃肿、升级互相牵制；插件式分层用多装几个小包，换来灵活与可维护。
A monolith (all implementations in one package) is simpler to install but bloats dependencies and couples upgrades; the plugin-style split trades a few extra small installs for flexibility and maintainability.
core 定接口，集成填实现core defines interfaces, integrations fill them in
# 核心 + 按需安装集成（每个集成是独立的 pip 包）
pip install llama-index-core
pip install llama-index-llms-openai llama-index-embeddings-openai

from llama_index.core import Settings           # 核心：全局配置
from llama_index.llms.openai import OpenAI       # 集成：换成别家只改这一行
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.llm = OpenAI(model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model='text-embedding-3-small')
换 provider 只改一行：core 之上的代码完全不变Swap provider in one line: everything above core stays the same
# 同一套链路，换一家 LLM 只动这一块
from llama_index.core import Settings

# A) OpenAI
from llama_index.llms.openai import OpenAI
Settings.llm = OpenAI(model='gpt-4o-mini')

# B) 换成 Anthropic：只改导入 + 这一行，其余代码原封不动
from llama_index.llms.anthropic import Anthropic
Settings.llm = Anthropic(model='claude-3-5-sonnet-latest')
✅ 本课要点✅ Key points
  • 导入路径带 core = 核心抽象；不带 = 第三方集成实现。Import path with core = core abstraction; without = a third-party integration.
  • 集成是独立 pip 包，按需安装，互不绑定版本。Integrations are separate pip packages, installed à la carte.
  • 换 LLM 只改一行；换 Embedding 还得重建索引（向量要重算）；换向量库改的是存储注入——三者代价不同。Swapping the LLM is a one-liner; swapping the embedding also forces a re-index (vectors must be recomputed); swapping the vector store changes storage wiring — different costs.
💡 设计亮点💡 Design insight
这套分层的真正红利：本地用 Ollama 把整条 RAG 跑通，上线只把 Settings.llm 换成 OpenAI——retriever / index / query engine 一行都不用改。
The real dividend of this layering: prototype the whole RAG locally on Ollama, then ship by swapping just Settings.llm to OpenAI — your retriever / index / query engine don't change a line.
