上一课的多 agent 小队很能干，也很自由——它可能自己决定去删库、转账、下单、对外发消息。这些动作不可逆：错了就收不回来，光靠模型自己拍板太冒险。人在回路（HITL）就是给这种高风险动作装一道刹车：在真正动手前暂停，把请求交给人，等人点头才继续、摇头就中止。关键不是“每一步都要人盯”——那会把系统拖垮——而是只在不可逆 / 高风险的那一步加闸，其余照常自动。Last lesson's multi-agent team is capable — and very free: it might decide on its own to drop a database, transfer money, place an order, or message the outside world. Those actions are irreversible: once wrong you can't take them back, and letting the model alone make the call is too risky. Human-in-the-loop (HITL) puts a brake on such high-risk actions: pause right before acting, hand the request to a human, and continue only on a yes — abort on a no. The point is not “a human watches every step” — that would grind the system to a halt — but to gate only the irreversible / high-risk step, leaving the rest automatic.

① 为什么要人在回路：高风险 / 不可逆动作不能全自动① Why human-in-the-loop: high-risk / irreversible actions can't be fully automatic

L32 的小队越自由就越能干，但LLM 是概率的——它总有选错的时候。对大多数动作（读文档、起草文字、再查一次资料）来说，选错的代价很低：重试一次就好，没人受伤。可另一类动作会越过边界、落到真实世界，代价就完全不同了——下面这一类，正是要盯紧的。
L32's team grows more capable as it grows freer, but an LLM is probabilistic — it will sometimes pick wrong. For most actions (reading a doc, drafting text, one more lookup) a wrong pick is cheap: just retry, no one gets hurt. But another class of action crosses the boundary into the real world, where the stakes are entirely different — and that class is the one to watch.
⚠️
删库、转账、下单、对外发邮件这类动作不可逆——一旦做错就收不回来，不能让模型独自拍板。
Dropping a database, transferring money, ordering, sending an outbound email are irreversible — once wrong you can't take them back, so the model must not decide alone.
正因如此，HITL 的思路很简单：在这一步真正落地前插入一个人，让人来做最后的批准。但人不是免费的——每一次确认都加延迟、打扰人，所以只把它放在代价最高、最不可逆的动作上；其余一切照常自动。这正是上一课结尾留下的那道“刹车”。
That is exactly why HITL's idea is simple: insert a human right before such a step actually lands and let them give the final approval. But a human isn't free — every confirmation adds latency and interrupts someone — so you reserve it for the costliest, least reversible actions and keep everything else automatic. This is the very “brake” the last lesson left off on.

② 机制：@step 返回 InputRequiredEvent 挂起，调用方回 HumanResponseEvent 恢复② Mechanism: a @step returns InputRequiredEvent to pause; the caller sends HumanResponseEvent to resume

承接 L32 的底层视角：agent 跑在 workflow 上，流程由 @step + 事件驱动。HITL 用的就是这套机器，只多了一对专门的事件，机制分两半、正好一来一回。① workflow 侧——挂起：某个 @step 在要动手前，把返回类型标成 InputRequiredEvent 并返回它。框架会自动把它写进事件流，而且不需要任何下游 step 去消费就能通过校验——于是流程停在这里、向人请求输入。InputRequiredEvent(prefix="…") 里的 prefix 就是要展示给人看的那句话（“确认要删账户吗？”）。
Continuing L32's under-the-hood view: an agent runs on a workflow, driven by @step + events. HITL uses that same machinery, plus one dedicated pair of events; the mechanism is two halves — exactly one round-trip. (1) Workflow side — pause: right before acting, a @step annotates its return as InputRequiredEvent and returns it. The framework auto-writes it to the event stream, and it needs no downstream step to consume it to pass validation — so the flow stops here and asks a human. The prefix in InputRequiredEvent(prefix="…") is the line to show the human (“confirm deleting the account?”).
② 调用方侧——回灌：发起 workflow 的人正用 async for ev in handler.stream_events() 盯着事件流；一旦看到 InputRequiredEvent，就拿 ev.prefix 去问真正的人（前端弹窗 / 审批系统 / 命令行），把人的回答装进 HumanResponseEvent(response=…)，用 handler.ctx.send_event(...) 送回 workflow。消费 HumanResponseEvent 的那个 @step 随即被唤醒，读 ev.response，据此恢复或中止。
(2) Caller side — inject back: whoever launched the workflow is watching the stream with async for ev in handler.stream_events(); on seeing an InputRequiredEvent, it takes ev.prefix to ask a real human (a frontend dialog / approval system / CLI), wraps the answer in HumanResponseEvent(response=…), and sends it back with handler.ctx.send_event(...). The @step consuming HumanResponseEvent then wakes up, reads ev.response, and resumes or aborts accordingly.
📌
一句话——这对“出—回”就是 HITL 的全部：workflow 返回 InputRequiredEvent 挂起、把 prefix 问题带出去；调用方流式取到后回 HumanResponseEvent 恢复、把 response 决定带回来。
In a line — this “out-and-back” pair is all of HITL: the workflow returns InputRequiredEvent to pause, carrying the prefix question out; the caller streams it and sends back HumanResponseEvent to resume, carrying the response decision back.
workflow 侧：act 这个 @step 返回 InputRequiredEvent——自动写入事件流并挂起；消费 HumanResponseEvent 的 finish 据 response 恢复或中止Workflow side: the act @step returns InputRequiredEvent — auto-written to the stream and pausing; finish, which consumes HumanResponseEvent, resumes or aborts on response
from llama_index.core.workflow import (Workflow, step, StartEvent, StopEvent,
                                        InputRequiredEvent, HumanResponseEvent)

class ApprovalFlow(Workflow):
    @step
    async def act(self, ev: StartEvent) -> InputRequiredEvent:
        return InputRequiredEvent(prefix="确认要执行『删除账户』吗？(yes/no) ")   # 写入事件流并挂起

    @step
    async def finish(self, ev: HumanResponseEvent) -> StopEvent:
        ok = ev.response.strip().lower() == "yes"
        return StopEvent(result="已执行" if ok else "已取消")
调用方侧：流式监听事件流，遇到 InputRequiredEvent 就拿 prefix 去问人，把答案装进 HumanResponseEvent 送回——await handler 取最终结果Caller side: stream the events; on InputRequiredEvent, use prefix to ask a human, wrap the answer in HumanResponseEvent and send it back — await handler for the result
handler = ApprovalFlow().run()
async for ev in handler.stream_events():
    if isinstance(ev, InputRequiredEvent):
        answer = input(ev.prefix)                              # 真实场景：来自前端/审批系统
        handler.ctx.send_event(HumanResponseEvent(response=answer))
result = await handler

③ 落点：审批闸、低置信度兜底、敏感工具调用前确认③ Where it lands: approval gates, low-confidence fallback, confirming before sensitive tool calls

HITL 在三处最常落地，但都遵循同一条线——只在该加闸的地方加闸。① 审批闸：在不可逆的对外动作前（删库 / 转账 / 下单 / 群发）强制一次人工 yes/no，就像上面的 ApprovalFlow——通过才执行，否则中止。② 低置信度兜底：当模型 / 检索没把握时（相似度过低、证据互相矛盾、模型自陈“不确定”），别硬编，转人工裁决；关键是设阈值——只有置信度低于线才弹给人，高置信的照常自动答。
HITL lands in three common spots, all following one line — gate only where a gate belongs. (1) Approval gate: before an irreversible outward action (drop DB / transfer / order / mass-send), force a human yes/no, just like the ApprovalFlow above — execute on pass, abort otherwise. (2) Low-confidence fallback: when the model / retrieval is unsure (similarity too low, conflicting evidence, the model says “not sure”), don't fake it — escalate to a human; the key is a threshold: route to a human only below the line, answer high-confidence cases automatically.
💡
只在该加闸处加闸：给低置信度设阈值、给工具分级，只把极少数不可逆 / 高风险 / 低置信的点交给人——别让 HITL 变成“凡事都问”。
Gate only where a gate belongs: set a threshold for low confidence and grade your tools, handing humans only the very few irreversible / high-risk / low-confidence points — so HITL never becomes “ask about everything”.
③ 敏感工具调用前确认：给 agent 的工具分级——把 send_email / run_sql / place_order 标成“敏感”，agent 真要调用前暂停让人点头；而只读 / 可撤销的工具（检索、查天气）放它自己跑。三者都在解同一个问题：把极少数不可逆 / 高风险 / 低置信的点拦下来交给人，其余维持自动——这也正好接上 L25 的护栏（护栏自动挡掉明显违规，HITL 处理那些需要人判断的灰色地带）与 L23 的可观测（人要看得到上下文 / trace 才能拍好这一板）。
(3) Confirm before sensitive tool calls: grade an agent's tools — mark send_email / run_sql / place_order as “sensitive” and pause for a human nod right before invoking them, while read-only / reversible tools (retrieval, weather) run on their own. All three solve the same problem: intercept the very few irreversible / high-risk / low-confidence points and hand them to a human, keeping the rest automatic — which dovetails with L25's guardrails (guardrails auto-block the obvious violations; HITL handles the gray zone that needs human judgment) and L23's observability (a human needs visible context / traces to make the call well).
🧩 生活类比🧩 Analogy
HITL 像网银支付前的“二次确认弹窗”：你平时查余额、看账单都畅通无阻，可一旦要真把钱转出去，屏幕就弹一个框——“确认向 ××× 转账 ¥5000？”——停在这里等你按下确认，你点了才扣款、取消就作罢。它不挡大多数操作，只在那一步不可逆的关键动作上拦一下，把最终决定权交回给你。
HITL is like the “confirm” dialog before an online-banking payment: browsing your balance or statements flows freely, but the moment you actually move money out, a box pops up — “Confirm transferring ¥5000 to ×××?” — and it waits for you to press confirm; only then does it debit, and cancel calls it off. It blocks almost nothing, stepping in only at that one irreversible, critical action, handing the final decision back to you.
✅ 本课要点✅ Key points
  • HITL 是给不可逆 / 高风险动作（删库、转账、下单、外发）装的刹车：动手前暂停、等人确认——但只在该加闸的那一步加，不是每步都要人。HITL is a brake for irreversible / high-risk actions (drop DB, transfer, order, outbound): pause before acting and wait for a human — but gate only that step, not every step.
  • 机制是一对事件的来回：一个 @step 返回 InputRequiredEvent（自动写入事件流、挂起）；调用方流式取到后回一个 HumanResponseEvent，消费它的 @step 据 response 恢复 / 中止（prefix 带问题出去）。The mechanism is one round-trip of a pair of events: a @step returns InputRequiredEvent (auto-written to the stream, pausing); the caller streams it and sends back a HumanResponseEvent, and the consuming @step resumes / aborts on response (prefix carries the question out).
  • 三个落点：审批闸（不可逆外部动作前）、低置信度兜底（分数过低 / 证据矛盾时转人）、敏感工具调用前确认（给工具分级，只对敏感的暂停）。Three landing spots: an approval gate (before irreversible external actions), a low-confidence fallback (escalate when scores are low / evidence conflicts), and confirming before sensitive tool calls (grade tools, pause only for sensitive ones).
  • 人是最贵的一层：每次确认都加延迟、打扰人——所以触发要省着用，并与 L25 护栏 + L23 trace 叠成三层纵深（机器能判的别劳烦人，人判时要有上下文）。A human is the most expensive layer: each confirm adds latency and interruption — so trigger it sparingly, and stack it with L25 guardrails + L23 traces into three-layer defense in depth (don't bother a human with what a machine can judge; give context when a human does).
💡 设计亮点💡 Design insight
人在回路的精髓，不是“让人盯着每一步”，而是把人放在最该放的那一个点上。L32 给了我们一支自由、能干的小队，可自由一旦延伸到不可逆的边界——删库、转账、外发——就危险了；HITL 就是那道刹车。但真正的设计判断不在“要不要人”，而在“在哪一步要人”：只在不可逆 / 高风险 / 低置信的极少数点上暂停，其余全自动——闸加错地方，要么拦不住事故，要么把吞吐拖垮、把人烦走。
The essence of human-in-the-loop isn't “make a human watch every step” but placing the human at the one point that matters most. L32 handed us a free, capable team, but once that freedom reaches an irreversible boundary — dropping a DB, transferring money, sending outbound — it turns dangerous; HITL is that brake. Yet the real design call isn't “human or not” but “at which step”: pause only at the very few irreversible / high-risk / low-confidence points and keep the rest automatic — gate the wrong place and you either fail to stop the incident or grind throughput and annoy the human away.
把它放进全书的脉络看，这是三层纵深的最后一层：L25 护栏自动挡掉明显违规（机器能判的，别劳烦人），L23 可观测把上下文 / trace 摆到人面前（让这一板拍得有依据），L33 HITL 在闸口让人做最后的拍板。人是最贵也最后的一层，所以只用在代价最高、最不可逆的地方。机制上它又轻得恰到好处：一个 @step 返回 InputRequiredEvent 就挂起、调用方回一个 HumanResponseEvent 就恢复——安全不是关掉自主，而是在关键一步把决定权短暂交回给人。下一课 L34，我们就把这套带着刹车的系统真正上线成服务。
Seen across the whole book, this is the last of three layers of defense in depth: L25 guardrails auto-block the obvious violations (don't bother a human with what a machine can judge), L23 observability puts context / traces in front of the human (so the call is well-grounded), and L33 HITL lets a human make the final decision at the gate. The human is the most expensive, last layer, so spend it only where the downside is costliest and least reversible. And the mechanism stays just light enough: a @step returning InputRequiredEvent pauses, a caller's HumanResponseEvent resumes — safety isn't switching off autonomy but briefly handing the decision back to a human at the critical step. Next, L34 takes this brake-equipped system and actually ships it as a service.

看参考要点Show key points 点击展开expand
🔑 重点：HITL 的成本来自“弹窗的频率”，所以核心是少弹、晚弹、聪明地弹——只在不可逆 / 高风险 / 低置信处触发，能批量就批量，并给默认超时与回退，别让流程死等人。① 收窄触发面：只给真正不可逆 / 高风险的动作（删库、转账、外发）和低置信（分数过低、证据矛盾）加闸；只读 / 可撤销的操作一律自动，别“凡事都问”。② 批量确认：把一段时间内的多个待批聚成一屏让人一次点完，而不是一个动作弹一次——既省打扰又省上下文切换。③ 设默认与超时：给每个确认配超时 + 默认策略（高风险默认“拒”、低风险默认“放”），人没在 N 分钟内回就走默认，绝不无限期挂起。④ 上下文一次给足：弹窗里把 prefix 写清楚（要做什么、影响什么、依据是什么，呼应 L23 的 trace），让人一眼能判，减少来回追问。⑤ 事后调阈值：统计确认通过率——某类动作若人几乎总点“yes”，说明这道闸加得过密，可以放宽或撤掉。🔑 Key: HITL's cost comes from “how often it pops up”, so the core is to pop up less, later, and smarter — trigger only at irreversible / high-risk / low-confidence points, batch when you can, and give defaults plus timeouts so the flow never dead-waits on a human. (1) Narrow the trigger surface: gate only truly irreversible / high-risk actions (drop DB, transfer, outbound) and low-confidence cases (score too low, conflicting evidence); auto-run read-only / reversible ops — don't “ask about everything”. (2) Batch confirmations: collect several pending approvals over a window into one screen to clear at once, instead of one popup per action — saving both interruption and context-switching. (3) Defaults and timeouts: pair each confirm with a timeout + default policy (high-risk defaults to “deny”, low-risk to “allow”); if no reply within N minutes, take the default and never hang indefinitely. (4) Give full context once: write a clear prefix (what it will do, what it affects, on what basis — echoing L23's traces) so the human can judge at a glance, cutting back-and-forth. (5) Tune thresholds afterward: track the approval rate — if a class of action is almost always “yes”, that gate is too dense and can be loosened or removed.

🔑 重点：三者不是重复，而是三层纵深防御——护栏自动挡掉明显违规，trace 让人看清上下文，HITL 是最后那道“人来拍板”的闸；越往后越贵，所以越往后用得越省。① L25 护栏：自动、廉价、挡明显的。规则 / 分类器能判定的（注入、越权、明显有害）就直接拒，根本不该惊动人——机器能判的别劳烦人。② L23 可观测：让决定有依据。当事情落到人手里，人得看到这一步要做什么、基于哪些证据、前面怎么走过来的——trace / 上下文就是人做判断的仪表盘，没有它人只能瞎点。③ L33 HITL：灰色地带的最终裁决。规则判不了、又不可逆 / 高风险的，才交给人点头——这是最贵的一层（要人、要等），所以放在最后、用得最省。④ 一句话：护栏管“明显该拦的”，HITL 管“该不该做得让人判的”，可观测让这两件事都看得见、查得到——三层叠起来才既安全又不拖垮效率。🔑 Key: the three aren't redundant but three layers of defense in depth — guardrails auto-block obvious violations, traces let a human see the context, and HITL is the final “human decides” gate; each later layer is pricier, so each is used more sparingly. (1) L25 guardrails: automatic, cheap, catch the obvious. What a rule / classifier can judge (injection, over-reach, plainly harmful) is rejected outright and should never reach a human — don't bother a person with what a machine can decide. (2) L23 observability: ground the decision. When something does fall to a human, they need to see what this step will do, on what evidence, and how it got here — traces / context are the human's dashboard; without it they're just guessing. (3) L33 HITL: final ruling for the gray zone. Only the irreversible / high-risk cases a rule can't settle go to a human nod — the most expensive layer (needs a person, needs waiting), so it sits last and is used most sparingly. (4) In a line: guardrails handle “the obviously blockable”, HITL handles “whether-to-do that needs human judgment”, and observability makes both visible and auditable — stacked, the three are safe without grinding down efficiency.

🎯 3. 现实里人可能几小时甚至隔天才点确认，可你的进程不能一直挂着干等。你怎么设计让 workflow 能挂起、等很久、再恢复？In reality a human may take hours or even a day to confirm, but your process can't sit there blocking. How do you design the workflow to pause, wait a long time, then resume?
看参考要点Show key points 点击展开expand
🔑 重点：把“等人”做成异步 + 可持久化——发出 InputRequiredEvent 后序列化 workflow 的上下文（Context）落库，进程可以释放；人回来时再从存档恢复、灌入 HumanResponseEvent 续跑，而不是让线程死等。① 别阻塞等待：挂起点不该占着进程 / 线程空转。拿到 InputRequiredEvent 就把待批请求 + 上下文快照存起来（ctx.to_dict() 落 DB / 队列），释放资源。② 持久化状态：workflow 的 Context 可序列化——这正是“等很久”的关键：状态在库里而不在内存里，进程重启 / 扩缩容都不丢。③ 回来再恢复：人在前端 / 审批系统点了确认，后端用存档重建 Context（Context.from_dict），send_event(HumanResponseEvent(response=…)) 让消费它的 step 续跑。④ 配套要有超时与对账：长时间没人理要能超时走默认或提醒重派，并保证“同一请求只批一次”（幂等），别因为重试把高风险动作执行两遍。🔑 Key: make “waiting on a human” async + persistable — after emitting InputRequiredEvent, serialize the workflow's Context to storage and let the process go; when the human returns, restore from that snapshot and inject the HumanResponseEvent to continue, rather than blocking a thread. (1) Don't block-wait: a pause point shouldn't hold a process / thread spinning. On InputRequiredEvent, persist the pending request + a context snapshot (ctx.to_dict() to DB / queue) and release resources. (2) Persist the state: the workflow's Context is serializable — exactly what makes “wait a long time” possible: state lives in storage, not memory, so restarts / autoscaling don't lose it. (3) Resume on return: when the human confirms in a frontend / approval system, the backend rebuilds the Context from the snapshot (Context.from_dict) and send_event(HumanResponseEvent(response=…)) lets the consuming step continue. (4) Pair it with timeouts and reconciliation: a long-unanswered request must time out to a default or nudge / reassign, and guarantee “each request is approved once” (idempotent) so a retry doesn't execute a high-risk action twice.
