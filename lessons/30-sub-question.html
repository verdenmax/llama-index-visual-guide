查询分解 Sub-QuestionQuery Decomposition

到这里，检索路径上的每一课都默认同一个形状：一个问题 → 一次检索 → 取回 top-k → 合成答案。可现实里有一整类问题，单次 top-k 天生答不全——“2023 比 2022 营收增长多少”“先查 X 是什么、再查它的兼容机型”。这些对比 / 跨源 / 多步问题，答案不现成写在某一段里，而要分头查好几处再合并；单次检索要么只召回一边，要么把两边混在一起。SubQuestionQueryEngine 换个思路：先把母问题拆成子问，各自检索后再汇总。Every lesson on the retrieval path so far assumed one shape: one question → one retrieval → fetch top-k → synthesize. But a whole class of questions can't be answered by a single top-k — “how much did revenue grow in 2023 vs 2022”, “first find what X is, then its compatible models”. For these comparison / cross-source / multi-step questions the answer isn't written ready in any one passage but must be looked up in several places and combined; a single retrieval either recalls only one side or mixes both together. SubQuestionQueryEngine flips the approach: first split the parent into sub-questions, retrieve each, then aggregate.

痛点：对比 / 跨源 / 多步问题，单次 top-k 答不全The pain: comparison / cross-source / multi-step questions a single top-k can't answer

朴素 RAG 的检索是“一次性”的：把问题编码成一个向量，取回最相似的 top-k 个片段，再让 LLM 据此作答。对“X 的退款政策是什么”这种单点问题，这一次检索就够了。但碰到“2023 比 2022 营收增长多少”，麻烦立刻显现：答案要同时拿到“2023 营收”和“2022 营收”两个事实、再相减，而它们往往分散在两份文档（两年的财报）里。单次 top-k 只有固定的几个名额：要么向量更偏向其中一年、把另一年挤出召回（漏一边），要么两年的片段混进同一批、彼此稀释，关键数字反而没进前 k（混在一起）。
Plain RAG's retrieval is “one-shot”: encode the question into a vector, fetch the top-k most similar chunks, then let the LLM answer from them. For a single-point question like “what is X's refund policy”, that one retrieval suffices. But ask “how much did revenue grow in 2023 vs 2022” and the trouble shows immediately: the answer needs two facts at once — “2023 revenue” and “2022 revenue” — then a subtraction, and those usually sit in two different documents (two years' reports). A single top-k has only a few fixed slots: either the vector leans toward one year and crowds the other out of recall (missing one side), or both years' chunks land in the same batch and dilute each other so the key number never makes the top-k (mixed together).
⚠️
一个问题里其实藏着好几个独立的检索需求——硬塞进一次 top-k 必然顾此失彼：召回一边就漏另一边。
One question actually hides several independent retrieval needs — force them into a single top-k and it must shortchange some: recall one side and you miss the other.
跨源问题（“我们的产品和竞品在 X 上谁更强”要查两套资料）、并行多部分问题（“把 2021、2022、2023 三年的营收各查一遍再汇总”——几个子问彼此独立、可并行）也都是同一个结构性短板的不同面孔：答案分散、需求并列，单次召回天生覆盖不全。
Cross-source questions (“are we or the competitor stronger on X” needs two sets of material) and parallel multi-part questions (“look up revenue for 2021, 2022 and 2023 each, then combine” — independent sub-questions that run in parallel) are different faces of the same structural weakness: the answer is scattered and the needs are side by side, so one recall inherently can't cover them all.

SubQuestion：拆母问题 → 路由到各工具 → 各自检索 → 汇总SubQuestion: split the parent → route to tools → retrieve each → aggregate

SubQuestionQueryEngine 把“一次检索”换成“先拆、再分别查、最后合”四步。① 拆：把母问题连同每个工具的名字与描述一起交给 LLM，让它生成一组子问题，并为每个子问题指定该用哪个工具（这一步是 LLM 驱动的）；② 路由：每个子问题被发往它对应的 QueryEngineTool——工具其实就是“一个查询引擎 + 一段说明它擅长什么的 metadata”，描述写得准不准，直接决定子问会不会被送对地方。
SubQuestionQueryEngine replaces “one retrieval” with four steps — split, route, retrieve separately, combine. (1) Split: hand the parent question plus each tool's name and description to the LLM and have it generate a set of sub-questions, each tagged with which tool to use (this step is LLM-driven); (2) route: each sub-question goes to its matching QueryEngineTool — a tool is just “a query engine plus metadata describing what it's good at”, and how accurately that description is written directly decides whether the sub-question is sent to the right place.
③ 各自检索：每个工具在自己的数据上独立跑一次检索与作答，互不干扰，于是“2023 营收”只在 2023 的库里找、“2022 营收”只在 2022 的库里找；④ 汇总：引擎把所有子问题的答案收集起来，作为上下文交给 LLM 合成最终答案（比如把两年的营收相减）。关键在于：母问题里“藏着的几个检索需求”被显式拆成了独立的几次检索，每一次都拥有自己完整的 top-k 名额，谁也不挤占谁。
(3) retrieve separately: each tool runs its own retrieval and answer over its own data independently, so “2023 revenue” is sought only in the 2023 store and “2022 revenue” only in the 2022 store; (4) aggregate: the engine collects every sub-answer and feeds them as context to the LLM to synthesize the final answer (e.g. subtract the two years). The key: the “several retrieval needs hidden in the parent” are made explicit as several independent retrievals, each with its own full top-k budget, none crowding out another.
🔑
Sub-Question 是一次性并行拆解：子问彼此独立、可并行，但不能把某个子答案回填进下一个子问；真正“前一步答案喂下一步”的依赖链要交给 Agent（见 L32）。
Sub-Question decomposes once, in parallel: the sub-questions are independent and run concurrently, but it cannot feed one sub-answer into the next sub-question; a true dependent chain (step 2 needs step 1's answer) belongs to an Agent (see L32).
两个引擎各包成带 description 的工具，SubQuestion 自动拆问、路由、各查、汇总——描述决定路由Wrap each engine as a tool with a description; SubQuestion auto-splits, routes, retrieves each, aggregates — the description drives routing
from llama_index.core.query_engine import SubQuestionQueryEngine
from llama_index.core.tools import QueryEngineTool, ToolMetadata

tools = [
    QueryEngineTool(query_engine=qe_2022, metadata=ToolMetadata(name="y2022", description="2022 年财报")),
    QueryEngineTool(query_engine=qe_2023, metadata=ToolMetadata(name="y2023", description="2023 年财报")),
]
engine = SubQuestionQueryEngine.from_defaults(query_engine_tools=tools)
print(engine.query("2023 年的营收比 2022 年增长了多少？"))   # 自动拆成两个子问题再相减

与 L18 的区别：换更强的检索器 vs 拆问题 + 多引擎编排vs L18: a stronger retriever vs decomposing the question + orchestrating engines

这一课很容易和 L18 的进阶检索搞混，但两者其实在不同维度上发力。L18（混合检索、rerank、HyDE）解决的是“同一次检索，怎么找得更准”——它换的是检索器本身：多一路 BM25、加一层交叉编码器精排、先造个假设文档再查，目标是让那一批 top-k 的质量更高。但无论检索器多强，它仍是“一个问题、一次检索”——对“2023 比 2022 增长多少”这种内含多个子需求的问题，再准的单次召回也答不全。
This lesson is easy to confuse with L18's advanced retrieval, but the two work on different axes. L18 (hybrid retrieval, rerank, HyDE) solves “within one retrieval, how to find more accurately” — it swaps the retriever itself: add a BM25 path, layer on a cross-encoder rerank, draft a hypothetical document first — all to make that one batch of top-k higher quality. But however strong the retriever, it's still “one question, one retrieval” — and for a question like “growth 2023 vs 2022” that hides several sub-needs, even a perfect single recall can't answer it fully.
SubQuestion 动的是另一个维度：它不改检索器，而是改“检索的次数与编排”——把一个问题拆成多个子问题、分别路由到多个引擎、最后汇总。
SubQuestion moves a different axis: it doesn't change the retriever, it changes “how many retrievals and how they're orchestrated” — splitting one question into sub-questions, routing each to a different engine, and aggregating.
📌
两者正交、可叠加：每个子问题底下，完全可以挂一个用了 L18 全套（混合 + rerank）的强检索引擎——SubQuestion 负责“拆对、问对引擎”，L18 负责“每一问都查得准”。
Orthogonal and stackable: under each sub-question you can absolutely hang a strong engine running L18's full kit (hybrid + rerank) — SubQuestion handles “split right, ask the right engine”, L18 handles “each question retrieves accurately”.

🧩 生活类比🧩 Analogy
像做一道复杂的大题：与其抱着一本书从头翻到尾、想一次找全（单次 top-k），不如先把大题拆成几个小问——“2023 年营收是多少？”“2022 年营收是多少？”——分别翻到对应的那一章查清楚，最后把两个小答案合起来相减得出大题的答案。复杂问题难的从来不是某一小步，而是“得分头查好几处再合并”；SubQuestion 干的就是替你拆题、分头查、再合卷。
Like solving a hard exam problem: rather than thumbing through one whole book hoping to find everything in a single pass (one top-k), first break it into small questions — “what was 2023 revenue?”, “what was 2022 revenue?” — look each up in its own chapter, then combine and subtract the two small answers into the big one. The hard part of a complex question was never any single step but “looking several places up separately and merging”; SubQuestion does exactly that for you — split, look up separately, then collate.
✅ 本课要点✅ Key points
  • 对比 / 跨源 / 多步问题，单次 top-k 天生答不全——要么漏一边，要么两边混在一起挤占名额。Comparison / cross-source / multi-step questions are inherently beyond a single top-k — it misses one side or mixes both and crowds the slots.
  • SubQuestionQueryEngine：LLM 把母问题拆成子问题 → 每个子问路由到对应 QueryEngineTool → 各自检索作答 → 汇总成最终答案。SubQuestionQueryEngine: the LLM splits the parent into sub-questions → each routes to its matching QueryEngineTool → each retrieves and answers → aggregate into the final answer.
  • QueryEngineTool = 查询引擎 + ToolMetadata（name / description）；description 写准与否直接决定子问路由对不对。A QueryEngineTool = a query engine + ToolMetadata (name / description); how accurately the description is written directly decides whether a sub-question routes correctly.
  • 与 L18 正交：L18 换更强的检索器（让一次检索更准），SubQuestion 改检索的次数与编排（拆 + 路由 + 汇总）；子问底下可叠加 L18 强引擎。Orthogonal to L18: L18 swaps in a stronger retriever (one retrieval, more accurate), SubQuestion changes the count and orchestration of retrievals (split + route + aggregate); you can stack an L18 engine under each sub-question.
💡 设计亮点💡 Design insight
Sub-Question 的精髓，是把检索从“一个问题配一次 top-k”升级成“一个问题配一套检索计划”。朴素 RAG 默认每个问题只对应一次召回，于是凡是内含多个独立事实的问题——对比、跨源、多步——都被那固定的 top-k 名额卡住：召回一方就漏另一方。Sub-Question 让 LLM 先把问题拆解成几个能被单独回答的子问，每个子问拥有自己完整的检索预算、被路由到最懂它的引擎，最后再汇总——本质上是把“检索”从一个固定动作，变成了一段可由 LLM 规划的编排。
The essence of Sub-Question is upgrading retrieval from “one question, one top-k” to “one question, a retrieval plan”. Plain RAG assumes each question maps to a single recall, so any question hiding several independent facts — comparison, cross-source, multi-step — gets stuck in those fixed top-k slots: recall one side and you miss the other. Sub-Question has the LLM first decompose the question into sub-questions that can each be answered on their own, give each its own full retrieval budget routed to the engine that knows it best, then aggregate — turning “retrieval” from one fixed action into an orchestration the LLM can plan.
要记住三件事：① 它和 L18 正交——L18 让每一次检索更准，Sub-Question 决定要检索几次、各问谁，两者叠加最强；② 成本是实打实多花的——拆问要一次 LLM 调用，n 个子问就是 n 次检索 + 多次生成，延迟和 token 都翻几倍，别为单点问题滥用；③ 路由的命脉是工具的 description——描述写歪了，子问就被送错引擎，再好的拆解也白费。
Remember three things: (1) it's orthogonal to L18 — L18 makes each retrieval more accurate, Sub-Question decides how many retrievals and whom to ask; stacked, they're strongest; (2) the cost is real and additive — splitting takes an LLM call, and n sub-questions mean n retrievals plus several generations, multiplying latency and tokens, so don't overuse it on single-point questions; (3) routing lives or dies by the tool's description — write it wrong and sub-questions go to the wrong engine, wasting even a perfect split.
一句话：当一个问题其实是好几个问题时，先替它拆题、分头查、再合并。它也为后面的 Router（选一条路）与 Agent（动态多步循环）埋下伏笔——同样是“编排多个能力”，只是编排的灵活度一路递增。
In a line: when one question is really several, split it, look each up separately, then combine. It also sets up the later Router (pick one path) and Agent (a dynamic multi-step loop) — all “orchestrating multiple capabilities”, just with steadily rising flexibility.

🔑 重点：拆解和路由都由 LLM + 工具描述驱动，所以先把“工具 description 写准”（决定路由对不对），再限定子问数量、给每个子问加 trace 让它可观测（呼应 L23）。① 工具 description 写准：子问被送到哪个引擎，全看工具的 ToolMetadata.description——把每个工具“擅长什么、覆盖什么数据”写清楚、彼此边界不重叠，子问才不会被路由错（这是最常见的“跑偏”根因）；② 限定子问数量 / 收敛拆解：约束最多拆几个子问，避免把单点问题过度拆解，既省成本又减少噪声；③ 每个子问加 trace、可观测：接入 L23 的可观测（回调 / Phoenix 等），把“拆出了哪些子问、各路由到哪个工具、各自召回了什么”一条条打出来——拆错时一眼看出是“拆的步骤错”还是“某个子问检索错”，分而治之；④ 评估闭环：用一组多步 / 对比金标问题看端到端答案对不对，针对错例回去改工具描述或拆解提示。一句话：拆错先别怪模型，多半是工具描述没写清、或没给它可观测的眼睛。🔑 Key: both the split and the routing are driven by the LLM plus the tool descriptions, so first “get the tool descriptions right” (they decide whether routing is correct), then cap the number of sub-questions and add a trace to each so it's observable (echoing L23). (1) Write tool descriptions accurately: which engine a sub-question goes to depends entirely on each tool's ToolMetadata.description — state clearly what each tool is good at and which data it covers, with non-overlapping boundaries, so sub-questions don't get misrouted (the most common “drift” root cause); (2) cap the sub-question count / tame the split: constrain how many sub-questions can be produced, avoiding over-splitting a single-point question — saving cost and cutting noise; (3) add a trace to each sub-question: wire in L23's observability (callbacks / Phoenix, etc.) and print out “which sub-questions were produced, which tool each was routed to, what each recalled” — when it splits wrong you see at a glance whether it's “the split step” or “one sub-question's retrieval”, and divide and conquer; (4) an eval loop: use a set of multi-step / comparison gold questions, check end-to-end answer correctness, and fix tool descriptions or the split prompt on the failures. In a line: don't blame the model first — bad splits are usually unclear tool descriptions or missing observability.

🔑 重点：三者在“编排灵活度”上递增——Router 从多条路里选一条走；SubQuestion 一次把问题拆成多条、并行各查、再汇总（固定的一轮）；Agent 则动态循环多步，边做边决定下一步（为 L32 铺垫）。① Router（选一条）：一个问题 → 选择器挑最合适的那一个引擎/工具 → 走完即止；适合“这问题该归谁管”清晰、但本质还是单路的场景。② SubQuestion（拆多条、一轮汇总）：一个问题 → 拆成若干并行子问 → 各自检索 → 一次性汇总；适合“对比 / 跨源 / 多步但步骤可预先拆清”的问题，编排是固定的一轮、不回头。③ Agent（动态多步循环）：在一个 think → act → observe 的循环里，根据上一步结果再决定下一步，可调工具、可回退、可迭代；适合步骤事先不知道、要边走边定的开放任务，代价是更慢更贵更难控。怎么选：单路分流用 Router；能预先拆成几问、一轮合并用 SubQuestion；步骤不确定、需要反复试探才用 Agent——别用大锤敲钉子。🔑 Key: the three rise in “orchestration flexibility” — a Router picks one path among many; SubQuestion splits one question into several, retrieves them in parallel, then aggregates (a fixed single round); an Agent runs a dynamic multi-step loop, deciding the next step as it goes (setting up L32). (1) Router (pick one): one question → the selector picks the single most suitable engine/tool → done; good when “who should handle this” is clear but it's still single-path. (2) SubQuestion (split many, one-round aggregate): one question → split into several parallel sub-questions → retrieve each → aggregate once; good for comparison / cross-source / multi-step questions whose steps can be split up front, with a fixed single round and no looping back. (3) Agent (dynamic multi-step loop): inside a think → act → observe loop it decides the next step from the last result, calling tools, backtracking, iterating; good for open tasks whose steps aren't known in advance, at the cost of more latency, money and harder control. How to choose: single-path dispatch → Router; can be pre-split into a few questions merged in one round → SubQuestion; uncertain steps needing trial and error → Agent — don't swing a sledgehammer at a tack.

🎯 3. 上线后发现接入 SubQuestion 让延迟和成本明显上升。你会怎么判断“哪些查询真的需要拆”、把这份开销花在刀刃上？After launch you find SubQuestion noticeably raised latency and cost. How would you decide “which queries truly need splitting” and spend that overhead only where it pays off?
看参考要点Show key points 点击展开expand
🔑 重点：拆解不是免费的——一次拆题 LLM 调用 + n 个子问 = n 次检索 + 多次生成，单点问题根本不该走它；用“问题类型路由 + 数据定阈值”只把对比/多步问题送去拆。① 先分流再拆：用一个轻量分类/路由（回想 L18）判断问题是不是“对比 / 跨源 / 多步”——是才交给 SubQuestion，普通单点问题走一次普通检索就好，别让所有查询都付拆解的钱；② 限子问数量：约束最多拆几个子问，n 越大成本越线性上涨；③ 能并行就并行：子问之间无依赖时并行检索，能砍掉大部分串行延迟（真正的依赖链需要串行、按上一步结果决定下一步，那是 Agent 的活，不是 Sub-Question）；④ 用数据定阈值：统计真实流量里对比/多步问题的占比，对比“全走 SubQuestion vs 仅这一小撮走”的答对率增量与延迟/成本差，确认这份开销值得；⑤ 缓存子答案：高频子问（如“2022 营收”）可缓存复用。一句话：SubQuestion 是给“真有多个子需求”的问题用的精装件，别默认全开。🔑 Key: splitting isn't free — one split LLM call + n sub-questions = n retrievals plus several generations, so single-point questions shouldn't go through it at all; use “question-type routing + data-set thresholds” to send only comparison/multi-step questions to be split. (1) Route before splitting: a lightweight classifier/router (recall L18) decides whether a question is “comparison / cross-source / multi-step” — only then hand it to SubQuestion, while ordinary single-point questions take one plain retrieval; don't make every query pay the split tax; (2) cap the sub-question count: constrain how many sub-questions are allowed — cost rises roughly linearly with n; (3) parallelize when you can: when sub-questions are independent, retrieve them in parallel to cut most of the serial latency (a true dependent chain must run serially, deciding each step from the last result — that's an Agent's job, not Sub-Question's); (4) set the threshold with data: measure the share of comparison/multi-step questions in real traffic and compare “all through SubQuestion vs only that minority” on answer-accuracy lift versus latency/cost delta to confirm the overhead pays; (5) cache sub-answers: frequent sub-questions (e.g. “2022 revenue”) can be cached and reused. In a line: SubQuestion is a premium part for questions that really have several sub-needs — don't leave it on by default.
