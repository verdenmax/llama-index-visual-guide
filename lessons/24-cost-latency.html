到了生产，光“答得对”不够，还得“答得起、答得快”。两个公式钉死优化方向：成本 = token 数 × 调用次数，延迟 = 多步串行累加。对着公式有四把刀（缓存、异步 / 批、流式、换小模型），按“省得多、伤得少”的顺序抡。先记牢一句最容易踩错的：流式优化的是“首字延迟”，不是“总延迟”——用户更早看到字，总时间几乎没变。In production, “correct” isn't enough — it must also be “affordable” and “fast”. Two formulas pin down where to optimize: cost = tokens × number of calls, latency = a serial sum of steps. Against them stand four knives (caching, async / batch, streaming, a smaller model), swung in “most saved, least harmed” order. Memorize the one people get wrong first: streaming optimizes time-to-first-token, not total latency — users see characters sooner, but the total time barely changes.

① embedding 缓存① Embedding cache相同文本不再重复向量化 —— 砍重复 embedding API 的钱（摄取 / 重建索引时最明显）identical text is never re-vectorized — cuts the repeat embedding-API bill (biggest at ingest / re-index)
② LLM 响应缓存② LLM-response cache相同 prompt 直接返回上次答案 —— 同时砍 LLM 生成的钱和那几秒延迟an identical prompt returns last time's answer — cuts both the LLM bill and those seconds of latency
③ 检索-响应缓存③ Retrieval–response cache相同问题缓存整条“检索 + 合成” —— 砍整条链（连检索都省）an identical question caches the whole “retrieve + synthesize” — cuts the entire chain (even retrieval is skipped)
三层缓存各管一段：embedding 缓存砍重复向量化，LLM 响应缓存砍重复生成（连带延迟），检索-响应缓存砍整条链——越靠外，命中一次省得越多Three cache layers, each covering a stretch: the embedding cache cuts repeat vectorization, the LLM-response cache cuts repeat generation (and its latency), the retrieval–response cache cuts the whole chain — the outer the layer, the more a single hit saves
🧩 生活类比🧩 Analogy
缓存像便利店的预制餐：第一次现做（慢、贵），做好摆上货架，下次同样的单子直接拿（近乎零成本）；流式则像餐厅先上一道前菜：整桌菜上齐的时间没变，但你马上有得吃，不用干等。
Caching is like a convenience store's pre-made meals: the first order is cooked fresh (slow, costly), then it sits on the shelf so the same order is grabbed instantly next time (near-zero cost); streaming is like a restaurant serving a starter first: the time for the full table of dishes is unchanged, but you have something to eat right away instead of waiting idle.
同步阻塞（等整段答完）Blocking (wait for the whole answer)
query() 一直阻塞到 LLM 整段生成完才返回，用户盯着空白转圈 3~5 秒才一次性看到全文。首字延迟 = 总延迟，体感“很久没反应”。
query() blocks until the LLM has generated the entire answer, so the user stares at a spinner for 3–5s before seeing anything all at once. Time-to-first-token = total latency, and it feels “unresponsive”.
vs
流式（逐 token 先吐）Streaming (emit token by token)
streaming=True 让 LLM 边生成边吐 token，几百毫秒就冒出第一个字，用户一边读一边等后文。总耗时几乎不变，但首字延迟骤降，体感“立刻在答”。
streaming=True makes the LLM emit tokens as it generates, so the first character appears in a few hundred ms and the user reads while the rest arrives. Total time barely changes, but time-to-first-token plummets, and it feels “answering right away”.
同一句答案：同步要等整段生成完才显示（首字 = 总延迟）；流式逐 token 先吐，首字延迟骤降但总时长几乎不变——流式优化体感，不缩短总延迟Same answer: blocking shows nothing until the whole thing is generated (first-token = total); streaming emits token by token, so first-token drops sharply while total time is nearly unchanged — streaming improves the feel, not the total latency

成本 = token × 调用次数，延迟 = 多步串行Cost = tokens × calls, latency = a serial sum of steps

把账算清楚，优化才有方向。成本几乎全在 token：≈ 每次调用的 token 数 × 调用次数 × 单价——所以要么少调（缓存命中就不调）、要么每次少烧 token（小模型 / 短 context / 控 top_k）。延迟则是一串串行步骤累加：检索 → rerank → 组 prompt → LLM 生成，每步都等前一步做完，LLM 生成通常是大头。
Get the bill straight and optimization has direction. Cost is almost all tokens: ≈ tokens per call × number of calls × unit price — so either call less (a cache hit means no call) or burn fewer tokens per call (smaller model / shorter context / capped top_k). Latency is a serial sum of steps: retrieve → rerank → build prompt → LLM generate, each waiting on the last, with LLM generation usually the big chunk.
💡
对着两个公式下刀，按“省得多、改得轻”排序：缓存永远是第一刀——命中即零 token、零等待，一刀同砍成本和延迟，且几乎不伤质量。
Cut along the two formulas in “most saved, lightest change” order: caching is always the first knife — a hit means zero tokens and zero wait, cutting cost and latency in one stroke, and barely hurting quality.
其余三刀各管一段：② 异步 / 批（aquery 并发多问、批量 embedding）提吞吐、不缩单条；③ 流式逐 token 先吐，只砍首字延迟、提体感，总延迟与成本不变；④ 换小模型 / 小 embedding + 控 top_k 直接压低每次的 token 与算力，省钱常顺带提速，但要盯着质量别掉。
The other three each cover a stretch: (2) async / batch (aquery runs questions concurrently, embeddings in batches) raises throughput without shrinking a single call; (3) streaming emits tokens first, cutting only time-to-first-token and the feel, with total latency and cost unchanged; (4) a smaller model / smaller embedding + capped top_k directly lowers the tokens and compute per call, saving money and often speeding up too, but watch that quality doesn't slip.

缓存命中率越高越省，但数据一变就可能答出过期内容：强时效问题（库存 / 余额）要短 TTL 或不缓存，文档更新要让相关缓存失效。
The higher the cache hit-rate the more you save, but the moment data changes you may serve stale content: give time-sensitive questions (stock, balance) a short TTL or no cache, and invalidate related entries when docs update.

怎么度量：先有基线，才知道砍对没How to measure: a baseline first, or you can't tell you cut right

优化前先定基线，否则砍完不知道有没有用、有没有砍到质量。延迟别只看平均——平均会被少数极慢请求拉偏，要看分位数：p50（一半请求快过它，代表“典型体感”）和 p95（95% 请求快过它，代表“最差也就这样”，决定用户骂不骂）。成本看每问平均成本（总 token 花费 ÷ 问题数）和缓存命中率。
Set a baseline before optimizing, or after a cut you can't tell whether it helped or quietly hurt quality. Don't watch the average for latency — a few very slow requests skew the mean — watch percentiles: p50 (half the requests are faster — the “typical feel”) and p95 (95% are faster — “the worst it usually gets”, which decides whether users complain). For cost, watch average cost per question (total token spend ÷ questions) and cache hit-rate.
🔑
先定基线，再每抡一刀重测：用同一批真实查询对比 p50 / p95 / 每问成本，并跑评测确认忠实度 / 命中率没掉——只有“更快更省且不更差”才算数。
Set a baseline, then re-measure after every knife: on the same batch of real queries compare p50 / p95 / cost-per-question, and run evals to confirm faithfulness / hit-rate didn't drop — only “faster and cheaper and no worse” counts.

🧪 示例：IngestionCache 让重建索引近乎零成本🧪 Example: IngestionCache makes re-indexing near-zero cost
给 IngestionPipeline 挂上 cache=IngestionCache()，它会按每个转换 + 输入文档的哈希缓存输出。第一次 run(documents=docs) 正常切块 + embedding；改一份文档再跑，只有那一份重算，其余命中缓存、不再调 embedding API。pipeline.persist('./pipeline_cache') 把缓存落盘，换进程 / 重启后照样命中——重建索引从“每次全量烧钱”变成“只为改动付费”。
Attach cache=IngestionCache() to an IngestionPipeline and it caches outputs keyed by the hash of each transform + input doc. The first run(documents=docs) chunks + embeds normally; change one doc and re-run, and only that one is recomputed while the rest hit the cache and never call the embedding API. pipeline.persist('./pipeline_cache') writes the cache to disk so it still hits across processes / restarts — re-indexing goes from “burn money on the whole set every time” to “pay only for what changed”.
❓ 为什么“缓存”是第一把刀❓ Why caching is the first knife
因为它性价比最高、风险最低：① 一刀同砍成本和延迟——命中即零 token、零等待，别的刀大多只砍一边；② 不动模型、不动检索质量——只是“别重复算”，几乎不牺牲答案质量（不像换小模型有掉质量风险）；③ 生产里重复查询天然多（热门问题、相同文档重建索引），命中率往往不低。所以先上缓存把“白烧的钱”省掉，再去碰更可能伤质量的小模型 / top_k。唯一要管的是新鲜度：数据变了要让旧缓存失效。
Because it has the best payoff and lowest risk: (1) it cuts cost and latency at once — a hit means zero tokens and zero wait, while most other knives cut only one side; (2) it touches neither the model nor retrieval quality — it just “stops recomputing”, barely sacrificing answer quality (unlike a smaller model, which risks quality); (3) production naturally has many repeats (popular questions, re-indexing the same docs), so hit-rates are often decent. So put caching in first to stop “money burned for nothing”, then touch the quality-risky knives (smaller model / top_k). The one thing to manage is freshness: invalidate stale cache when data changes.
⚙️ 内部怎么跑：流式 response_gen 与 aquery 异步⚙️ How it runs inside: streaming response_gen and async aquery
流式：as_query_engine(streaming=True) 返回 StreamingResponse，它持有一个 response_gen 生成器——LLM 每解码出一个 token 就 yield 出来，print_response_stream() 就是边迭代边打印；所以“首字”在 LLM 刚开口时就到，无需等整段。
Streaming: as_query_engine(streaming=True) returns a StreamingResponse holding a response_gen generator — the LLM yields each token as it decodes it, and print_response_stream() simply iterates and prints; so the “first token” arrives the moment the LLM starts, with no wait for the whole thing.
异步：aquery / aretrieve 是 async 版本，配合 asyncio.gather 能让多个问题并发走完各自的检索 + 生成，或在一次查询里并行跑多路检索——总墙钟时间趋近“最慢那一路”而非“逐条相加”。两者都不改变单条的计算量，改变的是何时拿到第一个字（流式）和多条怎么排布（异步）。
Async: aquery / aretrieve are the async versions; with asyncio.gather they let multiple questions run concurrently through their own retrieve + generate, or run parallel retrieval paths within one query — wall-clock time approaches “the slowest path” rather than “the sum of all”. Neither changes the per-call compute; they change when you get the first token (streaming) and how multiple calls are arranged (async).
🔀 取舍：缓存命中率 vs 新鲜度🔀 Trade-off: cache hit-rate vs freshness
缓存留得越久、范围越宽，命中率越高、越省钱，但越容易答出过期内容。两头要平衡：① TTL（过期时间）——给缓存设有效期，高频但时效性弱的问题（“公司地址”）可长留，强时效的（“今天库存”“当前余额”）短 TTL 或干脆不缓存；② 失效（invalidation）——文档更新就让相关缓存作废，IngestionCache 按输入哈希天然做到“内容变了 key 就变”；③ 分层——embedding 缓存最安全（向量只随文本变），LLM 响应 / 检索-响应缓存要按问题的时效性区别对待。一句话：命中率换新鲜度，按数据多久变一次来定。
The longer and broader you keep the cache, the higher the hit-rate and the more you save — but the easier it is to serve stale content. Balance both ends: (1) TTL — give entries an expiry; popular but low-volatility questions (“company address”) can live long, while time-sensitive ones (“today's stock”, “current balance”) get a short TTL or no cache; (2) invalidation — expire related entries when docs update; IngestionCache does this naturally by input hash (“content changed → key changed”); (3) layering — the embedding cache is safest (vectors only change with text), while the LLM-response / retrieval-response caches must be treated by each question's volatility. In a line: trade hit-rate for freshness, tuned to how often the data changes.

embedding 缓存：给 IngestionPipeline 挂 IngestionCache，第二次跑命中缓存、近乎零成本，persist 跨进程留存Embedding cache: attach IngestionCache to the IngestionPipeline; the second run hits the cache at near-zero cost, and persist keeps it across processes
from llama_index.core import Settings
from llama_index.core.ingestion import IngestionPipeline, IngestionCache
from llama_index.core.node_parser import SentenceSplitter

# embedding 缓存：相同输入直接复用上次向量，避免重复花钱
pipeline = IngestionPipeline(
    transformations=[SentenceSplitter(chunk_size=512), Settings.embed_model],
    cache=IngestionCache())
pipeline.run(documents=docs)          # 第二次跑命中缓存、近乎零成本
pipeline.persist('./pipeline_cache')  # 跨进程持久化缓存
✅ 本课要点✅ Key points
  • 成本 = token × 调用次数，延迟 = 多步串行累加；优化就是“少调 / 每次少烧 / 让步骤别干等”。Cost = tokens × calls, latency = a serial sum of steps; optimizing means “call less / burn fewer tokens / stop steps idling”.
  • 四把刀按序抡：缓存（同砍成本与延迟、第一刀）→ 异步 / 批（提吞吐）→ 流式（只砍首字延迟）→ 小模型 / 小 embedding + 控 top_k（压 token，但盯质量）。Swing the four knives in order: caching (cuts both cost and latency — first) → async / batch (throughput) → streaming (only time-to-first-token) → smaller model / embedding + capped top_k (fewer tokens, but watch quality).
  • 流式优化的是首字延迟，不是总延迟：用户更早看到字，答完整段话的总时间几乎不变。Streaming optimizes time-to-first-token, not total latency: users see characters sooner, but finishing the whole answer takes about the same time.
  • 先定基线再砍：看 p50 / p95 延迟和每问成本，每抡一刀用同一批查询重测、并跑评测确认质量没掉。Set a baseline before cutting: watch p50 / p95 latency and cost per question, re-run the same batch after each knife, and run evals to confirm quality held.
💡 设计亮点💡 Design insight
成本与延迟工程的精髓是先量化、再对着公式下刀：成本盯 token × 调用、延迟盯多步串行，然后按“省得多、伤得少”的顺序抡刀——缓存永远第一（同砍两边、几乎不伤质量），流式只买体感（首字延迟）别指望缩总时长，换小模型 / 降 top_k 最后上且必须用评测兜底。没有 p50/p95 和每问成本的基线，所有“优化”都是自我感动。
The essence of cost & latency engineering is to quantify first, then cut along the formulas: watch tokens × calls for cost and the serial steps for latency, then swing the knives in “most saved, least harmed” order — caching always first (cuts both sides, barely hurts quality), streaming buys only the feel (time-to-first-token, don't expect a shorter total), and a smaller model / lower top_k last, always backstopped by evals. Without a p50/p95 and cost-per-question baseline, every “optimization” is just self-congratulation.

🔑 重点：按“省得多、伤质量小”的顺序——先缓存（几乎不伤质量），再降 top_k / 砍多余 rerank（轻伤、可控），换小模型放最后（最伤质量）；每砍一刀都用固定金标集的忠实度 / 命中率当“质量没掉”的闸，并用每问成本 + p95 证明真省了。① 先缓存：embedding / LLM 响应 / 检索-响应三层，命中即零 token、零延迟，不动模型与检索质量，是白捡的便宜，唯一管好新鲜度（TTL + 失效）；② 降 top_k / 精简 rerank：top_k 从 10 砍到 5、关掉收益不大的 rerank，能省 context token 和一步串行延迟，但可能少召到依据，要看命中率；③ 换小模型 / 小 embedding 放最后：最省钱但最可能掉质量，必须有评测兜底。④ 怎么验证：固定一组真实查询，每步比改前后的每问成本、p50 / p95 延迟，同时用 Faithfulness / Relevancy 或命中率确认质量没跌破阈值——只有“更省且不更差”才保留这一刀，跌了就回退。🔑 Key: order by “most saved, least quality harmed” — caching first (barely hurts quality), then lower top_k / trim redundant rerank (light, controllable), and a smaller model last (most harmful); after each cut, gate “quality held” with faithfulness / hit-rate on a fixed gold set, and prove real savings with cost-per-question + p95. (1) Cache first: three layers (embedding / LLM-response / retrieval-response) — a hit is zero tokens and zero latency, touching neither model nor retrieval quality, free money, just manage freshness (TTL + invalidation); (2) lower top_k / trim rerank: cut top_k from 10 to 5, drop a low-payoff rerank — saves context tokens and one serial step, but may miss evidence, so watch hit-rate; (3) smaller model / embedding last: cheapest but most likely to drop quality, so it must be backstopped by evals. (4) How to verify: on a fixed set of real queries, compare cost-per-question and p50 / p95 latency before/after each step while confirming with Faithfulness / Relevancy or hit-rate that quality stays above threshold — keep a cut only if it's “cheaper and no worse”, otherwise revert.

🔑 重点：流式只砍“首字延迟”、改善体感，不缩短“总延迟”也不省成本；若抱怨的是总时长，得靠缓存 / 更快或更小的模型 / 压 context / 降 top_k，并用 p95 总延迟来证明。① 先分清抱怨的是哪种“慢”：是“干等好几秒才出第一个字”（首字延迟）还是“答得太长、总也答不完”（总延迟）。② 若是首字慢：streaming=True 立竿见影——逐 token 先吐，用户几百毫秒就开始读，但总耗时和 token 成本不变，别拿它当“变快”的银弹。③ 若是总时长慢：流式没用，要从真延迟下手——缓存命中省掉整段生成、换更快 / 更小的模型、压缩 context、降 top_k 少一步串行、把多路检索改异步并发。④ 怎么证明：固定一组真实查询，比较首字延迟（流式该骤降）与p50 / p95 总延迟（真优化才会降），别只看平均；同时跑评测确认质量没掉。一句话回 PM：流式买的是“体感”，不是“总时长”。🔑 Key: streaming only cuts “time-to-first-token” and improves the feel; it does not shorten “total latency” or save cost. If the complaint is about total time, you need caching / a faster or smaller model / trimmed context / lower top_k, proven by p95 total latency. (1) First separate which “slow”: “waiting seconds for the first character” (time-to-first-token) vs “the answer drags on forever” (total latency). (2) If first-token is slow: streaming=True helps instantly — tokens emit first and the user starts reading in a few hundred ms, but total time and token cost are unchanged; don't treat it as a “make it faster” silver bullet. (3) If total time is slow: streaming won't help — attack real latency: cache hits skip whole generations, swap a faster / smaller model, compress context, lower top_k to drop a serial step, make multi-path retrieval async. (4) How to prove: on a fixed query set compare time-to-first-token (should plummet with streaming) and p50 / p95 total latency (only a real optimization moves these), not just the average; and run evals to confirm quality held. In a line to the PM: streaming buys “feel”, not “total time”.

How would you layer the embedding cache, LLM-response cache and retrieval-response cache? Caching's biggest trap is serving stale content — how do you trade off hit-rate vs freshness, and how do you verify the cache truly saves money without hurting users?
看参考要点Show key points 点击展开expand
🔑 重点：按“安全度”分层上——embedding 缓存最安全先上，LLM / 检索-响应缓存按问题时效性配 TTL + 失效；用缓存命中率 + 每问成本证明在省钱，用“过期内容抽检 / 投诉率”守新鲜度。① 分层：embedding 缓存最该先上——向量只随文本变，IngestionCache 按输入哈希天然失效，几乎没有过期风险；LLM 响应 / 检索-响应缓存命中收益最大（连生成都省），但答案会过期，要谨慎。② 命中率 vs 新鲜度：给强时效问题（“今天库存”“当前余额”）短 TTL 或不缓存，弱时效问题（“公司地址”“产品说明”）长留；文档一更新就失效相关缓存，别让旧答案常驻。③ 怎么验证省钱：盯缓存命中率（多少请求白省）和每问平均成本 / p50 延迟（命中后应明显下降）。④ 怎么守住没坑用户：对缓存命中的回答做抽样核对、监控“内容已变但仍答旧值”的投诉 / 点踩率，并把易过期的问题纳入回归评测——省钱不能以答错为代价。🔑 Key: layer by “safety” — the embedding cache is safest and goes first; LLM / retrieval-response caches get TTL + invalidation by each question's volatility; prove savings with cache hit-rate + cost-per-question, and guard freshness with stale-content spot-checks / complaint rate. (1) Layering: the embedding cache should go first — vectors only change with text, IngestionCache invalidates naturally by input hash, almost no staleness risk; the LLM-response / retrieval-response caches have the biggest payoff (they skip generation) but answers can go stale, so be careful. (2) Hit-rate vs freshness: give time-sensitive questions (“today's stock”, “current balance”) a short TTL or no cache, and low-volatility ones (“company address”, “product spec”) a long life; the moment a doc updates, invalidate related entries so old answers don't linger. (3) Proving savings: watch cache hit-rate (how many requests are free) and average cost per question / p50 latency (should drop noticeably on hits). (4) Guarding users: spot-check cached answers, monitor the complaint / thumbs-down rate for “content changed but old value still served”, and add staleness-prone questions to the regression evals — savings must never come at the cost of wrong answers.
