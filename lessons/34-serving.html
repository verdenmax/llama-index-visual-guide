前面 33 课，我们的 RAG 一直活在 notebook 里：建一次索引、问几个问题、看看效果——能跑一次就算赢。可真正交付时，它得变成一个 7×24 营业的服务：通过 HTTP 同时接很多人、要稳要快。从原型到服务，最该避免的一件事就是——每来一个请求就重建索引：建库本是一次性的重活，摊进每一次查询就又慢又浪费、并发一上来就垮。正解只有一句：索引在启动时一次加载、常驻内存，查询引擎复用，每个请求只跑「检索+合成」这段轻活。For 33 lessons our RAG has lived in a notebook: build the index once, ask a few questions, eyeball the result — running once is a win. But to actually ship it, it must become a 24/7 service: taking many people's requests over HTTP at once, staying stable and fast. Going from prototype to service, the one thing to avoid most is rebuilding the index on every request: building the library is a one-time heavy job, and smearing it across every query is slow, wasteful, and collapses under concurrency. The fix is one sentence: load the index once at startup, keep it resident in memory, and reuse the query engine, so each request runs only the light “retrieve + synthesize” part.

上线第一课：把建库成本留在启动，别摊进每次请求Serving rule #1: keep the build cost at startup, don't smear it into every request

① 从 notebook 到服务：索引一次加载常驻，别每请求重建① From notebook to service: load the index once and keep it resident — don't rebuild per request

notebook 和服务，活法完全不同。在 notebook 里，你从上往下跑一遍：建索引那个 cell 只执行一次，之后的 query 都复用内存里那个 index——跑通一次就够了。可服务是长命的、并发的：它常驻进程、通过 HTTP 同时接很多请求，一天跑几万次。
A notebook and a service live completely differently. In a notebook you run top to bottom: the cell that builds the index runs once, and later query calls reuse that in-memory index — running it once is enough. A service is long-lived and concurrent: it sits resident in a process, takes many requests over HTTP at once, tens of thousands a day.
这时一个在 notebook 里看不出问题的写法会变成致命伤——把建索引写进请求处理函数里。建库是读文件 → 切块 → embedding → 写向量库这一整套重活，慢、烧钱、只需做一次；可一旦它躺在每个请求的代码路径上，就等于每问一个问题都把整个图书馆重建一遍——延迟高到没法用，embedding 费用成倍涨，并发稍一上来内存和算力就被重复建库吃垮。
Here a pattern that looks harmless in a notebook turns fatal — building the index inside the request handler. Building the library is the whole heavy chain of read files → chunk → embed → write the vector store: slow, costly, and needed only once; put it on every request's code path and you rebuild the entire library for every single question — latency too high to use, embedding bills multiplying, and memory/compute crushed by repeated rebuilds the moment concurrency rises.
💡
重活留在启动，请求里只剩轻活：服务启动时一次性把索引加载进内存、得到一个常驻引擎，之后每个请求只复用它跑「检索 + 合成」——别每请求重建。
Keep the heavy work at startup, leave only light work per request: at startup load the index into memory once into a resident engine, then every request just reuses it for “retrieve + synthesize” — never rebuild per request.
顺着这条线，后面三节各补一块：用 FastAPI + async 让「轻活」并发跑得更稳（②），用 persist / load_index_from_storage 让「启动加载」便宜到秒级（③），再用 llama-deploy / create-llama 把多个这样的服务编排起来、快速起项目（④）。
From here, the next three sections each add a piece: FastAPI + async makes the “light work” run concurrently and stably (②), persist / load_index_from_storage makes “startup loading” cheap down to seconds (③), and llama-deploy / create-llama orchestrate many such services and scaffold a project fast (④).
示例服务：模块级一次 load_index_from_storage 常驻、引擎复用；请求里只 await aquery（FastAPI 在 core 之外）Illustrative service: module-level load_index_from_storage once and resident, engine reused; the request only awaits aquery (FastAPI is outside core)
# pip install fastapi uvicorn  —— 以下为示例，FastAPI/llama-deploy 在 core 之外
from fastapi import FastAPI
from llama_index.core import StorageContext, load_index_from_storage

app = FastAPI()
index = load_index_from_storage(StorageContext.from_defaults(persist_dir="./store"))  # 启动时一次加载
qe = index.as_query_engine()

@app.post("/query")
async def query(q: str):
    return {"answer": str(await qe.aquery(q))}   # 异步，承 L24

上线关键：索引常驻、查询引擎复用，别每请求重建serving rule: keep the index resident and reuse the engine — don't rebuild per request

② 用 FastAPI 把查询引擎包成接口：async aquery / 流式扛并发（承 L24）② Wrap the query engine as an endpoint with FastAPI: async aquery / streaming for concurrency (from L24)

有了常驻的查询引擎，剩下的只是「把它接到网上」。FastAPI（pip install fastapi uvicorn，core 之外的 Web 框架，本课只作示例）正合适：它原生 async，和我们 L24 学的 aquery 天生一对。
With a resident query engine, all that's left is “wiring it to the network”. FastAPI (pip install fastapi uvicorn, a Web framework outside core, illustrative here) fits well: it is natively async and a natural match for the aquery we learned in L24.
为什么要 async？因为 RAG 的一次查询，大半时间卡在等 LLM / 等向量库返回——这是 I/O 等待，不是 CPU 在算。同步的 query 会把整个进程堵死在这一次等待上，后面排队的请求只能干等；而 async def + await qe.aquery(q) 在等待时把控制权交还事件循环，让别的请求继续跑——同样一台机器，并发吞吐立刻上一个台阶（承 L24 的「异步/批」）。
Why async? Because most of a RAG query's time is spent waiting on the LLM / the vector store — that's I/O wait, not CPU work. A synchronous query blocks the whole process on that one wait, so queued requests just sit idle; whereas async def + await qe.aquery(q) yields control back to the event loop while waiting, letting other requests keep running — on the same machine, concurrent throughput jumps a level (L24's “async / batch”).
另一半是体验：用流式（承 L24）让答案逐 token 边生成边吐，用户几百毫秒就看到第一个字，而不是干等整段生成完——首 token 延迟骤降，总时长几乎不变。要记牢 L24 的那句话：流式优化的是「首字延迟」与体验，不是总耗时。
The other half is experience: streaming (from L24) emits the answer token by token as it generates, so the user sees the first character in a few hundred milliseconds instead of waiting for the whole thing — first-token latency drops sharply while total time barely changes. Remember L24's line: streaming optimizes “first-token latency” and experience, not total time.
📌
分清边界：aquery 是 core 的真方法（直接收一个字符串问题）；FastAPI / uvicorn 是外部工具，只是把这个 core 能力暴露成 HTTP。
Keep the boundary clear: aquery is a real core method (it takes a string question directly); FastAPI / uvicorn are external tools that merely expose that core capability over HTTP.

③ 持久化：persist / load_index_from_storage 让启动即秒级恢复（承 L11）③ Persistence: persist / load_index_from_storage for second-level startup recovery (from L11)

「启动时一次加载」要真便宜，前提是别在启动时也重建一遍。如果每次服务启动都从原始文档重新切块、重新 embedding，那「建库的重活」只是从「每请求」挪到了「每次重启 / 每次扩容」——部署一次等几分钟、还白烧 embedding 钱。
For “load once at startup” to be truly cheap, you must not rebuild at startup either. If every service start re-chunks and re-embeds from the raw documents, the “build heavy lifting” has merely moved from “per request” to “per restart / per scale-out” — minutes of wait per deploy and wasted embedding spend.
🔑
persist 落盘一次、load 秒级读回：索引建一次、写一次盘，之后每次启动 / 扩容都读回、别重建。
persist once, load back in seconds: build the index and write it to disk once, then every startup / scale-out reads it back instead of rebuilding.
L11 的持久化正是为此：建一次、落一次盘——index.storage_context.persist(persist_dir="./store") 把 docstore / index / 向量整套写到磁盘；之后服务启动只需 load_index_from_storage(StorageContext.from_defaults(persist_dir="./store"))，秒级把索引读回内存，免去重建。这正是上面 code① 启动那一行在做的事。
L11's persistence exists for this: build once, write once — index.storage_context.persist(persist_dir="./store") writes the docstore / index / vectors to disk; afterward a service start only needs load_index_from_storage(StorageContext.from_defaults(persist_dir="./store")) to read the index back into memory in seconds, skipping the rebuild. This is exactly what the startup line in code① above does.
这两个符号都是实打实的 core（StorageContext 来自 storage_context.py、load_index_from_storage 来自 indices/loading.py），不依赖任何外部框架。
Both symbols are genuine core (StorageContext from storage_context.py, load_index_from_storage from indices/loading.py), depending on no external framework.
再往规模化看一步：当你跑多个副本（多进程 / 多机水平扩展）时，别让每个副本各存一份本地索引，而是让它们共享同一份持久化存储，或干脆把向量接到外置向量库（L09 学过的 Chroma / pgvector / Qdrant…）——这样「加载一次」对每个副本都成立，数据也只有一处真相。
One step toward scale: when you run multiple replicas (multi-process / multi-machine horizontal scaling), don't let each replica keep its own local copy — have them share one persisted store, or connect vectors to an external vector DB (the Chroma / pgvector / Qdrant… from L09) — so “load once” holds for every replica and the data has a single source of truth.

④ 更进一步：llama-deploy 编排多 workflow/服务、create-llama 脚手架起项目④ Going further: llama-deploy orchestrates multi-workflow services, create-llama scaffolds a project

一个 FastAPI 文件足够把单个查询引擎送上线；但当系统长大——L32 的多 agent / 多 workflow、L33 的 HITL 审批都要协同时，你会想要编排而不是手搓。这里有两件 core 之外的官方工具（同样只作示例）：① llama-deploy——把你写好的 workflow 部署成可独立伸缩、能互相通信的网络服务，多个 workflow/服务由它统一编排、扩缩容，等于把 L26–L33 搭的那套从「一个进程里的对象」升级成「一组分布式服务」。
A single FastAPI file is enough to ship one query engine; but as the system grows — L32's multi-agent / multi-workflow and L33's HITL approvals all need to cooperate — you'll want orchestration rather than hand-wiring. Here are two out-of-core official tools (again illustrative only): (1) llama-deploy — deploys the workflows you wrote as independently scalable, mutually communicating networked services, orchestrating and autoscaling many workflows/services, effectively upgrading the L26–L33 stack from “objects in one process” to “a set of distributed services”.
② create-llama——一个脚手架 CLI（npx create-llama），一条命令生成前端 + 后端 + 数据摄取的全栈起手项目，省去你手搓 API、UI、ingestion 的样板，几分钟就能跑起一个能问答的网站。
(2) create-llama — a scaffold CLI (npx create-llama) that generates a full-stack starter with frontend + backend + ingestion in one command, sparing you the boilerplate of hand-building API, UI, and ingestion, so a working Q&A site runs in minutes.
务必记住边界：llama-deploy / create-llama / FastAPI 都不是 core，它们是打包与编排层；真正干活的核心始终没变——还是你早已会的索引、查询引擎、workflow。怎么选：起步 / 单服务，FastAPI 足矣；要从零快速起一个全栈项目，用 create-llama 脚手架；要把多个 workflow/服务编排成分布式系统，再上 llama-deploy。
Keep the boundary firmly in mind: llama-deploy / create-llama / FastAPI are not core — they are a packaging and orchestration layer; the core that does the real work never changed — still the index, query engine, and workflow you already know. How to choose: for a start / single service, FastAPI is enough; to spin up a full-stack project from scratch fast, use the create-llama scaffold; to orchestrate many workflows/services into a distributed system, reach for llama-deploy.

🧩 生活类比🧩 Analogy
把 RAG 上线，就像把实验台上的原型搬进一家7×24 营业的店面。在实验台上，原型能点亮一次就算成功——你亲手接线、跑一遍、记下结果。可开门做生意完全是另一回事：店面要常年不打烊、要同时接待很多客人、上菜要快。
Shipping a RAG is like moving a prototype off the lab bench into a shop open 24/7. On the bench, lighting the prototype up once is success — you wire it by hand, run it through, note the result. Opening for business is another matter entirely: the shop must never close, serve many customers at once, and bring food out fast.
于是你不会每来一位客人就重盖一次厨房（那就是「每请求重建索引」）——厨房开业前就装好、一直备着（索引常驻），客人来了只管下单上菜（检索+合成）；还得能同时招呼多桌（async 并发）、先端上一道前菜稳住客人（流式降首字延迟）；打烊重开也不必重装厨房（持久化）。原型只要能跑一次，店面要稳、要快、要能同时接客——这一步之差，就是 notebook 与服务的全部距离。
So you don't rebuild the kitchen for every customer who walks in (that's “rebuild the index per request”) — the kitchen is installed before opening and kept ready (a resident index), and each guest just orders and is served (retrieve + synthesize); you also wait many tables at once (async concurrency) and bring out a starter first to settle them (streaming cuts first-token latency); closing and reopening needn't reinstall the kitchen (persistence). A prototype only has to run once; a shop has to be stable, fast, and able to serve many at once — that gap is the whole distance between a notebook and a service.

💡 设计亮点💡 Design insight
上线这一课，几乎没有新的 RAG 概念——它的全部重量压在一句运维直觉上：把一次性的重活留在启动，把高频的轻活留在请求。索引常驻、查询引擎复用，别每请求重建——这就是 notebook 与服务之间唯一却致命的那道坎。
This shipping lesson brings almost no new RAG concept — its whole weight rests on one ops instinct: keep the one-time heavy work at startup and the high-frequency light work in the request. Keep the index resident, reuse the query engine, don't rebuild per request — that is the single, fatal step between a notebook and a service.
想清楚这一点，剩下的拼图你早就备齐了：L24 的 async aquery + 流式解决「同时接客」与「首字延迟」，L11 的 persist / load 让「启动即恢复」便宜到秒级；FastAPI 只是把这套 core 能力接到网上的一层皮，llama-deploy / create-llama 则在更高处编排与脚手架——它们都在 core 之外，核心始终是你从第 1 课搭到第 33 课的那条 RAG 管线。
Get that, and the rest of the puzzle you already have: L24's async aquery + streaming handles “serving many at once” and “first-token latency”, and L11's persist / load makes “restore at startup” cheap down to seconds; FastAPI is just a skin that wires this core capability to the network, while llama-deploy / create-llama orchestrate and scaffold one level up — all outside core, the core still being the RAG pipeline you built from L1 through L33.
而「上线」从来不是终点：服务跑起来之后，质量靠全书攒下的那套纵深持续兜住——L22 回归闸在每次发布前挡住退化，L23 trace 在出错时定位是检索还漏在生成，L25 护栏自动挡掉明显违规，必要时再用 L33 HITL 在不可逆的关口让人拍板。一句话收束：上线不是写新魔法，而是把你已经会的东西，按「常驻 + 异步 + 持久化」的纪律稳稳地摆上货架。
And “going live” is never the finish line: once the service runs, quality is held up by the defense in depth the whole book accumulated — L22's regression gate blocks regressions before each release, L23's traces localize whether a miss was in retrieval or generation, L25's guardrails auto-block obvious violations, and L33's HITL hands the call to a human at irreversible gates when needed. To close in a line: shipping isn't writing new magic — it's placing what you already know on the shelf, steadily, under the discipline of “resident + async + persistent”.

答案：BAnswer: B 题目问的是「最该避免」。每个请求都重新加载 / 重建索引会把一次性建库成本（读文件 / 切块 / embedding / 写库）摊到每一次查询上——又慢又浪费，并发一上来就垮，正是上线第一大忌。其余三项恰恰都是该做的正解：「启动时加载一次、查询引擎常驻复用」把重活留在启动、请求里只剩轻活；「用 aquery 异步处理并发」在等 LLM / 向量库时让别的请求继续跑（承 L24）；「把索引持久化到磁盘」让启动用 load_index_from_storage 秒级恢复、免去重建（承 L11）。The question asks what to most avoid. Reloading / rebuilding the index on every request smears the one-time build cost (read files / chunk / embed / write) across every single query — slow, wasteful, and collapsing under concurrency: the cardinal serving sin. The other three are exactly the right things to do: “load once at startup and reuse a resident query engine” keeps the heavy work at startup and leaves only light work in the request; “use aquery to handle concurrency” lets other requests run while waiting on the LLM / vector store (from L24); and “persist the index to disk” lets startup restore in seconds via load_index_from_storage instead of rebuilding (from L11).

🔑 重点：先把「建一次」和「查很多次」彻底分开——索引常驻、只在启动加载一次；再让请求路径全异步、把外部连接池化复用、启动时预热，并用限流给峰值兜底。① 查雪崩的根因：十有八九是把建索引 / 加载写进了请求处理函数——每个请求都重切块、重 embedding，CPU 和内存被重复建库吃垮。把它提到模块级 / 启动钩子，全局只 load_index_from_storage 一次、查询引擎常驻复用。② 请求路径全异步：用 async def + await qe.aquery(q)（承 L24），别在 async 视图里夹同步阻塞调用，否则照样堵死事件循环。③ 连接池 / 复用：向量库、LLM 客户端都用连接池，别每请求新建连接；embedding / LLM 尽量批量。④ 治冷启动：靠持久化（承 L11）让启动 load 是秒级而非重建；启动后预热——先打一条假查询，把模型 / 缓存 / 连接热起来再挂上健康检查（readiness）接流量。⑤ 限流 + 超时：给并发设上限、给慢请求设超时，峰值时排队或拒绝而不是拖垮整机；配合水平扩容多副本共享同一份持久化存储。最后用 L23 的 trace 确认瓶颈到底在检索、LLM 还是 I/O 等待。🔑 Key: first fully separate “build once” from “query many” — keep the index resident, loaded only once at startup; then make the request path fully async, pool and reuse external connections, warm up at boot, and add rate limiting for peaks. (1) Find the avalanche's root cause: nine times out of ten it's building / loading the index inside the request handler — every request re-chunks and re-embeds, crushing CPU and memory with repeated rebuilds. Hoist it to module level / a startup hook so globally you load_index_from_storage once and reuse a resident query engine. (2) Fully async request path: use async def + await qe.aquery(q) (from L24), and never slip a synchronous blocking call into an async view or it still jams the event loop. (3) Connection pooling / reuse: use pools for the vector DB and LLM clients instead of a fresh connection per request, and batch embedding / LLM calls where possible. (4) Cure cold start: lean on persistence (from L11) so startup is a seconds-long load, not a rebuild; then warm up — fire one dummy query to heat models / caches / connections before the readiness check lets traffic in. (5) Rate limit + timeouts: cap concurrency and time out slow requests so peaks queue or shed instead of toppling the box; pair with horizontal replicas sharing one persisted store. Finally use L23's traces to confirm whether the bottleneck is retrieval, the LLM, or I/O wait.
🎯 2. 服务上线只是开始。跑起来之后，你怎么持续保证质量，不让它悄悄退化、出了错也能查、还能在关键处兜底？Going live is just the start. Once it's running, how do you sustain quality — keep it from silently regressing, stay able to debug failures, and have a fallback at the critical points?

🔑 重点：把全书攒下的能力串成一条上线后的质量闭环——L22 回归闸在发布前挡退化、L23 trace 在出错时定位、L25 护栏在运行时自动兜底、必要时L33 HITL 在不可逆处让人拍板；越靠后越贵，所以越靠后用得越省。① 发布前——L22 回归闸：用 BatchEvalRunner 跑一组带标准答案的题，把 Faithfulness / 命中率聚成通过率，低于阈值就挡下这次发布，别让改 prompt / 换模型悄悄把质量带崩。② 出错时——L23 可观测：全链路埋 trace，一条答错的 query 能摊开看检索召回了什么、rerank 留了谁、LLM 烧了多少 token，分清是检索没召到还是生成没用好，而不是瞎改。③ 运行时——L25 护栏：输入挡注入 / 越权，输出挡 PII / 有害 / 离题，机器能判的自动拦，根本不惊动人。④ 灰色地带——L33 HITL：规则判不了、又不可逆 / 高风险的，才在闸口让人点头。⑤ 持续校准：把线上 trace 里暴露的坏例子回灌进 L22 的评估集，让回归闸越用越严——质量不是上线那天的快照，而是一条测→观→兜→改的闭环。🔑 Key: stitch the book's capabilities into a post-launch quality loop — L22's regression gate blocks regressions before release, L23's traces localize failures, L25's guardrails auto-catch at runtime, and L33's HITL hands irreversible calls to a human when needed; each later layer is pricier, so each is used more sparingly. (1) Before release — L22 regression gate: run a labeled question set with BatchEvalRunner, aggregate Faithfulness / hit-rate into a pass rate, and block the release below threshold so a prompt edit / model swap can't quietly tank quality. (2) On failure — L23 observability: trace the whole chain so a wrong answer can be unfolded to see what retrieval returned, what rerank kept, how many tokens the LLM burned, telling a retrieval miss from poor generation instead of guessing. (3) At runtime — L25 guardrails: block injection / over-reach on input and PII / harmful / off-topic on output — what a machine can judge is auto-blocked and never bothers a human. (4) Gray zone — L33 HITL: only the irreversible / high-risk cases a rule can't settle go to a human nod. (5) Keep calibrating: feed the bad cases surfaced in production traces back into L22's eval set so the gate tightens over time — quality isn't a launch-day snapshot but a test → observe → guard → fix loop.

🎯 3. 你的服务带有状态的 chat / workflow（多轮记忆、L33 那种挂起等人的审批流）。要把它水平扩成多副本，光复制进程为什么不够？你会怎么改造？Your service carries stateful chat / workflows (multi-turn memory, L33-style approval flows that pause for a human). To scale horizontally to many replicas, why isn't just cloning the process enough, and how would you re-architect it?
看参考要点Show key points 点击展开expand
🔑 重点：水平扩展的前提是无状态的进程 + 外置的状态——把会话记忆、workflow 的 Context、待批请求统统挪出内存、落到共享存储，让任意副本都能接住任意请求，进程随时可重启 / 扩缩容而不丢状态。① 为什么复制不够：状态粘在某个进程的内存里——会话历史、挂起的 workflow 都在那一台上。负载均衡把下一轮打到另一副本就读不到上下文；那台一重启，等人确认的审批流直接蒸发。② 状态外置：对话记忆放共享存储（Redis / DB），按 session_id 存取，请求进来先load 记忆、答完写回，进程本身无状态。③ workflow 可持久化：承 L33——发出 InputRequiredEvent 挂起时，把 Context 序列化落库（ctx.to_dict()）、释放进程；人回来时任意副本用 Context.from_dict 重建、灌 HumanResponseEvent 续跑——状态在库里不在内存里。④ 索引仍共享：多副本指向同一份持久化存储 / 外置向量库（承 L11），别各建各的。⑤ 配套：会话亲和（sticky session）可减跨副本抖动但不能依赖它存状态；高风险动作要幂等，避免重试 / 故障转移把不可逆操作做两遍。🔑 Key: horizontal scaling needs stateless processes + externalized state — move chat memory, the workflow Context, and pending approvals out of memory into shared storage so any replica can pick up any request and processes can restart / autoscale without losing state. (1) Why cloning isn't enough: state sticks in one process's memory — conversation history and paused workflows live on that box. The load balancer routing the next turn to another replica finds no context; restart that box and an approval flow waiting on a human just evaporates. (2) Externalize state: keep chat memory in shared storage (Redis / DB) keyed by session_id — a request loads memory on entry and writes it back after answering, so the process itself is stateless. (3) Persist the workflow: continuing L33 — when an InputRequiredEvent pauses, serialize the Context to storage (ctx.to_dict()) and release the process; when the human returns, any replica rebuilds it with Context.from_dict and injects HumanResponseEvent to continue — state lives in storage, not memory. (4) Share the index: point all replicas at one persisted store / external vector DB (from L11) instead of each building its own. (5) Supporting bits: sticky sessions can cut cross-replica churn but must not be relied on to hold state; make high-risk actions idempotent so a retry / failover doesn't execute an irreversible op twice.
