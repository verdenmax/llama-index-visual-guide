前面 25 课搭的都是固定管道：每个问题都走同样的“检索 → 合成”，问法再不同，步骤也一模一样。但真实问题常常需要会决策的循环——先看懂问题，自己决定用哪个工具、要不要检索、检索几次，发现不够还能反思后重检索。这就是 Agent：把 QueryEngine 包成工具交给它，它自行决定何时、调用几次。而 Workflow 是把这种多步、事件驱动的流程显式写出来的底层框架——用 @step 方法 + 事件（StartEvent → … → StopEvent）串成图。The first 25 lessons all built a fixed pipeline: every question runs the same “retrieve → synthesize”, identical steps no matter how it's phrased. But real questions often need a deciding loop — read the question first, then decide for itself which tool to use, whether to retrieve, how many times, and if that's not enough, reflect and re-retrieve. That's an Agent: wrap a QueryEngine as a tool and hand it over, and it decides when and how often to call it. A Workflow is the lower-level framework that writes such multi-step, event-driven flows out explicitly — @step methods joined by events (StartEvent → … → StopEvent) into a graph.

Every question runs the same chain: retrieve top-k → synthesize → answer. The steps are hard-wired — it ignores the question's content, never retrieves twice, never switches tools. Fast, cheap, easy to debug, but it falls short on multi-source / multi-step / self-correcting questions.

🧩 生活类比🧩 Analogy
固定管道像按固定流程办事的自助机：无论你问什么，它都用“检索 → 合成”同一套动作吐出结果。Agent 像一位会查资料的专家助理：先听懂你要什么，自己决定翻哪本手册、查几遍，发现答不全还会再查一次，确认够了才开口——更靠谱，但也更慢、更费时。
A fixed pipeline is like a self-service kiosk that runs the same “retrieve → synthesize” motion for every question. An agent is like a research-savvy assistant: it first grasps what you need, decides on its own which manual to open and how many times to check, and if the answer is incomplete it looks again, speaking only once it's sure — more reliable, but also slower and more effortful.

从“写死的链”到“会决策的循环”From a “hard-wired chain” to a “deciding loop”

什么时候该从固定管道升级到 agent？看三个信号：① 多源——答案要跨多个库 / 工具（文档 + 数据库 + 计算），得按问题选工具；② 多步——一个问题要拆成几步、后一步依赖前一步的结果；③ 需自我纠错——第一次检索不够好，要能看结果再决定重检索。
When should you upgrade from a fixed pipeline to an agent? Watch three signals: (1) multi-source — the answer spans several stores / tools (docs + database + computation) and you must pick the tool per question; (2) multi-step — the question breaks into steps where each depends on the previous result; (3) self-correction — the first retrieval isn't good enough and it must look at the result and decide to re-retrieve.
📌
反过来：单源、单步、问法稳定的问题，固定管道就是最优解——别为了“显得高级”上 agent。三个信号一个都不沾，就别升级。
Conversely: for single-source, single-step, stable-phrasing questions a fixed pipeline is the optimum — don't reach for an agent just to “look sophisticated”. If none of the three signals applies, don't upgrade.
但升级是有代价的：每多一轮决策就多一次 LLM 调用，更慢、更贵；执行路径随问题而变、不再确定，更难调试。
But the upgrade has a cost: each extra decision turn is another LLM call — slower and pricier — and the execution path varies per question, no longer deterministic, so it's harder to debug.
⚠️
agent 越自主，trace 与评测就越不可少：路径不确定意味着出了问题更难复现——这正呼应 L23 的可观测性与 L22 的评估，没有 trace 等于盲飞。
The more autonomous the agent, the more indispensable tracing and evaluation become: a nondeterministic path means failures are harder to reproduce — echoing L23's observability and L22's evaluation; without tracing you're flying blind.

❓ agentic RAG 到底解决什么❓ What does agentic RAG actually solve
普通 RAG 每问都盲检一次就合成，遇到“对比退款和换货政策”这种需要查两次再综合的问题就容易答不全。agentic RAG 让模型自己规划：先查退款、再查换货，必要时补查，最后综合——把“一次性检索”变成“按需多次检索 + 推理”，解决多源 / 多步 / 要自我纠错的问题。代价是更多 LLM 调用、更慢更贵。
Plain RAG blind-retrieves once per question and synthesizes, so a question like “compare the refund and exchange policies” that needs two lookups then a synthesis often comes out incomplete. Agentic RAG lets the model plan for itself: look up refunds, then exchanges, fetch more if needed, then combine — turning “one-shot retrieval” into “on-demand multiple retrievals + reasoning”, solving multi-source / multi-step / self-correcting questions. The cost is more LLM calls — slower and pricier.
⚙️ 内部怎么跑：@step 事件 vs 工具循环⚙️ How it runs inside: @step events vs the tool loop
Workflow 是事件驱动的：每个 @step 方法订阅一种输入事件、返回一种输出事件，框架按“谁产出、谁消费”自动把它们连成图——StartEvent 触发第一个 step，某个 step 返回 StopEvent 就结束。FunctionAgent 则是更高层的封装：内部跑一个“调 LLM → 看它要不要调工具 → 执行工具 → 把结果回喂”的循环，直到 LLM 认为够了。本质上 agent 就是一个内置了工具调用循环的 Workflow。
A Workflow is event-driven: each @step method subscribes to one input event and returns an output event, and the framework wires them into a graph by “who produces what, who consumes it” — StartEvent triggers the first step, and a step returning StopEvent ends it. A FunctionAgent is a higher-level wrapper: inside it runs a loop of “call the LLM → see if it wants a tool → run the tool → feed the result back” until the LLM decides it has enough. An agent is essentially a Workflow with a built-in tool-calling loop.
🔀 取舍：固定管道 vs Router vs Agent🔀 Trade-off: fixed pipeline vs Router vs Agent
三档自主度递增：固定管道每问都同样检索，最快最好调，适合单源单步；Router（L18）让 LLM 从几条预设路径选一条，多一次路由调用，适合“几个明确子库二选一”；Agent 让 LLM 循环决定用什么工具、几步、要不要重检索，最强但最慢最贵最难调。按需选最低档：能固定就别路由，能路由就别上 agent——自主度是有代价的能力。
Three rising levels of autonomy: a fixed pipeline retrieves the same way every time — fastest and easiest to debug, fit for single-source single-step; a Router (L18) lets the LLM pick one of a few preset routes, one extra routing call, fit for “choose among a few clear sub-indexes”; an Agent lets the LLM loop to decide which tools, how many steps, whether to re-retrieve — most capable but slowest, priciest, hardest to debug. Pick the lowest level that works: if fixed suffices don't route, if routing suffices don't reach for an agent — autonomy is a capability with a cost.

最小 Workflow：用 @step + 事件把“检索 → 合成”显式串起来，StartEvent 进、StopEvent 出A minimal Workflow: wire “retrieve → synthesize” explicitly with @step + events; StartEvent in, StopEvent out
from llama_index.core import Settings
from llama_index.core.workflow import Workflow, step, StartEvent, StopEvent, Event

class Retrieved(Event):
    nodes: list

class RAGFlow(Workflow):
    @step
    async def retrieve(self, ev: StartEvent) -> Retrieved:
        return Retrieved(nodes=index.as_retriever(similarity_top_k=3).retrieve(ev.query))

    @step
    async def synthesize(self, ev: Retrieved) -> StopEvent:
        return StopEvent(result=str(Settings.llm.complete(f'据此作答：{ev.nodes}')))

✅ 本课要点✅ Key points

  • 把 QueryEngine 用 QueryEngineTool 包成工具交给 FunctionAgent，description 决定 agent 何时调用它。Wrap a QueryEngine as a tool via QueryEngineTool and give it to a FunctionAgent; the description tells the agent when to call it.
  • Workflow 是事件驱动的底层框架：每个 @step 收一种事件、发出下一种，StartEvent 进、StopEvent 出。Workflow is the event-driven base framework: each @step consumes one event and emits the next; StartEvent in, StopEvent out.
  • 升级到 agent 的信号是多源 / 多步 / 需自我纠错；代价是更慢、更贵、更难调（更依赖 L23 的可观测）。Signals to upgrade to an agent are multi-source / multi-step / self-correction; the cost is slower, pricier, harder to debug (leaning harder on L23's observability).
  • 自主度有代价：能用固定管道就别上 Router，能 Router 就别上 Agent。Autonomy has a cost: if a fixed pipeline works don't add a Router, if a Router works don't reach for an Agent.
💡 设计亮点💡 Design insight
Agent 与 Workflows 的精髓，是把 RAG 从“写死的链”升级成“会决策的循环”：Workflow 用 @step + 事件把多步流程显式化，FunctionAgent 在其上内置“调 LLM → 选工具 → 执行 → 回喂”的循环，让模型按问题自己规划检索几次、用什么工具、要不要重检索。
The essence of agents & workflows is upgrading RAG from a “hard-wired chain” to a “deciding loop”: a Workflow makes multi-step flows explicit with @step + events, and a FunctionAgent builds a “call the LLM → pick a tool → run → feed back” loop on top, letting the model plan for itself how many retrievals, which tools, and whether to re-retrieve.
这把能力上限抬高了——多源、多步、自我纠错都能做——但每多一轮自主就多一次 LLM 调用，更慢、更贵、执行路径不再确定、更难调，因此越 agentic 越离不开 L23 的 trace 与 L22 的评测。所以真正的工程判断不是“能不能上 agent”，而是“这个问题配不配得上 agent 的代价”——能用固定管道解决的，就别上 agent。
That raises the ceiling — multi-source, multi-step, self-correction all become possible — but each extra turn of autonomy is another LLM call: slower, pricier, with a no-longer-deterministic path that's harder to debug, so the more agentic you go, the more you depend on L23's tracing and L22's evaluation. The real engineering judgment isn't “can we use an agent” but “is this question worth an agent's cost” — if a fixed pipeline can solve it, don't reach for an agent.

答案：CAnswer: C agent 的价值在于会决策的多步循环：当一个问题需要多源 / 多步 / 看中间结果再自我纠错（比如“对比退款和换货政策”要分别查两次再综合）时，固定的一次性检索就答不全，才值得上 agent。而“问得简单稳定”恰恰应该留在固定管道（更快更好调）；“想更快出字”是流式（streaming）的范畴；agent 反而增加 LLM 调用、抬高 token 成本和延迟，绝不是降本手段。一句话：为多步决策付费，别为简单问题买单。An agent's value is its deciding, multi-step loop: when a question needs multiple sources / steps / self-correction from intermediate results (e.g. “compare the refund and exchange policies” needs two lookups then a synthesis), a fixed one-shot retrieval can't answer fully — that's when an agent earns its keep. A “simple, stable” question should stay on a fixed pipeline (faster, easier to debug); “answer faster” is what streaming is for; and an agent adds LLM calls, raising token cost and latency — never a cost-cutting move. In a line: pay for multi-step decisions, not for simple questions.

🔑 重点：只为“多步决策”付费——能用固定管道 / Router 解决就别上 agent；评测要同时盯“结果质量 + 过程行为 + 成本延迟”三类指标，并靠 L23 的 trace 看它每一步在干嘛。① 怎么权衡：按需选最低自主度——单源单步用固定管道，几个明确子库二选一用 Router，多源 / 多步 / 需自我纠错才上 agent。② 评测结果对不对：用 L22 的金标集 + Faithfulness / Correctness 看答案质量。③ 评测过程行为：看 agent 轨迹——工具选对没（错误工具调用率）、检索了几次（步数分布）、有没有反复重试 / 死循环。④ 评测成本：每问 LLM 调用次数、token、p95 延迟，和固定管道基线对比。⑤ 靠什么看：agent 是黑箱，必须靠 trace 把“决策 → 工具 → 结果”每步记录下来，否则无从评测、无从止损。🔑 Key: pay only for “multi-step decisions” — if a fixed pipeline / Router can solve it, don't reach for an agent; evaluate three metric families at once — “answer quality + process behavior + cost & latency” — and use L23's traces to see what it does at each step. (1) How to weigh: pick the lowest autonomy that works — fixed pipeline for single-step, Router to choose among a few clear sub-indexes, and an agent only for multi-source / multi-step / self-correction. (2) Is the answer right: use L22's gold set + Faithfulness / Correctness. (3) Process behavior: inspect the trajectory — did it pick the right tool (wrong-tool-call rate), how many retrievals (step distribution), any retry storms / loops. (4) Cost: LLM calls per question, tokens, p95 latency, compared against the fixed-pipeline baseline. (5) How you see it: an agent is a black box, so traces must record “decision → tool → result” at every step — otherwise you can neither evaluate nor stop the bleeding.

🎯 2. agent 自己调用工具时，可能选错工具、反复重试、甚至陷入死循环，把延迟和成本打爆。你在设计和上线时会怎么防？又怎么验证它真的不会失控？When an agent calls tools on its own, it may pick the wrong tool, retry endlessly, or even loop forever, blowing up latency and cost. How would you defend against this in design and at launch — and how do you verify it really won't run away?
看参考要点Show key points 点击展开expand
🔑 重点：给 agent 设“护栏”——精准的工具 description、最大步数 / 超时、循环检测、成本预算、高危动作隔离；上线靠 trace + 一组“刁钻用例”回归来验证它不乱来。① 工具描述要准：description 写清“什么时候该用我”，agent 全靠它选工具，写得模糊就会选错。② 硬性上限：max_iterations / 超时 / token 预算，超了就停，并降级到固定管道或礼貌拒答。③ 循环检测：同一工具同一参数反复调就熔断。④ 高危动作隔离（呼应 L25）：删除 / 发信 / 下单等绝不由检索内容或 agent 自动触发，必须人工确认。⑤ 怎么验证：维护一组容易诱发多步 / 重试的刁钻用例纳入回归，断言步数、成本、是否答对都在阈值内；线上监控每问步数与成本的 p95，越界即告警。🔑 Key: give the agent “guardrails” — precise tool descriptions, a max-step / timeout cap, loop detection, a cost budget, and high-risk-action isolation; verify with traces plus a regression suite of “tricky” cases. (1) Precise descriptions: the description must spell out “when to use me” — the agent picks tools entirely from it, so vagueness causes wrong picks. (2) Hard caps: max_iterations / timeout / token budget; on exceed, stop and fall back to a fixed pipeline or a polite refusal. (3) Loop detection: trip a breaker when the same tool is called with the same args repeatedly. (4) High-risk-action isolation (echoing L25): deletion / sending mail / placing orders must never be triggered automatically by retrieved content or the agent — require human confirmation. (5) How to verify: keep a regression suite of step/retry-inducing tricky cases, asserting step count, cost and correctness stay within thresholds; in production, monitor the p95 of steps and cost per question and alert on breaches.
🎯 3. 把“对比退款和换货政策”这种问题，分别用固定管道和 Workflow / agent 实现，会有什么不同？为什么 Workflow 要用“事件驱动的 @step”而不是直接写一个大函数？你又怎么验证 agent 版“确实多查了一次、且答得更全”？Implement “compare the refund and exchange policies” first with a fixed pipeline and then with a Workflow / agent — what differs? Why does a Workflow use “event-driven @step” instead of just one big function? And how do you verify the agent version “really retrieved an extra time and answered more completely”?
看参考要点Show key points 点击展开expand
🔑 重点：固定管道一次检索就合成、容易只命中一半；Workflow / agent 把“查两次再综合”拆成可观测的步骤，事件驱动让每步解耦、可单测、可插入 / 重排 / 并行；验证靠 trace 数检索次数 + 金标对比答案完整度。① 差异：固定管道盲检一次，往往只召回退款或换货之一→答不全；agent 规划两次检索（先退款、再换货）再综合→更完整。② 为什么用事件驱动 @step：每个 step 只声明“我吃什么事件、产出什么事件”，框架按依赖自动连图——天然解耦、可单独测试、易插入新步骤（如加一步校验 / 重排），还能并行 / 分支，比一个大函数更好维护、更可观测。③ 怎么验证：用 trace 数 agent 实际检索次数（应 ≥ 2），再用 L22 金标集比“固定管道 vs agent”的答案覆盖度 / Correctness，证明多查那一次换来了更全的答案、且延迟 / 成本的增量在可接受范围内。🔑 Key: a fixed pipeline retrieves once then synthesizes and easily catches only half; a Workflow / agent splits “retrieve twice then synthesize” into observable steps, and event-driven steps stay decoupled, unit-testable, insertable / reorderable / parallelizable; verify by counting retrievals in the trace and comparing answer completeness against the gold set. (1) Difference: a fixed pipeline blind-retrieves once and often recalls refunds or exchanges, not both → incomplete; the agent plans two retrievals (refunds, then exchanges) then synthesizes → more complete. (2) Why event-driven @step: each step only declares “what event I consume, what event I emit”, and the framework auto-wires the graph by dependency — naturally decoupled, unit-testable, easy to insert new steps (e.g. a validation / re-rank step) and to parallelize / branch, more maintainable and observable than one big function. (3) How to verify: count the agent's actual retrievals in the trace (should be ≥ 2), then compare “fixed vs agent” answer coverage / Correctness on the L22 gold set, proving the extra lookup buys a more complete answer while the added latency / cost stays acceptable.
