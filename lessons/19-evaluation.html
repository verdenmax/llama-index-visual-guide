🧩 生活类比🧩 Analogy
评估器就像给作文判卷的老师：不是一句“感觉不错”，而是拿固定的几把尺子逐项打分——分数能横向比较、也能复算。
An evaluator is like a teacher grading essays: not a vague “seems good” but scoring against a fixed set of rulers — numbers you can compare across runs and recompute.

把调优变成可度量的闭环Turn tuning into a measurable loop

没有评估，调 RAG 就是“凭感觉”：改了切块、换了检索、调了 Prompt，到底变好还是变坏，全靠主观印象。评估把它变成一个闭环：每次改动后，用同一套问题和指标跑一遍，对比分数再决定保留还是回退。于是每一步优化都有据可依，也能挡住“修好一个、悄悄弄坏一批”的回归。下面用一组真实感的分数，看这个闭环跑起来是什么样。
Without evaluation, tuning RAG is “by feel”: after changing chunking, retrieval or the prompt, whether it improved is a subjective impression. Evaluation turns it into a closed loop: after each change, run the same questions and metrics, compare the scores, then decide to keep or revert. Every optimization becomes evidence-based, and you catch the “fix one, silently break a batch” regressions. The numbers below show one turn of that loop in action.

🧪 示例：评一个回答🧪 Example: grade one answer
FaithfulnessEvaluator().evaluate_response(response=resp) 直接吃 QueryEngine 的返回（已含 source_nodes）；结果里 passing 表示是否“有据可依”，score 给出分值。
FaithfulnessEvaluator().evaluate_response(response=resp) consumes the QueryEngine response directly (it already carries source_nodes); the result's passing says whether it is “grounded”, and score gives the value.
❓ 为什么这么设计❓ Why designed this way
把质量量化，调优才能回归测试：同一套问题在每次改动后复跑，分数升降一目了然，避免“改好一个、悄悄弄坏一批”。
Quantifying quality makes tuning regression-testable: rerun the same question set after each change and the score moves are obvious, so you don't “fix one and silently break a batch”.
⚙️ 内部怎么跑⚙️ How it runs inside
多数评估器是 LLM-as-judge：把问题、答案、上下文（或参考答案）塞进一个评判 Prompt，让 LLM 给判断。Faithfulness / Relevancy 多为二值（passing 是/否），Correctness 则给 1–5 分的 score。
Most evaluators are LLM-as-judge: the question, answer and context (or reference) go into a judging prompt and an LLM rules on it. Faithfulness / Relevancy are usually binary (passing yes/no), while Correctness returns a 1–5 score.
🔀 替代方案🔀 Alternatives
人工评估最准但慢且贵，适合做小规模“金标准”；LLM 评估快且可自动化，适合每次改动的回归。实践中两者结合：维护一个固定的回归问题集，平时用 LLM 评，定期抽样人评来校准。
Human evaluation is most accurate but slow and costly — good for a small “gold standard”; LLM evaluation is fast and automatable — good for per-change regressions. In practice combine both: keep a fixed regression question set, judge with an LLM routinely, and sample human checks periodically to calibrate.
把“感觉还行”变成“可度量”turn “feels fine” into “measurable”
from llama_index.core.evaluation import FaithfulnessEvaluator

resp = index.as_query_engine().query('退款政策是什么？')
ev = FaithfulnessEvaluator()
result = ev.evaluate_response(response=resp)
print(result.passing, result.score)   # 答案是否有据可依
用 BatchEvalRunner 跑一小批问题，得到可对比的基线Run a small batch with BatchEvalRunner for a comparable baseline
from llama_index.core.evaluation import (
    FaithfulnessEvaluator, RelevancyEvaluator, BatchEvalRunner)

# 用一小批问题跑回归：每题同时算忠实度与相关性
runner = BatchEvalRunner(
    {'faith': FaithfulnessEvaluator(), 'rel': RelevancyEvaluator()},
    workers=4,
)
questions = ['退款政策是什么？', '保修期多久？', '支持哪些支付方式？']
results = await runner.aevaluate_queries(
    index.as_query_engine(), queries=questions)

# 汇总通过率，作为本次改动的基线指标
passed = sum(bool(r.passing) for r in results['faith'])
print(f'faithfulness: {passed}/{len(questions)} passed')
✅ 本课要点✅ Key points
  • 三类评估：忠实（防幻觉）、相关（切题）、正确（对参考）。Three checks: faithful (anti-hallucination), relevant (on-topic), correct (vs reference).
  • evaluate_response 直接吃 QueryEngine 的返回（含 source_nodes）。evaluate_response consumes the QueryEngine's response directly (with source_nodes).
  • 评估通常由一个 LLM 充当“裁判”。Evaluation is usually done by an LLM acting as judge.
💡 设计亮点💡 Design insight
评估把 RAG 调优从“凭感觉”变成“可度量的闭环”：改切块、换检索、调 Prompt 后，用同一套指标对比，才知道是真变好了。
Evaluation turns RAG tuning from gut feeling into a measurable loop: after changing chunking, retrieval or prompts, compare on the same metrics to know it actually improved.

🎯 2. 评估用 LLM-as-judge，有哪些局限？怎么缓解、怎么让分数可信？Evaluation uses LLM-as-judge. What are its limits, and how do you mitigate them to make scores trustworthy?
看参考要点Show key points 点击展开expand
🔑 重点：judge 本身有噪声/偏见、可能偏向某种风格、成本高、边界判断不稳。缓解：固定 judge 模型与 prompt、用二值/明确 rubric、对关键集做人评校准、看趋势而非单点分数、维护回归集比较相对变化——把它当“相对尺”而非“绝对真值”。🔑 Key: the judge is noisy/biased, may favor a style, costs money, and is shaky on edge cases. Mitigate by pinning the judge model and prompt, using binary/explicit rubrics, calibrating with human review on a key set, watching trends over single scores, and comparing relative change on a fixed set — treat it as a relative ruler, not absolute truth.
🎯 3. 你只有 200 条历史问答、没有标准答案。怎么起步做评估？You have 200 historical Q&A pairs but no reference answers. How do you start evaluating?
看参考要点Show key points 点击展开expand
🔑 重点：先做不需要参考答案的 Faithfulness 与 Relevancy——它们只看 response/source_nodes/query。Correctness 才需要 reference，可后续对高价值子集人工标注少量 gold 再补。先用前两把尺子建立基线和回归集，立刻就能挡住明显退化。🔑 Key: start with Faithfulness and Relevancy, which need no reference — they look only at response/source_nodes/query. Correctness needs a reference, so hand-label a small gold subset on high-value cases later. The first two rulers establish a baseline and regression set immediately, catching obvious regressions right away.
