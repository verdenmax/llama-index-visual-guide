Embedding 把文本变成向量，让“语义相近”变成“向量距离近”。检索时把问题也向量化，再找最近的若干 Node。模型由 Settings.embed_model 统一配置。Embeddings turn text into vectors so “semantically similar” becomes “close in vector space”. At query time the question is embedded too, then the nearest Nodes are found. The model is configured globally via Settings.embed_model.

同一个 embedding 模型，才有可比的坐标系One embedding model, one comparable space

既然检索靠“向量够不够近”，那“用什么尺子量近”就至关重要。LlamaIndex 默认用余弦相似度，量的是两个向量的夹角——夹角越小越相关。由此引出一个硬约束：查询和文档必须用同一个 embedding 模型。反例很直接——文档用模型 A 向量化、查询却用模型 B，两者落在互不相通的坐标系里，算出来的相似度只是两堆无关数字的巧合，毫无意义。至于换“更强”的模型：通用榜单更高未必在你的领域里召回更好，领域专用或微调过的小模型常常胜过更大的通用模型；而且一旦更换，整个索引都得重新向量化。
Since retrieval hinges on “are the vectors close enough”, what you measure closeness with matters. By default LlamaIndex uses cosine similarity, measuring the angle between two vectors — the smaller the angle, the more related. That implies a hard constraint: the query and the documents must use the same embedding model. The counter-example is stark — embed the documents with model A but the query with model B, and the two live in incompatible coordinate systems; any similarity you compute is a coincidence of unrelated numbers, meaningless. As for switching to a “stronger” model: a higher general-leaderboard score doesn't guarantee better recall in your domain — a domain-specific or fine-tuned smaller model often beats a bigger general one — and any switch forces a full re-embedding of the index.

🧪 示例：把一句话变成向量🧪 Example: turn a sentence into a vector
Settings.embed_model.get_text_embedding('退款政策是什么？') 返回一个定长浮点数组（如 1536 维）。同一模型对“退款要多久”给出的向量，会和它在空间里靠得很近。
Settings.embed_model.get_text_embedding('What is the refund policy?') returns a fixed-length float array (e.g. 1536 dims). The same model's vector for “how long do refunds take” lands right next to it in space.
❓ 为什么这么设计❓ Why designed this way
关键词匹配只认字面，换个说法就失灵。把文本投射到语义向量空间，让“意思相近”而非“用词相同”决定距离，才是语义检索的地基——这也是 RAG 能听懂同义改写的根本原因。
Keyword matching only sees the literal words and breaks on paraphrase. Projecting text into a semantic vector space lets “similar meaning”, not “same wording”, set the distance — the bedrock of semantic search and the reason RAG understands rephrasings at all.
⚙️ 内部怎么跑⚙️ How it runs inside
所有 embedding 实现都遵循 BaseEmbedding 接口：get_text_embedding 给文档用、get_query_embedding 给查询用；检索时用 similarity（默认余弦）给候选打分排序，取分数最高的 top-k。
Every embedding implementation follows the BaseEmbedding interface: get_text_embedding for documents, get_query_embedding for queries; at retrieval time similarity (cosine by default) scores and ranks candidates, keeping the top-k.
🔀 替代方案🔀 Alternatives
关键词检索（如 BM25）按词频精确命中，擅长专有名词与代码符号，但不懂同义。向量检索懂语义却可能漏掉确切关键词。生产里常把两者混合检索（hybrid）再融合排序，兼得字面精确与语义召回。
Keyword search (e.g. BM25) matches exactly by term frequency — great for proper nouns and code symbols but blind to synonyms. Vector search grasps meaning but may miss an exact keyword. In production you often combine them in hybrid retrieval and fuse the rankings — exact and semantic at once.
一次向量化，多次检索复用embed once, reuse across many queries
from llama_index.core import Settings
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.embed_model = OpenAIEmbedding(model='text-embedding-3-small')
v = Settings.embed_model.get_text_embedding('退款政策是什么？')
print(len(v))   # 向量维度，如 1536

# 语义越近，余弦相似度越高（检索就按它排序取 top-k）
同义不同词也能算出高相似度——这正是向量检索强于关键词的地方Paraphrases score high even with different words — exactly why vectors beat keywords
from llama_index.core import Settings
from llama_index.embeddings.openai import OpenAIEmbedding

emb = Settings.embed_model = OpenAIEmbedding(model='text-embedding-3-small')
a = emb.get_text_embedding('退款多久到账？')
b = emb.get_text_embedding('退款需要几天？')      # 近义、不同词
c = emb.get_text_embedding('如何注册新账号？')    # 另一主题

print(round(emb.similarity(a, b), 3))   # 高：语义相近
print(round(emb.similarity(a, c), 3))   # 低：语义无关
✅ 本课要点✅ Key points
  • Embedding 把语义相似变成向量距离，是语义检索的地基。Embeddings turn semantic similarity into vector distance — the bedrock of semantic search.
  • 查询与文档必须用同一个 embedding 模型。Query and documents must use the same embedding model.
  • 统一 BaseEmbedding 接口让你随时换模型而不改主链路。The unified BaseEmbedding interface lets you swap models without touching the pipeline.
💡 设计亮点💡 Design insight
embedding 模型是管道里最“牵一发而动全身”的一环：它定义了整套语义坐标系，所以换 LLM、换切块器都只是局部调整，唯独换 embedding 模型必须把全部向量重算一遍——选型要趁早，更要贴合你的领域。
The embedding model is the pipeline's most load-bearing link: it defines the entire semantic coordinate system, so swapping the LLM or the splitter is a local change — but swapping the embedding model forces you to re-embed every vector. Choose it early, and choose it for your domain.

🎯 2. 用户搜精确条款编号“X-2000”搜不到——字面明明就在文档里。为什么向量会漏？怎么补救？Users search the exact clause id 'X-2000' and miss it — though it's literally in the docs. Why does the vector miss, and how do you fix it?
看参考要点Show key points 点击展开expand
🔑 重点：向量偏“语义”，对精确符号/罕见 token/编号不敏感，语义上“X-2000”与别的编号很近、可能排不进 top-k。补救：混合检索(向量 + BM25/关键词把精确匹配召回回来)、把编号存进 metadata 做精确过滤、或对这类查询加 rerank。🔑 Key: embeddings capture meaning and are weak on exact symbols/rare tokens/ids — semantically 'X-2000' sits near other ids and may miss top-k. Fixes: hybrid retrieval (vector + BM25/keyword to recover exact matches), store the id in metadata for exact filtering, or add a reranker for such queries.
🎯 3. 你想把 embedding 模型从 A 升级到 B。怎么评估“值不值得升”、怎么安全上线？You want to upgrade the embedding model from A to B. How do you assess whether it's worth it, and how do you roll it out safely?
看参考要点Show key points 点击展开expand
🔑 重点：升级 embedding ＝ 全量重算向量、坐标系不可比，必须重建索引后再切——不能热替换。评估：在 staging 用 B 重建一份索引，对同一回归集比较命中率/Faithfulness 与延迟成本；只有提升明显才升。上线：双索引灰度，确认 B 不退化再切流量；切换期间查询必须用与建索引相同的模型。🔑 Key: upgrading the embedding means recomputing all vectors into an incomparable space — you must re-index, then cut over; never hot-swap. Assess by rebuilding an index with B in staging and comparing hit-rate/Faithfulness and latency/cost on the same regression set; upgrade only on a clear win. Roll out with dual indexes and a gradual cutover; during the switch, queries must use the same model the index was built with.
