到这里，检索一直在做同一件事——“找相似”：把问题和文档变成向量、取回最像的片段。可一旦问题是“上个月各区域的总销售额、降序排列”这种要精确算数字的，向量就无能为力：答案要把成千上万行逐笔加总、按区域分组、再排序，不在某段话里现成写着。这一课换一种思路：别让向量算账，把结构化数据交给天生会精确计算的工具——text-to-SQL 让数据库算、PandasQueryEngine 让内存表算；同时盯紧一个绕不开的代价：这些引擎在执行 LLM 生成的代码。Up to here retrieval kept doing one thing — “find similar”: turn the question and docs into vectors and fetch the most alike chunks. But the moment the question is something like “total sales per region last month, sorted descending” that needs an exact number, vectors are helpless: the answer requires summing row by row, grouping by region and then sorting thousands of rows — it isn't sitting ready in any passage. This lesson flips the approach: don't make vectors do the accounting; hand structured data to tools born to compute exactly — text-to-SQL so the database computes, PandasQueryEngine so an in-memory table computes — all while watching one unavoidable cost: these engines execute LLM-generated code.

痛点：数字、聚合、精确筛选，塞进向量必然不准The pain: numbers, aggregation, exact filtering — forced into vectors, bound to be wrong

向量检索的本事是“找像的”：把每段文字编码成语义向量，召回和问题最相似的 top-k 片段。这对“讲了什么”的非结构化文字很灵，碰到“算出确切数字”却立刻露馅。问“上个月每个区域的总销售额、降序排列”，答案不是哪段话里写好的句子，而要把成千上万行订单加总、分组、排序，向量根本不会做加法。
Vector search is good at “finding the alike”: encode each passage into a semantic vector and recall the top-k chunks most similar to the question. That shines on unstructured text about “what was said”, but it falls apart the instant you need an exact number. Ask “total sales per region last month, sorted descending” and the answer isn't a ready-made sentence in some passage — it requires totalling, grouping and sorting thousands of rows of orders, and vectors simply can't do arithmetic.
⚠️
向量连“全”都保证不了：top-k 只取几片，漏一行数字结果就错——这不是调大 k 能救的。
Vectors can't even guarantee “completeness”: top-k takes a few chunks, and miss one row and the number is wrong — bumping k won't save you.
把订单表、财务报表、监控指标这种结构化数据硬编码成 embedding，等于让一个擅长“模糊联想”的系统去做“精确算账”，结果必然是“看起来差不多、其实算错”。正解只有一个方向：别让向量算，把精确计算交给本来就为它而生的工具——数据库与 DataFrame。
Hard-coding an orders table, a financial report or a monitoring metric into embeddings asks a system built for “fuzzy association” to do “exact accounting” — the result is bound to “look about right but compute wrong”. There's only one right direction: don't make vectors compute; hand exact calculation to the tools born for it — databases and DataFrames.

text-to-SQL：LLM 写 SQL，数据库做精确计算text-to-SQL: the LLM writes SQL, the database does the exact compute

数据库天生就会“精确算账”：GROUP BY 分组、SUM 求和、ORDER BY 排序，几十年的引擎优化保证又快又准。text-to-SQL 的思路就是分工：把表结构（schema）喂给 LLM，让它据此写出 SQL，交数据库执行，再把结果用自然语言讲回来——LLM 只做“把人话翻成 SQL”这件它擅长的事，真正的计算落在数据库上。SQLDatabase 包住数据库连接（include_tables 限定可见表），NLSQLTableQueryEngine 把“自然语言 → SQL → 执行 → 答案”串成一个查询引擎。
A database is born to do “exact accounting”: GROUP BY to group, SUM to total, ORDER BY to sort — decades of engine optimization make it fast and correct. text-to-SQL is just a division of labor: feed the schema to the LLM, have it write SQL from that, hand it to the database to execute, then phrase the result back in natural language — the LLM only does the one thing it's good at, “translate human words into SQL”, while the real computation lands on the database. SQLDatabase wraps the connection (include_tables limits the visible tables) and NLSQLTableQueryEngine chains “NL → SQL → execute → answer” into one query engine.

可 schema 一大就有新麻烦：库里有几百张表时，把所有表结构塞进提示词既超长又贵，还会让 LLM 在无关表里挑花眼、写错 SQL。解法是先选表、再写 SQL：把每张表的描述做成对象、建一个 ObjectIndex，查询时先检索出与问题最相关的少数几张表，只把这几张的 schema 交给 LLM。承担这一步的是 SQLTableRetrieverQueryEngine——它比 NLSQLTableQueryEngine 多了“先按问题召回相关表”的检索器，正是为大库设计：提示词短了，选错表、串错列的概率自然降下来。
But a big schema brings a new problem: with hundreds of tables, stuffing every schema into the prompt is huge and expensive and tempts the LLM to pick the wrong table and write wrong SQL. The fix is select tables first, then write SQL: turn each table's description into an object and build an ObjectIndex; at query time first retrieve the few tables most relevant to the question and give only those schemas to the LLM. That job falls to SQLTableRetrieverQueryEngine — it adds a “retrieve relevant tables by the question first” retriever on top of NLSQLTableQueryEngine, designed exactly for large databases: a shorter prompt naturally lowers the odds of the wrong table or a crossed-up column.

不是所有结构化数据都躺在数据库里。很多时候你手上就是一个 pandas.DataFrame——一份刚读进来的 CSV、一段 API 返回、一张算好的中间表。为这点数据起一个数据库太重，PandasQueryEngine 直接让 LLM 对 DataFrame 干活：把列名和样例给 LLM，让它写一小段 pandas 代码（如 df.groupby(...).sum()），执行后把结果讲回来。
Not all structured data lives in a database. Often you just hold a pandas.DataFrame — a freshly read CSV, an API response, a computed intermediate table. Spinning up a database for that is overkill; PandasQueryEngine lets the LLM work the DataFrame directly: give it the column names and a sample, have it write a small snippet of pandas code (e.g. df.groupby(...).sum()), execute it, and phrase the result back.
💡
verbose=True 会打印它生成的 pandas 代码——方便你核对它到底算了什么，也是上线前的必做一步。
verbose=True prints the pandas code it generated — handy for checking what it actually computed, and a must before shipping.
“销量最高的 3 个产品”这种小分析，几行就出。它和 text-to-SQL 是同一套思路（LLM 写代码、引擎执行），只是把“数据库 + SQL”换成了“内存表 + pandas”。
A small analysis like “the top-3 products by sales” comes out in a few lines. It's the same idea as text-to-SQL (the LLM writes code, an engine runs it), just swapping “database + SQL” for “in-memory table + pandas”.

这两个引擎好用，但有一个必须正视的共同前提：它们都在执行 LLM 生成的代码。PandasQueryEngine 会 eval 一段 LLM 写的 Python，NLSQLTableQueryEngine 会把 LLM 写的 SQL 真的跑在你的数据库上。
These two engines are handy, but they share one premise you must face: both execute LLM-generated code. PandasQueryEngine evals a snippet of LLM-written Python, and NLSQLTableQueryEngine actually runs the LLM's SQL on your database.
⚠️
一旦输入不可信，提示注入就能诱模型写出 DROP TABLE、DELETE、越权读表，甚至（Pandas 这边）执行任意 Python——这不是“答错了”，是“被人借你的手执行命令”。
Once input is untrusted, prompt injection can coax the model into DROP TABLE, DELETE, reading tables it shouldn't, or (on the Pandas side) running arbitrary Python — that's not “a wrong answer”, it's “someone executing commands through your hands”.
所以上线前的第一要务不是跑通，而是关进笼子：① 数据库一律用只读、最小权限的账号，能 SELECT 不能写、只看该看的表 / 视图；② Pandas 这类引擎放进沙箱 / 隔离进程 / 容器，限制可用模块与文件系统；③ 只对可信输入开放，别把这种引擎直接怼到匿名用户面前。
So the first priority before shipping isn't getting it to run — it's caging it: (1) give the database a read-only, least-privilege account — SELECT but no writes, only the tables/views it should see; (2) put Pandas-style engines in a sandbox / isolated process / container, restricting modules and filesystem; (3) expose it only to trusted input, never wire such an engine straight to anonymous users.
事实上，llama-index-core 0.14.22 已经把 PandasQueryEngine 做成会抛弃用警告的占位实现，把真正能跑的版本挪到了 llama-index-experimental（from llama_index.experimental.query_engine import PandasQueryEngine），并白纸黑字写明“会任意执行代码、请在安全环境使用”——库本身就用这种“默认不让用”的姿态提醒你：方便和危险是绑在一起的。
In fact, llama-index-core 0.14.22 already ships PandasQueryEngine as a shim that raises a deprecation warning, moving the runnable version to llama-index-experimental (from llama_index.experimental.query_engine import PandasQueryEngine) and stating in black and white that it “allows arbitrary code execution; use in a secure environment” — the library itself uses this “off by default” stance to remind you that convenience and danger come bound together.
🧩 生活类比🧩 Analogy
向量检索像凭“模糊回忆”找最像的一段话：问“上个月华东区卖了多少”，它只会捞回几段“看起来相关”的文字，真要把数字加起来还得靠你自己、还容易漏。text-to-SQL 像“拿计算器精确算”：让数据库去 GROUP BY · SUM · ORDER BY，算出来的数字又全又准。要数字，就别问“模糊回忆”，去按“计算器”。
Vector search is like fishing by “fuzzy memory” for the passage that looks most alike: ask “how much did the East region sell last month” and it pulls back a few “seems-related” snippets, leaving the actual arithmetic to you — and easy to miss rows. text-to-SQL is like “reaching for a calculator”: let the database GROUP BY · SUM · ORDER BY, and the number comes back complete and correct. When you need a number, don't ask “fuzzy memory” — press the “calculator”.
✅ 本课要点✅ Key points
  • 数字 / 聚合 / 精确筛选别塞向量——向量只会“找相似”，算不准 SUM/COUNT/排序；结构化数据交给能精确计算的引擎。Don't force numbers / aggregation / exact filtering into vectors — vectors only “find similar” and can't compute SUM/COUNT/sort; hand structured data to engines that compute exactly.
  • text-to-SQL：让 LLM 据表结构写 SQL、数据库执行（NLSQLTableQueryEngine）；表多时先用 ObjectIndex + SQLTableRetrieverQueryEngine 选相关表再生成 SQL。text-to-SQL: the LLM writes SQL from the schema and the database executes it (NLSQLTableQueryEngine); with many tables, first select relevant tables via ObjectIndex + SQLTableRetrieverQueryEngine, then generate SQL.
  • 内存 DataFrame 用 PandasQueryEngine 做快速表分析（LLM 写 pandas、引擎执行）。For an in-memory DataFrame, use PandasQueryEngine for quick table analysis (the LLM writes pandas, the engine runs it).
  • ⚠️ 安全：Pandas/SQL 引擎都在执行 LLM 生成的代码——必须沙箱 / 只读账号 / 最小权限 / 限可信输入；core 0.14.22 已把 PandasQueryEngine 移到 llama-index-experimental 并要求“安全环境”。⚠️ Safety: Pandas/SQL engines both execute LLM-generated code — you must sandbox / use a read-only account / least privilege / restrict to trusted input; core 0.14.22 has moved PandasQueryEngine to llama-index-experimental and requires a “secure environment”.
💡 设计亮点💡 Design insight
结构化数据查询的精髓，是把“该谁来算”想清楚：向量擅长“找相似”，却天生不会“精确算数字”——SUM、COUNT、GROUP BY、环比这些，硬塞进 embedding 只会得到“看起来差不多”的错答。正确做法是按数据形态选工具：非结构化文档走向量，数据库表走 text-to-SQL，内存表走 Pandas，让数据库 / 解释器去做它最擅长的精确计算，LLM 只负责“把人话翻成查询”。
The essence of structured-data querying is getting “who should do the math” right: vectors are great at “finding similar” but inherently can't “compute exact numbers” — SUM, COUNT, GROUP BY, quarter-over-quarter forced into an embedding only yield a “close-looking” wrong answer. The fix is to pick the tool by data shape: unstructured docs → vectors, database tables → text-to-SQL, in-memory tables → Pandas, letting the database / interpreter do the exact compute it's best at while the LLM only “translates human questions into a query”.
代价也很实在：这些引擎都在执行 LLM 生成的代码，把“翻译”的便利和“任意执行”的风险绑在了一起——所以工程上第一件事不是跑通，而是关进沙箱、换只读账号、收最小权限。一句话：数字别硬塞向量，但给数据库写 SQL 的权力也要锁死。
The cost is real too: these engines all execute LLM-generated code, binding the convenience of “translation” to the risk of “arbitrary execution” — so the first engineering move isn't getting it to run, it's caging it: sandbox, swap in a read-only account, cut to least privilege. In a line: don't force numbers into vectors, but lock down the power to write SQL against your database too.

看参考要点Show key points 点击展开expand
🔑 重点：大库的关键不是“让 LLM 更聪明”，而是“先把 schema 缩小”——只把相关的几张表喂给它，再用示例和约束兜底。① 先选表再写 SQL：用 ObjectIndex + SQLTableRetrieverQueryEngine，查询时先按问题检索出最相关的少数表，把上百张表的 schema 压到几张，提示词短了、选错率自然降；② 给好表/列描述：为每张表、关键列写清用途，LLM 才分得清 amount 和 net_amount；③ few-shot 示例：放几条“问题→正确 SQL”的范例，教它本库的 join 习惯与命名；④ 限定可用面：用只读视图把复杂表预先 join、改成业务友好的字段，缩小它能碰的范围；⑤ 评估闭环：建一组“问题→金标结果”，用执行结果是否一致量准确率，针对错例迭代描述与示例。一句话：缩小 schema，比换更大的模型更有效。🔑 Key: the lever for a big database isn't “make the LLM smarter”, it's “shrink the schema first” — feed it only the few relevant tables, then backstop with examples and constraints. (1) Select tables before writing SQL: with ObjectIndex + SQLTableRetrieverQueryEngine, first retrieve the few most relevant tables per question, collapsing hundreds of schemas to a handful — a shorter prompt means fewer wrong picks; (2) good table/column descriptions: document each table and key column so the LLM can tell amount from net_amount; (3) few-shot examples: include a few “question → correct SQL” pairs to teach this DB's join habits and naming; (4) restrict the surface: use read-only views that pre-join complex tables into business-friendly fields, shrinking what it can touch; (5) an eval loop: build “question → gold result” pairs and measure accuracy by whether execution results match, iterating descriptions and examples on the failures. In a line: shrinking the schema beats reaching for a bigger model.

🔑 重点：用 RouterQueryEngine 按“问题类型”分流——找文字走向量、算数字走 SQL；一个问题同时要两类时再拆成子问题各查各的、最后合并（呼应 L18 的路由、L30 的查询分解）。① 路由：给 Router 挂两个工具——“文档向量引擎”和“text-to-SQL 引擎”，各写清能力描述（“擅长政策/条款类语义问答” vs “擅长订单/指标的精确统计”），由 LLM 选择器按问题选；② 拆分：用 SubQuestionQueryEngine 把复合问题拆成“政策是什么”（走向量）和“退货率多少”（走 SQL）两个子问题，各取所长后合并作答；③ 判据：路由不靠猜——给每个引擎清晰描述，再用一批带标注的问题评测路由命中率，错分多就改描述或加示例。一句话：别让一个引擎硬扛两类活，按数据形态分流。🔑 Key: use RouterQueryEngine to split by “question type” — text lookups to vectors, number-crunching to SQL — and when one question needs both, decompose it into sub-questions answered separately, then merge (echoing L18's routing and L30's query decomposition). (1) Route: give the Router two tools — a “document vector engine” and a “text-to-SQL engine” — each with a crisp capability description (“good at policy/clause semantic Q&A” vs “good at exact stats over orders/metrics”), and let the LLM selector pick per question; (2) decompose: use SubQuestionQueryEngine to split the compound question into “what is the policy” (→ vectors) and “what's the return rate” (→ SQL), then combine; (3) judge it: don't route by guesswork — give each engine a clear description and measure routing hit-rate on a labeled question set, fixing descriptions or adding examples when misrouted. In a line: don't make one engine carry both jobs — split by data shape.

🔑 重点：text-to-SQL 在真的执行 LLM 生成的 SQL，等于把“对你数据库下命令”的能力间接交给了提问者——提示注入可诱导出 DROP/DELETE/越权读表。核心防线是“最小权限 + 隔离 + 限面 + 校验”，绝不靠“在提示词里求它别乱来”。① 只读、最小权限账号：只 SELECT、不能写、只授权该看的表/视图，从根上堵掉破坏与越权；② 限定可见面：include_tables / 业务视图把暴露的表/列收到最小，敏感列根本不进 schema；③ 沙箱 / 超时 / 行数上限：查询跑在隔离环境，加语句超时、LIMIT 和结果行数上限，防爆库；④ 执行前校验：对生成的 SQL 做白名单校验（只允许 SELECT、只允许这些表），可疑就拒；⑤ Pandas 更甚：PandasQueryEngine 会 eval Python、风险更大，core 0.14.22 已把它移到 llama-index-experimental 并标注“仅安全环境”，对外场景能不用就不用。🔑 Key: text-to-SQL actually executes LLM-generated SQL, so you've indirectly handed “issue commands to your database” to the asker — prompt injection can elicit DROP/DELETE/unauthorized reads. The real defense is “least privilege + isolation + a small surface + validation”, never “asking it nicely in the prompt”. (1) read-only, least-privilege account: SELECT only, no writes, granted just the tables/views it should see — closing destruction and over-reach at the root; (2) shrink the visible surface: include_tables / business views expose the minimum tables/columns; sensitive columns never enter the schema; (3) sandbox / timeouts / row caps: run queries in isolation with statement timeouts, LIMIT and result-row caps to prevent blowups; (4) validate before executing: allow-list the generated SQL (SELECT only, these tables only) and reject anything suspicious; (5) Pandas is worse: PandasQueryEngine evals Python and is riskier — core 0.14.22 has moved it to llama-index-experimental marked “secure environments only”, so avoid it for external-facing use.
