前面 26 课的检索都在做同一件事：把问题和文档块都变成向量，取回最相似的 top-k 片段。可“相似”召不回“连起来的事实”——当答案要顺着“A 的供应商 → 供应商的总部 → 总部所在国”这样的多跳关系走时，没有任何单个片段同时写着这三件事，纯向量自然答不出。图谱 RAG 换一种存法：用 LLM 把文档抽成实体—关系—实体的三元组、连成一张图，查询时从问题里的实体出发顺着边一跳跳走。一句话：向量问“像不像”，图谱问“连不连”。The first 26 lessons' retrieval all did the same thing: turn the question and the doc chunks into vectors and fetch the top-k most similar chunks. But “similar” can't recall “connected facts” — when an answer must follow a multi-hop chain like “A's supplier → the supplier's HQ → the HQ's country”, no single chunk states all three at once, so pure vectors simply can't answer. Graph RAG stores things differently: an LLM extracts documents into entity—relation—entity triples wired into a graph, and at query time starts from the entities in the question and walks edge by edge, hop by hop. In a line: vectors ask “similar?”, a graph asks “connected?”.

为什么要图谱：向量召回“相似”，图谱召回“连起来”Why a graph: vectors recall the “similar”, a graph recalls the “connected”

纯向量检索把每个文档块独立编码成一个语义向量，召回的是和问题最像的片段。这对“换种问法、近义改写”很在行，但它有个结构性盲区：跨片段的关系会丢。问“X-2000 的供应商总部在哪个国家”，答案要顺着“X-2000 → 供应商 → 总部 → 国家”连走三跳，而这三段事实往往分散在不同文档里、没有任何一个片段同时写全。
Pure vector search encodes each doc chunk independently into one semantic vector and recalls the chunks most similar to the question. That's great for “rephrasings and synonyms”, but it has a structural blind spot: cross-chunk relations are lost. Ask “which country is the HQ of X-2000's supplier in” and the answer must follow “X-2000 → supplier → HQ → country” across three hops, yet those three facts usually sit in different documents with no single chunk stating them all.
🔑
这盲区是结构性的：无论 top-k 取多大，纯向量都召不回那条把多跳事实串起来的链。
The blind spot is structural: however large top-k is, pure vectors never recall the chain that strings the multi-hop facts together.
图谱把事实存成实体—关系—实体的三元组（如 X-2000 —替代→ X-1000），查询时顺着边一跳跳遍历，正好把“连起来的事实”找回来。这就是图谱补的那块：不是“更像”，而是“能连”。
A graph stores facts as entity—relation—entity triples (e.g. X-2000 —replaces→ X-1000) and at query time traverses edge by edge, hop by hop, recovering exactly those “connected facts”. That's the gap a graph fills: not “more similar”, but “can connect”.

怎么建：PropertyGraphIndex 用 LLM 抽三元组How to build: PropertyGraphIndex extracts triples with an LLM

建图的核心是抽取：把自然语言里的事实变成结构化的实体—关系—实体三元组。PropertyGraphIndex.from_documents(...) 接受一组 kg_extractors（知识图谱抽取器），逐个文档跑 LLM，把“X-2000 兼容配件 B”这样的句子抽成 (X-2000, 兼容, 配件 B) 的边。
The heart of building a graph is extraction: turning facts in natural language into structured entity—relation—entity triples. PropertyGraphIndex.from_documents(...) takes a list of kg_extractors (knowledge-graph extractors) and runs an LLM over each document, turning a sentence like “X-2000 is compatible with part B” into a (X-2000, compatible-with, part B) edge.
其中 SchemaLLMPathExtractor 是最常用的一种：你预先声明允许的实体类型（产品 / 配件 / 厂商）和关系类型（兼容 / 替代 / 属于），抽取被约束在 schema 内——LLM 不会乱造关系，图更干净、更可控。
The most common one is SchemaLLMPathExtractor: you declare in advance the allowed entity types (product / part / vendor) and relation types (compatible / replaces / belongs-to), and extraction is constrained to that schema — the LLM won't invent stray relations, so the graph is cleaner and more controllable.
⚠️
代价：抽取要逐文档跑 LLM，建库比纯向量更贵更慢——但这是一次性成本，换来的是可多跳的结构。
The cost: extraction runs an LLM per document, making indexing pricier and slower than plain vectors — but it's a one-time cost that buys a multi-hop structure.
用 PropertyGraphIndex 抽三元组建图，再 include_text 查询：既给关系骨架又附原文证据Build a triple graph with PropertyGraphIndex, then query with include_text: relation skeleton plus source evidence
from llama_index.core import PropertyGraphIndex
from llama_index.core.indices.property_graph import SchemaLLMPathExtractor

index = PropertyGraphIndex.from_documents(
    documents,
    kg_extractors=[SchemaLLMPathExtractor(llm=llm)],   # 可传 possible_entities / possible_relations 声明 schema
)
qe = index.as_query_engine(include_text=True)
print(qe.query("X-2000 和它的替代型号有什么兼容差异？"))

怎么查：同义词扩展找种子 + 向量上下文，再顺边多跳How to query: synonym seeds + vector context, then multi-hop

查询分两步：先找种子节点，再顺边遍历。as_query_engine() 默认挂两路子检索器：LLMSynonymRetriever（同义词扩展）把问题里的词扩成若干近义说法，去匹配图里的实体名，解决“用户叫‘旧款’、图里存的是‘X-1000’”这种对不上；VectorContextRetriever（向量上下文）则用 embedding 给问题找语义最近的种子节点，并捎回节点周边的原文。
Querying is two steps: find seed nodes, then traverse edges. as_query_engine() attaches two default sub-retrievers: LLMSynonymRetriever (synonym expansion) expands the question's words into several near-synonyms to match entity names in the graph, fixing the “user says ‘the old model’ but the graph stores ‘X-1000’” mismatch; VectorContextRetriever (vector context) uses embeddings to find the semantically nearest seed nodes and brings back the text around them.
📌
两路子检索器只负责“找种子”：真正的多跳发生在拿到种子之后的顺边遍历——别把“向量上下文”误当成最终检索。
The two sub-retrievers only “find seeds”: the real multi-hop happens after the seeds are found, walking the edges — don't mistake “vector context” for the final retrieval.
拿到种子实体后，检索器顺着边多跳，把连起来的三元组收集成上下文。include_text=True 还会把每个命中节点对应的原始段落一并带上——这样 LLM 既看到关系骨架（谁连着谁），又有原文证据，答案能溯源、不靠编。
With seed entities in hand, the retriever walks the edges hop by hop, gathering the connected triples into context. include_text=True also attaches each hit node's original passage — so the LLM sees both the relation skeleton (who connects to whom) and the source evidence, keeping answers traceable rather than made up.

何时用图、何时用向量When a graph, when vectors

不是所有 RAG 都该上图谱，判断只看一件事：问题靠不靠“关系”。当知识本身是一张关系网（产品-配件-兼容、人物-公司-职位、论文-引用），问题又要多跳推理（A 的 B 的 C），图谱的“顺边遍历”几乎是唯一能答全的方式。反过来，如果问题只是找一段语义相近的话（“退款政策怎么写的”），或关系稀疏、一跳就够，那纯向量更便宜、更快、够用，花 LLM 抽一整张图并不划算。实务里两者常互补：向量管“找到相关段落”，图谱管“把相关事实连起来”，按问题类型路由（回想 L18 的 Router）。
Not every RAG needs a graph; the test is one thing: does the question hinge on “relations”. When the knowledge is itself a web of relations (product-part-compatibility, person-company-role, paper-citation) and the question needs multi-hop reasoning (A's B's C), a graph's edge traversal is almost the only way to answer it fully. Conversely, if the question just finds a semantically close passage (“what does the refund policy say”) or relations are sparse and one hop suffices, plain vectors are cheaper, faster, and enough — paying an LLM to extract a whole graph isn't worth it. In practice the two often complement: vectors “find the relevant passage”, a graph “connects the relevant facts”, routed by question type (recall L18's Router).

🧩 生活类比🧩 Analogy
纯向量检索像在一摞书里翻找长得像的书页：哪一页和你手里的纸看起来最像就抽出来——换种说法也能找到，但它不知道页与页之间的关系。图谱像顺着引用链一页页翻下去：从这一页的脚注找到它引用的那一页，再顺着那一页的引用接着翻，把一条线索串起来的内容都找齐——这正是多跳问题需要的。
Pure vector search is like thumbing through a stack for the page that looks alike: pull out whichever page looks most like the sheet in your hand — rephrasings still match, but it knows nothing about how pages relate. A graph is like following a citation chain page by page: from this page's footnote you reach the page it cites, then follow that page's citations onward, collecting everything strung along one thread — exactly what a multi-hop question needs.

💡 设计亮点💡 Design insight
图谱 RAG 的精髓，是把检索的问题从“哪段话最像”换成“哪些事实连得上”。向量把文档压成一堆互不相连的点，只能按距离取近邻；图谱先用 LLM 把文档还原成实体与关系的网，于是“A 的供应商的总部在哪国”这种跨多跳的问题，能顺着边一步步走到答案——这是纯相似度结构上做不到的。代价也很实在：抽取要烧 LLM、建库更慢，图的质量取决于抽取的质量。所以工程判断不是“图谱更高级所以要上”，而是“我的问题到底靠不靠多跳关系”：靠，就为那张图付抽取的钱；不靠，向量又快又省，别为用不上的关系买单。
The essence of Graph RAG is turning retrieval's question from “which passage looks most alike” into “which facts can be connected”. Vectors squash documents into a cloud of unconnected points and can only take nearest neighbors by distance; a graph first has an LLM reconstruct documents into a web of entities and relations, so a multi-hop question like “which country is the HQ of A's supplier in” can be walked edge by edge to the answer — something pure similarity structurally cannot do. The cost is real: extraction burns LLM calls and indexing is slower, and graph quality rides on extraction quality. So the engineering call isn't “a graph is fancier, so use it” but “does my question actually hinge on multi-hop relations”: if it does, pay the extraction price for that graph; if it doesn't, vectors are fast and cheap — don't pay for relations you'll never traverse.

🔑 重点：图的质量 = 抽取的质量——抽错 / 抽漏会造出错误的边，多跳时一路放大成错答；靠 ① schema 约束、② 评估抽取准确率、③ 人工校验高价值实体 三道闸控住。① 用 SchemaLLMPathExtractor 上 schema：预先声明允许的实体 / 关系类型，把 LLM 关进栏里，不让它乱造关系（“幻觉边”）；② 评估抽取准确率：抽样三元组和人工标注的金标比 precision / recall，差就换更强的抽取 LLM 或细化 schema；③ 人工校验高价值实体：被很多跳路过的枢纽实体（核心产品、关键厂商）一旦错，下游全错，优先人工过一遍；④ 端到端验证：用一组多跳问答金标，看修抽取前后答案对不对——别只盯抽取本身，要落到下游答案质量。🔑 Key: graph quality = extraction quality — wrong/missing extractions create bad edges that multi-hop then amplifies into wrong answers; control it with three gates: (1) schema constraints, (2) measuring extraction accuracy, (3) human-verifying high-value entities. (1) Add a schema via SchemaLLMPathExtractor: declare the allowed entity/relation types up front to cage the LLM so it can't invent stray relations (“hallucinated edges”); (2) measure extraction accuracy: sample triples against a human-labeled gold set for precision/recall, and if low, swap in a stronger extraction LLM or refine the schema; (3) human-verify high-value entities: hub entities that many hops pass through (core products, key vendors) corrupt everything downstream if wrong, so review them first; (4) end-to-end check: use a set of multi-hop QA gold questions and compare answer correctness before/after — don't just stare at extraction, tie it back to downstream answer quality.

看参考要点Show key points 点击展开expand
🔑 重点：关系稀疏、一跳就够、或构建 / 维护成本扛不住时，纯向量更划算——别为用不上的关系付抽取的钱。① 关系稀疏：语料就是一篇篇独立文档、没什么实体间链接，建图抽不出几条有用的边；② 一跳就够：问题大多是“找一段相关的话 / 查单个事实”，向量的相似召回已经够好；③ 成本扛不住：图要 LLM 逐文档抽三元组，建库慢且贵，语料还频繁更新就要反复重抽，延迟 / 预算紧时不划算。怎么用数据：统计真实查询里真正多跳 / 关系型的占比——若很低，就只把这一小撮路由到图谱（混合），其余留给向量；再对比“向量 vs 图谱”在这批问题上的命中率 / 答案质量增量，是否抵得过多出的成本与延迟。🔑 Key: when relations are sparse, one hop suffices, or build/maintenance cost is too high, plain vectors win — don't pay extraction cost for relations you'll never traverse. (1) Sparse relations: the corpus is independent documents with few entity links, so building a graph extracts few useful edges; (2) one hop is enough: most questions are “find a relevant passage / look up a single fact”, where vector similarity already does well; (3) cost is too high: a graph needs an LLM to extract triples per document — slow and pricey to build, and a frequently-changing corpus forces repeated re-extraction; not worth it under tight latency/budget. How to use data: measure the share of real queries that are genuinely multi-hop / relational — if it's low, route only that minority to a graph (hybrid) and keep the rest on vectors; then compare “vector vs graph” hit-rate / answer-quality lift on that subset against the extra cost and latency.
🎯 3. 为什么用新的 PropertyGraphIndex 而不是旧的 KnowledgeGraphIndex？在关系表达 / 构建成本 / 可解释性上，它和纯向量又各自怎么取舍？Why use the newer PropertyGraphIndex instead of the legacy KnowledgeGraphIndex? And how do the two trade off against pure vectors on relation modeling / build cost / explainability?
看参考要点Show key points 点击展开expand
🔑 重点：PropertyGraphIndex 是新一代图谱抽象，实体 / 关系能带属性与类型、可挂多路子检索器、可上 schema 约束，表达力和可控性都强过只存裸三元组的旧 KnowledgeGraphIndex；和纯向量比，图用“可解释的多跳路径”换“更高的构建成本”。选型沿一条轴：关系越重、越需要可解释，越往图谱走。PropertyGraph 表达最丰富、可解释性最好（能看到走了哪条边），但抽取要烧 LLM；旧 KnowledgeGraph 只存简单三元组、工具较旧，新项目一般直接上 PropertyGraph；纯向量不建模关系、最便宜最快，但答案只有一个相似分、近乎黑箱。一句话：按“问题靠不靠关系”和“要不要可解释路径”选档。🔑 Key: PropertyGraphIndex is the newer graph abstraction whose entities/relations carry properties and types, plugs in multiple sub-retrievers, and supports schema constraints, so it's more expressive and controllable than the legacy KnowledgeGraphIndex that stores bare triples; versus pure vectors, a graph trades “higher build cost” for an “explainable multi-hop path”. The choice runs along one axis: the more relations matter and the more you need explainability, the more you lean toward a graph. PropertyGraph is the most expressive and explainable (you can see which edges were walked) but its extraction burns LLM calls; the legacy KnowledgeGraph stores only simple triples with older tooling, so new projects usually go straight to PropertyGraph; pure vectors model no relations and are cheapest/fastest but give only a similarity score, nearly a black box. In a line: pick the tier by “does the question hinge on relations” and “do you need an explainable path”.
