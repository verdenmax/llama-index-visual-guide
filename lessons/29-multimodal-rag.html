前面所有检索都是“文字找文字”：把问题和文本块塞进同一个向量空间，取回最相似的片段。可现实里大量知识长在图里——架构图、仪表盘截图、产品照、电路图。只把它们 OCR / 配文字成纯文本，视觉细节就丢了（“图里设备有几个接口”光靠文字答不出）。多模态 RAG 换一种存法：用多模态 embedding 把图和文映进同一个向量空间，于是能“用文字查图、用图查图”，再交给看得见图的多模态 LLM 作答。一句话：先让图文“坐标统一”，再让会看图的模型来回答。Every retrieval so far has been “text finds text”: push the question and text chunks into one vector space and fetch the most similar. But in the real world a lot of knowledge lives in images — architecture diagrams, dashboard screenshots, product photos, schematics. OCR-ing or captioning them into plain text throws away the visual detail (“how many ports does the device in the picture have” can't be answered from text alone). Multimodal RAG stores things differently: a multimodal embedding maps images and text into the same vector space, so you can “query images with text, or with images”, then hand them to a vision-capable multimodal LLM to answer. In a line: unify the coordinates of image and text first, then let a model that can see answer.

图文进同一向量空间：于是能“用文字查图、用图查图”Image and text in one vector space: now you can “query images with text, or with images”

纯文本 RAG 能成立，靠的是一个前提：问题和文档块被同一个 embedding 模型映进同一个向量空间，所以“算两段文字的余弦相似度、取最近邻”才有意义。多模态的关键，是把这个前提扩展到图像：用图文对训练出来的多模态 embedding（最有名的是 CLIP），会让“一张猫的照片”和“cat 这个词”落在彼此邻近的坐标上——图和文共用一套坐标系。
Text-only RAG works because of one premise: the question and the doc chunks are mapped by the same embedding model into the same vector space, so “cosine similarity between two pieces of text, take the nearest” actually means something. The crux of going multimodal is extending that premise to images: a multimodal embedding trained on image–text pairs (the famous one is CLIP) makes “a photo of a cat” and “the word cat” land at neighboring coordinates — image and text share one coordinate system.
🔑
“同一向量空间”不是口号，是跨模态检索能否成立的根：图文各在两个空间比距离，就像拿经纬度比体温——毫无意义。
“The same vector space” isn't a slogan; it's the root of whether cross-modal retrieval can exist at all: measure distance across two separate spaces and it's like comparing latitude to body temperature — meaningless.
一旦共用，跨模态的最近邻就有了意义：拿“文字问题”的向量，去和图像向量比距离，就能召回“画的就是这件事”的图片——这正是“用文字查图”的全部底气。
Once shared, cross-modal nearest-neighbor becomes meaningful: take the vector of a text question, measure its distance to image vectors, and you recall the picture that “depicts exactly this” — which is the entire basis of “querying images with text”.

建多模态索引：文本库、图像库各一，统一检索Build a multimodal index: a text store and an image store, retrieved together

把图文放进同一空间后，索引也要相应地分两套向量库。MultiModalVectorStoreIndex 内部维护两个 store：一个存文本节点（段落，走文本 embedding），一个存图像节点（ImageNode，走图像 embedding）。SimpleDirectoryReader 读一个混合文件夹时，文本读成普通 Document、图片读成 ImageDocument，from_documents 自动按类型分流进各自的 store。
Once image and text share a space, the index splits into two vector stores to match. MultiModalVectorStoreIndex keeps two stores internally: one for text nodes (passages, via the text embedding) and one for image nodes (ImageNode, via the image embedding). When SimpleDirectoryReader reads a mixed folder, text becomes ordinary Documents and pictures become ImageDocuments, and from_documents automatically routes each type into its own store.
查询时两个 store 都召回，再把图、文候选合并交给下游——所以一次检索既可能捞回相关段落，也可能捞回相关图片。为什么要分两套、而不是混成一堆？因为两种模态的向量来自不同的 embedder，分库便于各自索引、各自调 top-k，最后在统一的检索结果里汇合。
At query time both stores are queried and the image + text candidates are merged for the downstream step — so one retrieval may pull back relevant passages and relevant pictures alike. Why two stores rather than one mixed pile? Because the two modalities' vectors come from different embedders; separate stores let each be indexed and top-k-tuned independently, then meet in one unified result set.

检索件 + 生成件，都要换成多模态版Both the retrieval side and the generation side must go multimodal

把一套纯文本 RAG 升级成多模态，要同时换两个件，缺一不可。① 检索件：embedding 必须是多模态的，图片才能被编码进那个共享空间、被文字查到——这一步落在图像 embedding（如 CLIP）和索引里的图像 store 上。② 生成件：作答的 LLM 必须看得见图——检索召回的是图片节点，得交给一个多模态 LLM（视觉模型），它把图像当输入直接“看”着回答。
Upgrading a text-only RAG to multimodal means swapping two components at once, neither optional. (1) Retrieval side: the embedding must be multimodal so images get encoded into that shared space and become findable by text — this rides on the image embedding (e.g. CLIP) and the index's image store. (2) Generation side: the answering LLM must be able to see the image — retrieval returns image nodes, which must go to a multimodal LLM (a vision model) that takes the image as input and actually “looks” to answer.
⚠️
两个件缺一不可：换了多模态 embedding 却仍用纯文本 LLM，召回的图它根本读不了——多模态就白做了。
Neither component is optional: swap in a multimodal embedding but keep a text-only LLM and it simply can't read the recalled images — the whole effort is wasted.
还要分清哪是 core、哪是集成：抽象层是 core 的——MultiModalLLM 基类、MultiModalVectorStoreIndex、ImageNode 都在核心里；但具体能跑的模型——一个会看图的多模态 LLM（如 GPT-4o / Gemini）、CLIP 图像 embedding——都在 core 之外的集成包，要按需另装。core 给“插槽”，模型是“插件”。
Keep what is core vs an integration straight: the abstraction layer is core — the MultiModalLLM base class, MultiModalVectorStoreIndex and ImageNode all live in core; but the concrete runnable models — a vision-capable multimodal LLM (e.g. GPT-4o / Gemini), the CLIP image embedding — sit in integration packages outside core and are installed as needed. Core gives the “sockets”, the models are the “plugins”.

什么时候值得上多模态、什么时候不必When multimodal is worth it, and when it isn't

判断只看一件事：答案到底在不在“像素里”。值得上的典型场景，是信息只长在图里且文字旁注补不全的：读图表 / 仪表盘截图取趋势与数值、看产品照认型号与接口、查电路图 / 架构图 / 扫描件里的连接与标注——这些问题“看图才能答”，纯文本 RAG 结构上答不了。反过来，如果图只是装饰、或它的信息本就写在正文 / 标题 / 配文里，那纯文本检索又快又便宜，没必要上多模态。
The test is one thing: does the answer actually live “in the pixels”. The cases worth it are where information exists only in the image and no text caption fully covers it: reading charts / dashboard screenshots for trends and values, reading a product photo for a model number and ports, checking a schematic / architecture diagram / scan for connections and labels — these need you to “look at the picture”, and text-only RAG structurally can't answer them. Conversely, if the image is merely decorative, or its information is already written in the body / heading / caption, then text-only retrieval is faster and cheaper and multimodal isn't needed.
成本也要算清：多模态 embedding 和视觉 LLM 更贵更慢、图像 store 也更占资源。还有一种降级方案——先把图 OCR / 配成 caption 文字、再做纯文本 RAG：便宜，但丢视觉细节（布局、颜色、相对位置、图里没写成字的东西）。
Count the cost too: multimodal embeddings and vision LLMs are pricier and slower, and an image store consumes more resources. There's also a downgrade path — OCR/caption the image into text first, then do text-only RAG: cheap, but it loses visual detail (layout, color, relative position, anything not written out in the picture).
💡
稳妥姿态：默认文本优先、图像按需启用——只对“确有图、且要看图”的查询和文档走多模态。
Safe stance: text-first by default, images enabled on demand — route only the queries and documents that “truly have an image and need to look at it” through the multimodal path.
🧩 生活类比🧩 Analogy
想象给每张照片和每个词都标上同一张地图上的坐标：一旦“猫的照片”和“猫”这个词被放到地图上同一个点，你就能站在词上、找最近的照片——这就是“用文字查图”。要是照片用的是一张地图、文字用的是另一张互不对齐的地图，那“谁离谁近”就无从谈起。多模态 embedding 干的就是“把图和字标到同一张地图上”这件事；剩下的检索，不过是在同一张图上找最近的点。
Picture giving every photo and every word coordinates on the same map: once “a photo of a cat” and the word “cat” are placed at the same spot, you can stand on the word and find the nearest photo — that's “querying images with text”. If photos used one map and words used another, misaligned map, then “who is near whom” is meaningless. A multimodal embedding does exactly the “plot images and words on one shared map” part; the retrieval that follows is just finding the nearest point on that one map.
✅ 本课要点✅ Key points
  • 多模态 RAG 的根：用多模态 embedding（如 CLIP）把图和文映进同一向量空间，跨模态最近邻才有意义，于是能“用文字查图、用图查图”。The root of multimodal RAG: a multimodal embedding (e.g. CLIP) maps images and text into the same vector space, making cross-modal nearest-neighbor meaningful — so you can “query images with text, or with images”.
  • MultiModalVectorStoreIndex 内部文本 store + 图像 store 各一，from_documents 按类型分流（文本 → Document、图片 → ImageDocument），查询时两库都召回再合并。MultiModalVectorStoreIndex keeps a text store and an image store; from_documents routes by type (text → Document, image → ImageDocument), and a query recalls from both stores and merges.
  • 升级要同时换两个件：检索件换多模态 embedding，生成件换看得见图的多模态 LLM；纯文本 LLM 读不了召回的图。Upgrading swaps two components at once: a multimodal embedding on the retrieval side and a vision-capable multimodal LLM on the generation side; a text-only LLM can't read the recalled images.
  • core 给抽象、集成给模型：MultiModalVectorStoreIndex / MultiModalLLM / ImageNode 在 core；会看图的多模态 LLM（如 GPT-4o / Gemini）、CLIP embedding 在集成包。按需上多模态——答案在像素里才值得，否则文本优先更省。Core ships abstractions, integrations ship models: MultiModalVectorStoreIndex / MultiModalLLM / ImageNode are in core; a vision-capable multimodal LLM (e.g. GPT-4o / Gemini) and the CLIP embedding are in integration packages. Go multimodal on demand — worth it only when the answer is in the pixels, otherwise text-first is cheaper.
💡 设计亮点💡 Design insight
多模态 RAG 的精髓，是把“找相似”从纯文本扩展到图文之间，而这一切的支点只有一个：图和文落在同一个向量空间。有了这层跨模态对齐，“用文字查图”才从不可能变成一次普通的最近邻；没有它，图像向量和文本向量就是两套不能互比的坐标。
The essence of multimodal RAG is extending “find similar” from pure text to between images and text, and the whole thing pivots on one fulcrum: images and text landing in the same vector space. With that cross-modal alignment, “querying images with text” turns from impossible into an ordinary nearest-neighbor; without it, image vectors and text vectors are two coordinate systems you can't compare.
工程上要记住三件事：① 两个件一起换——检索端的多模态 embedding 让图进得了空间，生成端的视觉 LLM 让模型看得懂召回的图，缺一边都不成；② 分清 core 与集成——core 只给 MultiModalVectorStoreIndex / MultiModalLLM 这层抽象插槽，真正能跑的视觉模型和 CLIP embedding 是外部集成，别把它们当核心内置；③ 按需启用——多模态更贵更慢，只有当“答案确实在像素里”时才值得，否则文本优先、必要时再用 caption 降级。一句话：先让图文坐标统一，再让会看图的模型来回答。
Three engineering points: (1) swap both components together — the retrieval-side multimodal embedding lets images enter the space, the generation-side vision LLM lets the model understand the recalled images, and neither alone suffices; (2) keep core vs integrations straight — core ships only the abstraction sockets like MultiModalVectorStoreIndex / MultiModalLLM, while the actual runnable vision models and CLIP embedding are external integrations, not core built-ins; (3) enable on demand — multimodal is pricier and slower, worth it only when “the answer is truly in the pixels”, otherwise go text-first and fall back to captioning when needed. In a line: unify the coordinates of image and text first, then let a model that can see answer.

(1) Route by document: at ingest, separate “image-bearing” docs (charts, screenshots, product photos) from “text-only” ones, and build an image store / run multimodal embeddings only for the former; (2) route by query: use routing (recall L18) to decide whether a question needs to look at an image — if so, recall pictures and call the multimodal LLM, otherwise take the faster, cheaper text-only path; (3) downgrade fallback: in many cases OCR/captioning images into text and doing text RAG is enough, upgrading to true multimodal only when captioning drops key visual detail; (4) set the threshold with numbers: measure the share of queries that genuinely need the image, and compare multimodal vs text answer-accuracy lift on that subset against the extra latency and cost. In a line: multimodal is an on-demand upgrade, not on by default.

🔑 重点：金标问答里必须包含一批“靠图才能答”的题——答案只在图里、文字旁注补不全，否则多模态和纯文本会打平，测不出它的价值（呼应 L19 评估、L22 回归集）。① 题目要对路：除常规问答外，专门设计“读图表取数值”“认产品图型号”“看架构图连接”这类必须看图的题，纯文本基线在这批题上应当明显答不好；② 分轨看指标：检索侧看“相关图片有没有被召回”（图像命中率），生成侧看视觉 LLM 读图后的 Faithfulness / 正确率，别把两者混成一个分；③ 和文本基线对照：同一批题跑“纯文本 RAG” vs “多模态 RAG”，用答对率的增量证明多模态值这个钱；④ 接回归闸：把这批“看图题”固定成回归集，挡住“换了 embedding / 模型把读图能力悄悄弄坏”的退化。一句话：测不出差异，往往是题目里根本没有需要看图的题。🔑 Key: the gold Q&A set must include a batch of “only answerable from the image” questions — the answer lives only in the picture and no caption covers it — otherwise multimodal and text-only tie and you can't measure the value (echoing L19's evaluation, L22's regression set). (1) Questions on point: beyond ordinary Q&A, deliberately design “read a value off a chart”, “identify a product model from a photo”, “trace connections in an architecture diagram” items that require looking at the image, on which a text-only baseline should clearly do poorly; (2) metrics by track: on the retrieval side check “were the relevant images recalled” (image hit-rate), on the generation side check the vision LLM's Faithfulness / correctness after reading the image — don't blend them into one score; (3) compare against a text baseline: run “text-only RAG” vs “multimodal RAG” on the same set and use the accuracy lift to prove multimodal earns its cost; (4) wire a regression gate: freeze those “look-at-the-image” questions into a regression set to block “swapping the embedding / model quietly broke image reading” regressions. In a line: if you can't measure a difference, usually the set has no questions that actually need the image.

🔑 重点：caption / OCR 降级法便宜又好维护，能覆盖“信息一句话能说清”的图；但把图压成文字就丢了视觉细节，所以选型取决于你的图是“配文字就够”还是“非看原图不可”。① caption 够用的场景：图里的信息本就能被一段描述覆盖（写着规格的表格、一句话能说清的示意图），转成文字后纯文本 RAG 又快又省，还能复用现成的文本检索栈；② 必须真多模态的场景：答案藏在布局 / 颜色 / 相对位置 / 没写成字的细节里（电路怎么连、仪表盘哪条线在涨、产品照上有几个接口），caption 几乎必然漏，得让图进共享向量空间、再交视觉 LLM 直接看；③ 成本对照：caption 省掉了图像 embedding 和视觉 LLM 的开销，但 caption 本身要花一次模型调用且质量参差；真多模态更贵更慢但信息无损。一句话：caption 是有损压缩，能不能用先问“丢掉的那部分要不要紧”。🔑 Key: the caption/OCR downgrade is cheap and easy to maintain and covers images whose “information fits in a sentence”; but squashing an image into text drops visual detail, so the choice depends on whether your images are “fine with a caption” or “must be seen in the original”. (1) When captioning suffices: the image's information is already coverable by a description (a spec table, a one-sentence schematic), so converting to text and doing text RAG is fast and cheap and reuses your existing text retrieval stack; (2) when true multimodal is required: the answer hides in layout / color / relative position / details never written as words (how a circuit connects, which line on a dashboard is rising, how many ports are on a product photo), where captioning almost certainly misses, so the image must enter the shared vector space and go to a vision LLM to be seen directly; (3) cost contrast: captioning saves the image-embedding and vision-LLM cost, but the caption itself takes a model call of uneven quality; true multimodal is pricier and slower but lossless. In a line: captioning is lossy compression — first ask “does the part it drops matter”.
