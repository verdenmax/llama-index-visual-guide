全局配置 Settings 与 PromptSettings & Prompts

Settings 是全局默认配置（llm / embed_model / node_parser / chunk_size…），取代了旧的 ServiceContext。Prompt 模板则控制“到底怎么问 LLM”，可按领域定制语气与约束。Settings holds global defaults (llm / embed_model / node_parser / chunk_size…), replacing the old ServiceContext. Prompt templates control “how exactly we ask the LLM”, customizable per domain for tone and constraints.

🧩 生活类比🧩 Analogy
Settings 像项目的默认设置面板：定一次，处处生效，单处仍可覆盖。Prompt 模板像填空式提问公式，把检索到的上下文和用户问题填进固定结构。
Settings is the project's defaults panel: set once, applies everywhere, still overridable locally. A prompt template is a fill-in-the-blank question form that slots the retrieved context and the user's question into a fixed structure.

全局默认 + 局部覆盖；Prompt 决定“最后一公里”Global defaults + local overrides; the prompt owns the “last mile”

Settings 给全管道一份默认值，省去到处传配置对象；但任何一处都能在构造时就地覆盖，不影响全局。检索决定“拿到哪些上下文”，Prompt 决定“拿同一份上下文怎么问”——检索结果一字不变，换一个语气、约束或输出格式的模板，答案就大不相同。这一步正是 RAG 的“最后一公里”。
Settings hands the whole pipeline a set of defaults, sparing you from threading a config object everywhere; yet any call site can override locally at construction without touching the global. Retrieval decides “which context you get”; the prompt decides “how to ask with that same context” — with identical retrieved snippets, swapping the template's tone, constraints or output format yields a very different answer. That step is RAG's “last mile”.
全局 Settings（基线）Global Settings (baseline)
一处设定，llm / embed_model / chunk_size 成为全管道默认，适合统一基线。
Set once and llm / embed_model / chunk_size become pipeline-wide defaults — ideal for a uniform baseline.
vs
局部覆盖（特例）Local override (special case)
构造某个组件时显式传参，如 as_query_engine(llm=…, text_qa_template=…)，只改这一处而不动全局。
Pass arguments explicitly when building one component, e.g. as_query_engine(llm=…, text_qa_template=…) — changes this spot only, leaving the global intact.
同一套 RAG：全局定基线，局部按需特例化One RAG setup: set a global baseline, then special-case locally on demand
Prompt A · 宽松Prompt A · loose
模板只说“根据资料回答”。检索到的同一份上下文里其实没提保修期，宽松的措辞却纵容 LLM 顺嘴补一句“一般为一年”——看似贴心，实为编造、无出处。
The template merely says “answer from the materials”. The retrieved context never actually mentions the warranty period, yet the loose wording lets the LLM blurt out “usually one year” — helpful-looking, but fabricated and unsourced.
vs
Prompt B · 严格Prompt B · strict
模板写死“只用资料回答；未提及就说‘资料未提及’”。检索结果一字未变、问题也一样，这次 LLM 老实回答“资料未提及”——不编、可追溯。
The template hard-codes “answer only from the materials; if not mentioned, say ‘not in the materials’”. With the retrieved snippets byte-for-byte identical and the same question, the LLM now honestly says “not in the materials” — no fabrication, fully traceable.
同一份上下文、同一个问题，只换 Prompt：答案从“编造”翻转为“老实说不知道”——这就是 RAG 的最后一公里Same context, same question, only the prompt changes: the answer flips from fabrication to an honest “I don't know” — RAG's last mile

🧪 示例：设默认 + 定制模板🧪 Example: set defaults + a custom template
Settings.llm = OpenAI(...) 与 Settings.embed_model = OpenAIEmbedding(...) 设好全局默认；再用 PromptTemplate('…{context_str}…{query_str}…') 为某个 query engine 定制问答语气与约束。
Settings.llm = OpenAI(...) and Settings.embed_model = OpenAIEmbedding(...) set the global defaults; then a PromptTemplate('…{context_str}…{query_str}…') customizes the answering tone and constraints for a given query engine.
❓ 为什么这么设计❓ Why designed this way
全局默认去样板：不必把 llm / embed_model 一路当参数传给每个组件；把 Prompt 显式成模板，则让“最后一公里”的措辞集中、可审、可版本化，而不是散落在代码里的隐式字符串。
Global defaults kill boilerplate: you no longer thread llm / embed_model as arguments into every component. Making the prompt an explicit template keeps the “last-mile” wording centralized, auditable and versionable instead of buried as implicit strings.
⚙️ 内部怎么跑⚙️ How it runs inside
Settings 是一个进程内单例（_Settings）；组件构造时若没收到显式参数，就回退到它的字段。问答模板（text_qa_template）最终流向响应合成器，在合成阶段用检索到的上下文填满占位符，再交给 LLM。
Settings is an in-process singleton (_Settings); when a component is built without explicit arguments, it falls back to these fields. The QA template (text_qa_template) ultimately flows into the response synthesizer, which fills the placeholders with retrieved context at synthesis time before calling the LLM.
🔀 替代方案🔀 Alternatives
想要全局一致就用 Settings；只想在某一处用不同模型或模板，就在构造时局部传参（如 as_query_engine(llm=…, text_qa_template=…)），覆盖全局而不污染其它链路。
Use Settings for global consistency; when only one spot needs a different model or template, pass it locally at construction (e.g. as_query_engine(llm=…, text_qa_template=…)), overriding the global without polluting other paths.
全局配一次 + 局部定制 Promptconfigure globally + customize the prompt locally
from llama_index.core import Settings, PromptTemplate
from llama_index.llms.openai import OpenAI

Settings.llm = OpenAI(model='gpt-4o-mini')   # 全局默认，一处生效
Settings.chunk_size = 512

qa = PromptTemplate(
    '只根据下面的上下文回答，未提及就说不知道，不要编造。\n'
    '上下文:\n{context_str}\n问题: {query_str}\n答案: '
)
engine = index.as_query_engine(text_qa_template=qa)   # 定制问答 Prompt
print(engine.query('退款政策？'))
局部覆盖：换掉这一处的 llm 与模板，全局 Settings 原封不动Local override: swap this engine's llm and template; global Settings untouched
from llama_index.core import PromptTemplate
from llama_index.llms.openai import OpenAI

# 全局 Settings 保持不变；只在这个 query engine 上局部覆盖
strict = PromptTemplate(
    '只依据下面的上下文回答；若上下文未提及，请回答“资料中未提及”。\n'
    '上下文:\n{context_str}\n问题: {query_str}\n答案: '
)
engine = index.as_query_engine(
    llm=OpenAI(model='gpt-4o', temperature=0),   # 这一处换更强的模型
    text_qa_template=strict,                     # 这一处换更严格的模板
)
print(engine.query('保修期是多久？'))
✅ 本课要点✅ Key points
  • Settings 取代 ServiceContext，集中管理默认 llm/embed/切块等。Settings replaces ServiceContext, centralizing default llm/embed/chunking.
  • Prompt 模板用 {context_str} / {query_str} 注入上下文与问题。Prompt templates inject context and question via {context_str} / {query_str}.
  • 全局默认 + 局部覆盖：既省事又灵活。Global defaults + local overrides: convenient yet flexible.
💡 设计亮点💡 Design insight
“全局 Settings + 局部覆盖”避免了到处传配置对象的样板；而把 Prompt 显式成模板，让 RAG 的“最后一公里语气与约束”可控可审。
“Global Settings + local overrides” kills the boilerplate of threading a config object everywhere; making the prompt an explicit template keeps RAG's “last-mile tone and constraints” controllable and auditable.

🔑 重点：检索决定“依据对不对”、prompt 决定“依据用得对不对”——先把召回调达标，再用 prompt 调忠实度/格式。召回不到，prompt 再好也是巧妇难为无米之炊；指标上，检索看命中率/MRR，prompt 看 Faithfulness 与格式/约束达成率。别把两类问题混在一起调。🔑 Key: retrieval decides whether the evidence is right, the prompt decides whether it's used right — get recall to target first, then tune faithfulness/format with the prompt. No recall, no prompt can save it; measure retrieval by hit-rate/MRR and the prompt by Faithfulness plus format/constraint adherence. Don't conflate the two while tuning.

🔑 重点：全局少样板但隐式依赖、难追踪、测试易串味；显式可读可测、可多配置共存但啰嗦。规范：全局设默认值，关键/差异化组件(评估用的 judge、特殊 query engine)显式覆盖并就近声明——既省样板又可追踪。🔑 Key: global means less boilerplate but an implicit, hard-to-trace dependency that can leak state in tests; explicit is readable/testable and lets configs coexist but is verbose.
