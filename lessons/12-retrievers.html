查询路径的第一步：Retriever 把问题向量化，去索引里取回最相关的 top-k 个 Node（带相似度分数的 NodeWithScore）。index.as_retriever() 是入口，此刻还没调用 LLM。First stop of the query path: a Retriever embeds the question and fetches the most relevant top-k Nodes (scored NodeWithScore). index.as_retriever() is the entry point — no LLM has been called yet.

好召回Good recall
问「退款政策是什么？」→ retrieve() 取回 top-3：
0.89 退款政策条款 ✓
0.86 退款到账时限 ✓
0.83 退款申请流程 ✓
三块全相关、分数高，正确依据都在 top-k 里。
Ask “What's the refund policy?” → retrieve() returns top-3:
0.89 refund-policy clause ✓
0.86 refund timing ✓
0.83 how to file a refund ✓
all relevant, all high-scoring — the right evidence sits in the top-k.
vs
坏召回Bad recall
同一个问题，却取回：
0.71 配送时效 ✗ 跑题
0.55 账户注册 ✗ 跑题
0.31 退款政策条款 —— 正确块分数偏低、险些没进 top-k
无关块挤进来、正确依据沉到末尾，召回失准。
Same query, yet it fetches:
0.71 delivery times ✗ off-topic
0.55 account signup ✗ off-topic
0.31 refund-policy clause — the right chunk, scoring low, barely in the top-k
noise crowds in and the right evidence sinks to the bottom — recall is off.

为什么把检索单独拎出来Why isolate retrieval as its own step

检索与生成解耦后，“召回对不对”可以脱离 LLM 单独度量：直接看 retrieve() 返回的 Node 是否包含正确依据。于是调相似度阈值、换检索器、改 top_k 都能立刻验证，而不必每次都跑一遍昂贵的生成。
Once retrieval is decoupled from generation, “did we recall the right content?” can be measured without the LLM at all: just inspect whether the Nodes from retrieve() contain the right evidence. Tuning the cutoff, swapping retrievers or changing top_k is then verified instantly, without paying for generation every time.
这正是把检索单列一课的原因：命中率（hit-rate）、MRR 这类指标只对 retrieve() 的输出打分，完全不跑生成。召回这关没过，再强的 LLM 也只是在错误的资料上一本正经地作答；先把“取回对的内容”调好，后面的后处理与合成才谈得上意义。
That's exactly why retrieval earns a lesson of its own: metrics like hit-rate or MRR score only the output of retrieve(), never running generation. If recall fails, even the strongest LLM merely reasons fluently over the wrong material; get “fetch the right content” right first, and only then do post-processing and synthesis even matter.

🧪 示例：只检索不生成🧪 Example: retrieve without generating
index.as_retriever(similarity_top_k=3).retrieve('…') 返回 List[NodeWithScore]，你可以直接打印分数和文本，确认召回是否命中——完全不碰 LLM。
index.as_retriever(similarity_top_k=3).retrieve('…') returns a List[NodeWithScore]; print the scores and text to confirm the recall hit — without touching the LLM.
❓ 为什么这么设计❓ Why designed this way
把检索独立成一等公民，你就能单独评估与优化召回，再分别优化生成，两个问题不再纠缠。
Making retrieval a first-class citizen lets you evaluate and optimize recall on its own, then optimize generation separately — the two problems stop entangling.
⚙️ 内部怎么跑⚙️ How it runs inside
BaseRetriever.retrieve() 是模板方法：它处理 QueryBundle 等通用逻辑，再调用各子类实现的 _retrieve()（如 VectorIndexRetriever 做向量近邻搜索）。
BaseRetriever.retrieve() is a template method: it handles shared logic like the QueryBundle, then calls each subclass's _retrieve() (e.g. VectorIndexRetriever does vector nearest-neighbor search).
🔀 替代方案🔀 Alternatives
除了向量检索，还有关键词检索（BM25）、知识图谱检索，以及把多者融合的混合检索——它们都实现同一个 BaseRetriever 接口，可直接替换。
Beyond vector search there's keyword retrieval (BM25), knowledge-graph retrieval, and hybrid retrieval that fuses several — all implementing the same BaseRetriever interface, so they drop in interchangeably.
检索独立于生成，可单独评估retrieval is separate from generation — evaluate it alone
retriever = index.as_retriever(similarity_top_k=3)
nodes = retriever.retrieve('退款政策是什么？')   # List[NodeWithScore]
for n in nodes:
    print(round(n.score, 3), n.node.text[:50])
# 注意：此处只检索，尚未生成答案
显式构造 VectorIndexRetriever，逐条查看分数与来源Construct VectorIndexRetriever explicitly to inspect each score and source
from llama_index.core.retrievers import VectorIndexRetriever

# 直接构造检索器（等价于 index.as_retriever，但更显式）
retriever = VectorIndexRetriever(index=index, similarity_top_k=4)
for n in retriever.retrieve('国际订单的退款要多久？'):
    print(f'{n.score:.3f}  {n.node.get_content()[:50]}')
    print('  来源:', n.node.metadata.get('file_name'))
✅ 本课要点✅ Key points
  • Retriever 返回 top-k 个带分数的 Node，不产生最终答案。A Retriever returns the top-k scored Nodes, not the final answer.
  • similarity_top_k 太小漏召回、太大引噪声。Too small a similarity_top_k misses; too large adds noise.
  • 检索与生成解耦，便于单独调试与评估检索质量。Decoupling retrieval from generation lets you debug and evaluate retrieval on its own.
💡 设计亮点💡 Design insight
把“取相关片段”独立成 Retriever，意味着你能在不动 LLM 的前提下，专门优化“召回了对的内容没有”这一核心问题。
Isolating “fetch relevant pieces” as a Retriever means you can optimize the core question — “did we recall the right content?” — without ever touching the LLM.
