01 · LlamaIndex 与 RAG 是什么

查询路径Query path检索Retrieve后处理Post-process合成Synthesize回答Answer

LlamaIndex 是一个把检索增强生成（RAG）这条数据管道标准化、可编排的框架。它不训练模型，只负责模型周边的全部管道：把你的数据加载、切块、向量化、建索引，再在提问时检索相关片段、喂给 LLM 生成有依据的回答。LlamaIndex is a framework that standardizes and orchestrates the Retrieval-Augmented Generation (RAG) pipeline. It doesn't train models — it owns everything around them: loading, chunking, embedding and indexing your data, then at query time retrieving the relevant pieces and feeding them to the LLM for a grounded answer.

🧩 生活类比🧩 Analogy
把 LLM 想成一个聪明但在闭卷考试的学生：只能凭记忆答题，记不住就开始编。RAG 让它开卷——先翻到相关页（检索），再据此作答（生成）。LlamaIndex 就是帮你建好这座图书馆和索书系统的工具。
Picture an LLM as a smart student in a closed-book exam: it answers from memory and invents things it can't recall. RAG makes it open-book — find the relevant pages first (retrieve), then answer from them (generate). LlamaIndex builds that library and its call-number system.

三个痛点，同一个根因Three pains, one root cause

知识截止、幻觉、私有数据看似三件事，根因只有一个：模型的知识在训练时就被冻结在参数里，既无法更新，也说不清依据。RAG 不去改这套被冻结的参数，而是在提问时临时挂载一份可检索的外部资料，让答案有据可查、随时可换。
Knowledge cutoff, hallucination and private data look like three problems but share one root cause: the model's knowledge is frozen into its weights at training time — it can neither refresh nor cite. RAG doesn't touch those frozen weights; it mounts a searchable external corpus at query time so answers stay grounded and the knowledge stays swappable.

🧪 示例：直接塞进 prompt 会怎样🧪 Example: what if you just stuff the prompt
把整本手册粘进 prompt 看似省事，但会撞上上下文长度上限、推理更慢更贵，且模型容易“迷失在中间”而漏掉关键句。RAG 只挑最相关的几段进 prompt，既省 token 又更准。
Pasting a whole manual into the prompt seems easy, but you hit the context-length ceiling, pay more for slower inference, and the model tends to get “lost in the middle” and miss key lines. RAG feeds only the few most relevant snippets — cheaper and more accurate.
❓ 为什么这么设计❓ Why designed this way
把知识外置成可检索索引，意味着更新知识只需重建索引、模型零改动；同一个模型可服务无数套互不相同的私有库。
Externalizing knowledge into a searchable index means updating knowledge is just rebuilding the index — the model never changes, and one model can serve countless private corpora.
⚙️ 内部怎么跑⚙️ How it runs inside
from_documents 在写入路径里把文档切块、向量化、写进索引；query 在查询路径里把问题向量化、取最近邻、再交给 LLM 据此作答——两行 API 各藏了一条管道。
from_documents runs the write path (split → embed → store); query runs the query path (embed the question → nearest-neighbor search → LLM answers from them) — each one-liner hides a pipeline.
🔀 替代方案🔀 Alternatives
微调把知识焊进参数（更新贵、难溯源）；超长上下文直接全塞（贵且会“迷失在中间”）；RAG 取两者之间——外置、可更新、可溯源，是私有或时效性知识的默认选择。
Fine-tuning welds knowledge into weights (costly to update, hard to cite); ultra-long context just dumps everything in (expensive, “lost in the middle”); RAG sits in between — external, refreshable, citable — the default for private or time-sensitive knowledge.

✅ 本课要点✅ Key points
  • RAG = 先检索相关片段，再让 LLM 据此生成，避免重训与幻觉。RAG = retrieve relevant snippets, then generate from them — no retraining, fewer hallucinations.
  • LlamaIndex 负责 RAG 的数据与编排，不做训练或推理本身。LlamaIndex owns the data + orchestration of RAG, not training or inference.
  • 两条主线贯穿全书：写入路径（建索引）与查询路径（问答）。Two through-lines: the write path (build the index) and the query path (ask).
💡 设计亮点💡 Design insight
RAG 的精妙在于把模型的“记忆”外置为一个可随时更新、可检索的索引——更新知识只需重建索引，模型本身原封不动。
RAG's elegance: it externalizes the model's “memory” into a searchable index you can refresh anytime — update knowledge by rebuilding the index, never touching the model.

为什么把接口和实现拆开Why split interfaces from implementations

core 只定义“能做什么”的抽象接口，具体“怎么做”交给独立集成包。于是新增一个 LLM 或向量库厂商，只需发布一个集成包，core 不必改动、也不必跟着发版；你的应用只依赖 core 的稳定接口，换厂商几乎零成本。
core defines only the abstract “what”; each concrete “how” lives in a separate integration. So adding a new LLM or vector-store vendor is just publishing one integration package — core never changes or re-releases, and because your app depends only on core's stable interface, swapping vendors is near-free.

切块（chunking）把 Document 拆成大小合适的 Node。为什么要切？因为 embedding 与 LLM 上下文都有限，且检索希望命中精准的小片段。chunk_size 与 chunk_overlap 是核心旋钮。Chunking splits a Document into right-sized Nodes. Why split? Because embeddings and LLM context are limited, and retrieval wants to hit a precise small piece. chunk_size and chunk_overlap are the key knobs.

chunk_size / chunk_overlap 的权衡The chunk_size / chunk_overlap trade-off

chunk_size 太大，块里混入大量无关内容，检索噪声多、还挤占 LLM 上下文；太小，完整语义被切碎，块与块之间断裂、读不成句。chunk_overlap 用少量冗余换“不把一句话切两半”。这对“精准命中 vs 保留上下文”的张力，sentence-window 给了一个漂亮的折中：用单句去检索（足够精准），却把前后句组成的窗口喂给 LLM（补回上下文）。
Too large a chunk_size mixes in lots of irrelevant text — noisy retrieval that also hogs LLM context; too small and a complete thought is shredded, with chunks that break mid-sentence. chunk_overlap trades a little redundancy for “never cut a sentence in half”. For the tension between “precise hits vs. retained context”, sentence-window offers an elegant compromise: retrieve on a single sentence (precise enough), yet feed the LLM the window of surrounding sentences (context restored).

生产数据是活的：每天只有一小部分变化。幂等 + 缓存 + 去重让“重建索引”的代价正比于变化量而非总量，既省钱又省时，还保证重复运行结果一致。
Production data is alive: only a small slice changes each day. Idempotency + cache + dedup make the cost of “rebuilding the index” proportional to the delta, not the total — cheaper, faster, and consistent across repeated runs.
⚙️ 内部怎么跑⚙️ How it runs inside
run 时，管道对每个转换的（输入 Node + 转换配置）算一个哈希作为缓存键，命中就复用上次结果；docstore 则按 doc_id 记录每个文档的内容哈希，配合 DocstoreStrategy 决定是 UPSERT 还是跳过。
On run, the pipeline hashes each transformation's (input Nodes + config) into a cache key and reuses last time's output on a hit; the docstore records each document's content hash by doc_id and, via a DocstoreStrategy, decides whether to UPSERT or skip.
🔀 替代方案🔀 Alternatives
不用管道也能跑：每次都 from_documents 全量重建。小数据时无所谓，但随着库变大，全量重算的时间和 embedding 费用会线性膨胀——这正是增量摄取存在的理由。
You can skip the pipeline and just from_documents from scratch every time. Fine for small data, but as the corpus grows the recompute time and embedding bill scale linearly — exactly why incremental ingestion exists.

from llama_index.core import (
    SimpleDirectoryReader, VectorStoreIndex, StorageContext,
    load_index_from_storage, Settings, get_response_synthesizer,
)
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.postprocessor import SimilarityPostprocessor
from llama_index.core.query_engine import RetrieverQueryEngine
from llama_index.core.evaluation import FaithfulnessEvaluator
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
import os

Settings.llm = OpenAI(model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model='text-embedding-3-small')

PERSIST = './storage'
if os.path.exists(PERSIST):                      # 已建过就秒加载
    index = load_index_from_storage(StorageContext.from_defaults(persist_dir=PERSIST))
else:                                            # 首次：摄取 -> 建索引 -> 落盘
    docs = SimpleDirectoryReader('./data').load_data()
    nodes = IngestionPipeline(transformations=[SentenceSplitter(chunk_size=512, chunk_overlap=50)]).run(documents=docs)
    index = VectorStoreIndex(nodes)
    index.storage_context.persist(persist_dir=PERSIST)

engine = RetrieverQueryEngine.from_args(
    retriever=index.as_retriever(similarity_top_k=4),
    node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=0.4)],  # 阈值随数据而定
    response_synthesizer=get_response_synthesizer(response_mode='compact'),
)
resp = engine.query('退款政策是什么？')
print('答案:', resp)
print('忠实度:', FaithfulnessEvaluator().evaluate_response(response=resp).passing)

chat = index.as_chat_engine(chat_mode='condense_plus_context')
print(chat.chat('那国际订单也一样吗？'))

from llama_index.core.evaluation import FaithfulnessEvaluator, RelevancyEvaluator

# 在 capstone 的 engine 之上：把单次评估扩成可回归的质量闸
faith = FaithfulnessEvaluator()        # 答案是否忠于检索到的上下文
relev = RelevancyEvaluator()           # 检索到的上下文是否切题

eval_set = ['退款要多久到账？', '国际订单能退吗？', '怎么申请换货？']
passed = 0
for q in eval_set:
    resp = engine.query(q)
    ok = (faith.evaluate_response(response=resp).passing
          and relev.evaluate_response(query=q, response=resp).passing)
    passed += int(ok)
    print(q, 'PASS' if ok else 'FAIL')
print(f'通过率 pass-rate: {passed}/{len(eval_set)}')
